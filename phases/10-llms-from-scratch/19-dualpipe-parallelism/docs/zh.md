# DualPipe并行

> DeepSeek-V3 在 2048 块 H800 GPU 上训练，MoE 专家分散在各个节点上。跨节点专家全对全通信的成本是每 1 GPU 小时计算对应 1 GPU 小时通信。GPU 有一半时间处于空闲状态。DualPipe（DeepSeek，2024年12月）是一种双向流水线，它将前向和后向计算与它们触发的全对全通信重叠。气泡减少，吞吐量提升，并且一旦专家并行已经在各个秩之间分散专家，保留两个模型参数副本（即名称中的“dual”）是廉价的。本课是 Learn 类型的演练，介绍 DualPipe 实际做了什么，以及为什么 Sea AI Lab 的 DualPipeV 改进版以稍微收紧的气泡为代价去掉了 2 倍参数成本。

**类型：** 学习
**语言：** Python（标准库，调度模拟器）
**先修知识：** 阶段 10 · 05（分布式训练，FSDP，DeepSpeed），阶段 10 · 14（开放模型架构和 MoE）
**时间：** 约 60 分钟

## 学习目标

- 列举 DualPipe 前向-后向块的四个组成部分，并解释为什么每个部分都有自己的重叠窗口。
- 解释大规模下的流水线气泡问题，以及“无气泡”在实践与营销中的不同含义。
- 手动追踪 8 个 PP 秩和 16 个微批次下的 DualPipe 调度，并确认正向和反向流互相填充对方的空闲槽。
- 说明 DualPipeV（Sea AI Lab，2025）所做的权衡：当专家并行未激活时，以略大的气泡为代价去掉 2 倍参数复制。

## 问题

在 2k 块 H800 GPU 上训练 671B MoE 模型会遇到三个叠加的瓶颈：

1. **内存压力。** 每个 GPU 持有模型的一个切片。在序列长度 8k、61 层、128 头的情况下，激活内存非常巨大。
2. **流水线气泡。** 传统的流水线并行（GPipe，1F1B）在 GPU 等待其阶段输入或梯度时使其闲置。即使在 8 阶段且使用 1F1B 调度的情况下，大约 12% 的 GPU 时间可能成为气泡。
3. **跨节点全对全。** 带有专家并行的 MoE 将专家分散到各个节点。每次前向传递触发一次全对全通信以将 token 分发给其专家，另一次通信以合并结果。在 2k GPU 的情况下，这很容易变成 1:1 的计算与通信比。

每个问题都有单独的解决方案：梯度检查点解决内存问题，Zero Bubble（Sea AI Lab，2023）解决流水线气泡问题，专家并行通信内核解决全对全问题。DualPipe 所做的是让它们协同工作。调度在单个前向-后向块内重叠计算和通信，同时从流水线两端注入微批次，并利用由此产生的调度将全对全隐藏在计算窗口中。

报告结果：几乎消除了流水线气泡，在 DeepSeek-V3 的 14.8T token 训练运行中 GPU 利用率超过 95%。

## 核心概念

### 流水线并行回顾

将一个 N 层模型分割到 P 个设备上。设备 `i` 持有层 `i * N/P .. (i+1) * N/P - 1`。一个微批次从前向后流经设备 0 到 P-1，然后从后向前从 P-1 到 0。每个设备只能在前一个设备发送其输出后开始其前向阶段，并且只能在下游设备发送上游梯度后开始后向阶段。

GPipe（Huang et al., 2019）一次调度一个微批次，这浪费了大部分 GPU 时间。1F1B（Narayanan et al., 2021）交错多个微批次的前向和后向传递。Zero Bubble（Qi et al., 2023）将后向传递拆分为两部分——后向-用于输入（B）和后向-用于权重（W）——并调度它们以填充气泡。在 Zero Bubble 之后，流水线几乎是紧凑的。

DualPipe 是下一步。它在上面增加了两个想法：

### 想法 1：块分解

每个前向块被拆分为四个组成部分：

- **注意力。** Q/K/V 投影，注意力，输出投影。
- **全对全分发。** 将 token 发送到其专家的跨节点通信。
- **MLP。** MoE 专家计算。
- **全对全合并。** 将专家输出带回的跨节点通信。

一个后向块会增加每个这些的梯度版本。DualPipe 调度它们，使得全对全分发与下一个块的注意力计算并行，全对全合并与后续块的 MLP 计算并行。

### 想法 2：双向调度

大多数流水线调度从阶段 0 注入微批次并流向阶段 P-1。DualPipe 从两端同时注入微批次。阶段 0 看到源自那里的前向微批次；阶段 P-1 也看到源自那里的前向微批次。两股流在中间相遇。

为了实现这一点，设备 `i` 必须同时持有流水线早期层 `i` 和流水线后期层 `P - 1 - i`。这就是 DualPipe 的“dual”部分：每个设备保留其需要服务的模型层的两个副本（每个方向一个）。在 DeepSeek-V3 的规模下，这是 2 倍参数复制成本。这是可以承受的，因为专家并行已经将 MoE 专家分散得很薄，因此复制非专家层两次是小菜一碟。

关键的是，一个方向的前向流和另一个方向的后向流正好重叠在单方向调度中会出现气泡的位置。气泡消失了。

### 手动追踪的调度

考虑 P = 4 个秩，8 个微批次，分为 4 个正向 / 4 个反向。时间从左到右移动；行是设备秩。

```
           Time →
rank 0:  F1 F2 F3 F4  F5R F6R F7R F8R  B1 B2 B3 B4  ...
rank 1:     F1 F2 F3  F4/F5R F6R F7R   B1 B2 ...
rank 2:        F1 F2  F3/F5R F4/F6R    B1 ...
rank 3:           F1  F2/F5R F3/F6R    ...
```

解读“F4/F5R”标记：秩 1 在同一时间槽内运行微批次 4 的前向（在流水线中从左到右）和微批次 5 的前向（从右到左）。这就是“双向”在操作上的含义。

在秩 2，交叉流更早重叠，在秩 0 和 P-1，它们最晚重叠。在调度的稳定中间阶段，每个秩运行 X 方向的前向与 Y 方向的后向重叠。计算繁忙。前向传递的全对全分发隐藏在后向计算中。全对全合并隐藏在前向计算中。气泡被挤出。

### 气泡核算

标准 1F1B 流水线气泡（每个秩浪费的时间）：

```
bubble_1F1B = (P - 1) * forward_chunk_time
```

Zero Bubble 改进将其降低但未归零。DualPipe 在稳定阶段具有零气泡，如果微批次数量能被 2 倍流水线深度整除。在稳定阶段之外（预热和冷却），存在一些气泡，但它不会随着微批次数量增长——这是论文强调的一个关键特性。

在营销术语中：“无气泡”。在技术术语中：气泡不会随着微批次数量增长。Sea AI Lab 的后续分析（DualPipeV / 对半切）表明，只有当专家并行不是瓶颈时才能实现完全零气泡；在 EP 驱动的全对全下，总是存在一些调度折衷。

### DualPipeV — 改进版

Sea AI Lab（2025）观察到，当 EP 通信重叠不是关键时，2 倍参数复制是浪费的。他们的 DualPipeV 调度将双向注入折叠成“V形”调度，在单个参数副本上运行。气泡比 DualPipe 稍大，但内存节省显著。DeepSeek 在其开源 DualPipe 实现中采用了 DualPipeV 作为 EP 关闭模式。

权衡：

|  特性  |  DualPipe  |  DualPipeV  |  1F1B  |  Zero Bubble  |
|---------|---------|-----------|------|------------|
|  每设备参数副本数  |  2  |  1  |  1  |  1  |
|  气泡与微批次  |  常数  |  小幅增长  |  增长  |  增长  |
|  计算-通信重叠  |  完全  |  部分  |  极小  |  部分  |
|  适用场景  |  EP密集型MoE  |  密集或EP轻量  |  基线  |  任意流水线  |

### 对14.8T token训练的意义

DeepSeek-V3的预训练在大约2.8M GPU小时内使用2,048块H800 GPU消耗了14.8T个token。采用朴素的1F1B方法，他们会因流水线气泡(pipe bubble)损失12-15%——即340-420K GPU小时，足以训练一个完整的70B模型。DualPipe恢复了其中的大部分。直接量化其贡献在没有内部日志的情况下很困难，但论文中声称平均训练GPU利用率超过95%。

对于较小规模（低于1k GPU）的训练，DualPipe是过度设计的——相对总成本而言，流水线气泡较小，且密集模型训练很少遇到all-to-all瓶颈。对于数千GPU规模的前沿MoE训练，它实际上是必需的。

### 在技术栈中的位置

- 与**FSDP**（阶段10 · 05）互补。FSDP将模型参数分片到各个rank；DualPipe调度跨rank的计算。两者结合。
- 与**ZeRO-3**梯度分片兼容。双副本复用的记账需要与ZeRO的分片梯度协作。
- 需要**定制的all-to-all内核**，针对特定集群拓扑调优。DeepSeek的开源内核是参考实现。

## 使用它

`code/main.py`是一个流水线调度模拟器。它接受`(P, n_micro_batches, schedule)`，并输出1F1B、Zero Bubble、DualPipe和DualPipeV的稳定阶段利用率。这是一个教学工具——数字与论文中的定性声明一致，并不代表生产环境实测加速。

模拟器的价值：使用不同的P和微批次数量运行它，观察1F1B的气泡比例如何增长，而DualPipe则不增长。

实际训练运行的集成考虑：

- 选择一个能干净地整除微批次数量的流水线并行深度。
- 确保你的专家并行网格支持双向all-to-all。DeepSeek的内核是参考实现。
- 预计首次会花费一周时间在调度本身调试上。记账工作很繁琐。
- 监控每个rank的GPU利用率，而不仅仅是聚合值。DualPipe的优势来自收紧拖后腿的节点。

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|----------------|------------------------|
|  流水线气泡(pipe bubble)  |  "每个rank的空闲时间"  |  由于流水线阶段等待其输入或梯度而浪费的GPU周期  |
|  1F1B  |  "默认流水线调度"  |  一个前向/一个反向交错调度；DualPipe超越的基线  |
|  Zero Bubble  |  "Sea AI Lab，2023年"  |  将反向拆分为B（输入梯度）和W（权重梯度）；几乎完全收紧流水线  |
|  DualPipe  |  "DeepSeek-V3调度"  |  双向流水线+计算-通信重叠；气泡不随微批次数量增长  |
|  DualPipeV  |  "切半"  |  牺牲2倍参数复制以换取略微增大的气泡的V形改进  |
|  块(Chunk)  |  "流水线工作单元"  |  一个微批次通过一个流水线阶段的一次前向或反向传播  |
|  All-to-all分散  |  "将token发送到专家"  |  将token路由到其分配的MoE专家的跨节点通信  |
|  All-to-all组合  |  "将专家输出带回"  |  MLP后收集专家输出的跨节点通信  |
|  专家并行(Expert Parallelism, EP)  |  "跨GPU的专家"  |  将MoE专家分片到各个rank，使不同GPU持有不同专家  |
| 管道并行(PP)  |  "跨GPU的层"  |  将模型层分片到各秩；DualPipe调度的维度 |
| 气泡占比  |  "浪费的GPU时间"  |  (气泡时间/总时间)；DualPipe趋近于零的占比 |
