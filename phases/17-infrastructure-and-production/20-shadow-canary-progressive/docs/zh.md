# LLM 的影子流量、金丝雀发布与渐进式部署

> LLM 部署集合了软件部署中最困难的部分：没有单元测试、模糊的故障模式、延迟的信号。流程是 (1) 影子模式(Shadow Mode)——将生产请求复制到候选模型，记录日志，在零用户影响下进行比较；能捕获明显的分布问题，但不是质量保证；(2) 金丝雀发布(Canary Rollout)——逐步转移流量 10% → 25% → 50% → 75% → 100%，每一步设闸门；追踪延迟百分位数、每次请求成本、错误/拒绝率、输出长度分布、用户反馈率；(3) 稳定性确认后对不同方案进行 A/B 测试。非确定性是不可消除的——由于 GPU FP 非结合性和批次大小变化，相同输入在不同运行中精度差异可达 15%。成本是变量而非常量——一个好 20% 的模型每次调用可能贵 3 倍。回滚速度至关重要：如果回滚需要重新部署，那你太慢了。策略存在于配置/标志中；模型存在于带有固定摘要的注册表中；回滚 = 翻转策略 + 恢复阈值 + 在几秒内固定旧模型。

**类型：** 学习
**语言：** Python（标准库，简易金丝雀进程模拟器）
**前置条件：** 阶段 17 · 13（可观测性），阶段 17 · 21（A/B 测试）
**时间：** 约 60 分钟

## 学习目标

- 区分影子模式（零影响比较）、金丝雀（实时流量逐步）和 A/B（稳定性确认后的比较）。
- 列举五个 LLM 特有的金丝雀指标（延迟、每次请求成本、错误/拒绝、输出长度分布、用户反馈）。
- 解释为什么 LLM 的非确定性（高达 15%）改变了发布中“稳定”的含义。
- 设计一个只需几秒（策略翻转）而非几小时（重新部署）的回滚路径。

## 问题

你发布了一个新模型。离线评估显示提升了 3% 的准确率。你在生产环境中启用它。24 小时内，成本上升了 40%，用户点踩率上升了 8%，三个客户工单报告“奇怪的回答”。你进行回滚。重新部署需要 3 小时。你的周末毁了。

每一步都是可以避免的。影子模式会在任何用户看到之前就发现 40% 的成本激增。金丝雀会在点踩率变化时在 10% 流量处停止。策略标志回滚只需 30 秒。这种纪律填补了“离线评估看起来不错”和“真实用户满意”之间的鸿沟。

## 核心概念

### 影子模式

候选模型接收与生产环境相同的请求；输出被记录，不返回给用户。零用户影响。记录：

- 输出内容（与生产环境的差异）。
- 令牌数（成本增量）。
- 延迟。
- 拒绝和错误。

能捕获：成本激增、长度退化、明显的拒绝变化、硬错误。不能捕获：用户会感知到的质量差异。影子模式是冒烟测试，不是质量测试。

### 金丝雀发布

带闸门的逐步流量转移。典型过程：1% → 10% → 25% → 50% → 75% → 100%。每一步对 5 个指标设置闸门：

1. **延迟百分位数** — P50, P95, P99。触发条件：金丝雀的 P99 > 基线的 1.5 倍。
2. **每次请求成本** — 混合成本。触发条件：高于基线 20%。
3. **错误/拒绝率** — 5xx 加上显式拒绝。触发条件：基线的 2 倍。
4. **输出长度分布** — 平均值 + P99。触发条件：分布偏移。
5. **用户反馈率** — 点踩/工单提交。触发条件：基线的 1.5 倍。

### 非确定性是新的方差

相同输入产生不同输出。原因：

- GPU FP 非结合性（浮点归约顺序因批次而异）。
- 批次大小差异（同一提示在 128 批次与 16 批次中）。
- 采样（温度 > 0）。

测量：相同评估集上运行间准确率差异高达 15%。发布中的“稳定”意味着指标在预期方差内，而非与基线完全相同。将闸门设置在噪声层之上。

### 成本是变量

一个好 20% 的模型每次调用可能贵 3 倍。每次请求成本是五个闸门之一。发布一个破坏单位经济性的“更好”模型属于回滚情况。

### 回滚是武器

- 策略标志（特性标志系统）：在配置中翻转百分比；耗时数秒。
- 模型固定（注册表摘要）：固定的模型不会自动升级。
- 回滚 = 恢复标志 + 将固定摘要设置为上一个。几秒，而非几小时。

如果你的技术栈需要重新部署来回滚，在发布前先修复这个问题。

### 工具

**Argo Rollouts** / **Flagger** — Kubernetes 渐进式交付控制器。与 Istio/Linkerd 加权路由集成。

**Istio 加权路由** — 服务网格级别的流量拆分。

**KServe / Seldon Core** — 带有内置金丝雀功能的模型服务。

**特性标志** — LaunchDarkly, Flagsmith, Unleash。策略级别的翻转，无需重新部署。

### 指标节奏

金丝雀闸门根据流量大小每 5-15 分钟检查一次。1% 流量，每分钟 10 个请求，每个窗口有 50-150 个数据点 —— 对延迟足够，但对用户反馈噪声大。10% 流量提供约 10 倍的数据。进展应在每一步暂停足够长时间以积累足够样本。

### A/B 步骤是可选的

如果新模型明显不同（不同的行为、不同的成本曲线、不同的语气），在金丝雀通过后以 50% 流量进行 A/B 测试。如果只是改进版本，金丝雀闸门通过后直接跳到 100%。

### 你应该记住的数字

- 金丝雀部署进程：1% → 10% → 25% → 50% → 75% → 100%。
- 非确定性上限：相同输入下运行间差异高达 15%。
- 五个金丝雀指标：延迟、成本、错误/拒绝、输出长度、用户反馈。
- 成本门控：超过基线 20% 即视为违规。
- 回滚：秒级，而非小时级。

1. 运行 `code/main.py`。注入 25% 的成本回归。金丝雀会在哪个阶段停止？
2. 你的新模型离线准确率提升 3%，但每次请求成本增加 18%。是否可以发布？取决于策略——写出两种路径。

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|----------------|------------------------|
| 影子模式 | "复制到新" | 零影响发送到候选进行日志记录 |
| 金丝雀 | "渐进流量" | 带有门控的逐步用户可见发布 |
| 门控 | "发布检查" | 阻隔进展的指标阈值 |
| 非确定性 | "LLM 方差" | 不可减少的运行间差异 |
| 策略标志 | "标志翻转回滚" | 配置级回滚，秒级而非小时级 |
| 模型固定 | "注册表摘要" | 模型版本的不可变引用 |
| Argo Rollouts | "K8s 渐进" | 原生的 Kubernetes 金丝雀/回滚控制器 |
| KServe | "推断 K8s" | 带有金丝雀原语的模型服务 |
| Istio 权重 | "网格拆分" | 服务网格流量拆分器 |
