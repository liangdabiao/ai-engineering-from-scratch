# 批量 API —— 以 50% 折扣作为行业标准

> 每个主流提供商都提供异步批处理API，享受50%的折扣和约24小时的周转时间。OpenAI、Anthropic、Google以及大多数推理平台（Fireworks批量层、Together批量层）都实现了相同的模式。将批处理与提示缓存结合，隔夜流水线的成本降至同步非缓存成本的约10%。规则非常简单：如果不是交互式的，就应该使用批处理。内容生成流水线、文档分类、数据提取、报告生成、批量标注、目录标记——任何能容忍24小时延迟的任务，如果不迁移到批处理，就是在浪费钱。2026年的生产模式是将每个新的LLM工作负载分流到三个通道：交互式（带缓存的同步）、半交互式（带回退的异步队列）、批处理（隔夜、缓存输入叠加）。那些假装是交互式但能容忍几分钟延迟的工作负载浪费了大部分成本。

**先决条件：** 阶段 17 · 14 (提示词与语义缓存)

## 学习目标

- 说出三个提供商的批处理API（OpenAI、Anthropic、Google）以及通用的50%折扣+24小时周转保证。
- 计算将批处理+缓存输入叠加到隔夜分类工作负载上的成本，并与同步非缓存基线进行比较。
- 将工作负载分流到交互式/半交互式/批处理，并证明该分流合理。
- 说出两个陷阱：部分交互性（用户期望速度比24小时快）和输出模式漂移（批处理文件格式因提供商而异）。

## 问题

你的团队负责一个夜间报告生成流水线。5万份文档，每份生成摘要，对摘要进行聚类，撰写执行摘要。同步运行需要4小时，每晚花费2000美元。你听说了批处理API。

批处理给你50%的折扣。你还对系统提示（所有5万次调用共享）启用了提示缓存。叠加后，账单降至每晚180美元——约为基线的9%。相同的流水线，只需三个配置更改。

批处理是LLM成本工具包中最便宜的杠杆，却很少有人使用。原因主要是组织上的：当服务级别协议实际上是“到早上”时，团队却认为是“实时”。本课是关于不要浪费90%的账单。

## 核心概念

### 三个批处理API

**OpenAI批处理API**：上传JSONL文件，包含一系列请求。承诺24小时周转（实际通常约2-8小时）。输入和输出令牌享受50%折扣。`/v1/batches`端点。可缓存输入还额外享受缓存输入定价。

**Anthropic消息批处理**：JSONL上传。24小时周转。50%折扣。支持`cache_control`——缓存写入是显式的，读取在批处理内自动进行。

**Google Vertex AI批处理预测**：BigQuery或GCS输入。Gemini同样享受50%折扣。与Vertex流水线集成。

### 语义：异步，而非缓慢

批处理是“我保证在24小时内返回”——而不是“这将花费24小时”。典型的P50是2-6小时。提供商在GPU库存利用率较低的离峰窗口调度你的批处理。

### 与缓存叠加

一个5万文档的摘要任务，使用相同的4K令牌系统提示：

- 同步非缓存：50000 × ($input × 4000 + $输出 × 200) 按全费率计算。
- 同步缓存：首次写入后系统提示被缓存；剩余49999次调用输入成本降低10倍。
- 批处理缓存：以上所有加上读和写均享受50%折扣。

叠加效果：批处理+缓存 = 同步非缓存账单的约10%。任何隔夜运行且共享系统提示的工作负载都应使用此方法。

### 工作负载分流

**交互式**——用户等待响应。首字时间(TTFT)很重要。带提示缓存的同步调用。不能批处理。

**半交互式**——用户提交任务，几分钟后回来查看结果。异步队列，如果批处理不可用则回退到同步。例如中等规模的RAG索引。

**批处理**——用户期望结果“到早上”或“下个小时”。内容流水线、大规模分类、离线分析。始终批处理，始终叠加缓存。

常见错误：因为流水线是生产环境，就将所有内容分类为交互式。生产环境不是延迟规格——服务级别协议(SLA)才是。

### 部分交互性陷阱

某些功能看起来是交互式的，但能容忍5-10分钟。例如：一个夜间客户健康报告，带有“刷新”按钮。用户点击刷新；等待10分钟没问题。团队却将其部署为同步。50个并发刷新所花费的成本是批处理并通过电子邮件发送的10倍。

要问的问题是：“对于这个用户，24小时意味着什么？”如果答案是“他们不会注意到”，那就用批处理。

### 输出模式陷阱

批处理文件格式因提供商而异：

- OpenAI: JSONL，每行一个请求。
- Anthropic: JSONL，每行一个消息；响应格式嵌入其中。
- Vertex: BigQuery表或GCS前缀带TFRecord。

编写“一个批处理客户端”以跨提供商使用，意味着每个提供商都需要适配器代码。号称支持多提供商批处理的网关（Portkey、LiteLLM某些层级）仍然只是对原始格式的薄封装。

### 你应该记住的数字

- 跨提供商的批处理折扣：输入+输出统一50%折扣。
- 周转服务级别协议：保证24小时，典型P50为2-6小时。
- 叠加批处理+缓存输入：约同步非缓存成本的10%。
- 工作负载分流规则：如果能接受24小时延迟，始终用批处理。

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|----------------|------------------------|
| 批处理API  |  “异步折扣”  |  24小时内周转节省50% |
| JSONL  |  “批处理格式”  |  每行一个JSON请求；OpenAI/Anthropic标准 |
| 消息批处理  |  “Anthropic批处理”  |  Anthropic的批处理API产品名称 |
| 批处理预测  |  “Vertex批处理”  |  Vertex AI的批处理API产品 |
| 周转SLA  |  “24小时承诺”  |  保证值，非典型值；典型为2-6小时 |
| 工作负载分类  |  “交互性决策”  |  交互式/半批量/批量路由决策 |
| 输出模式  |  “响应格式”  |  各提供商JSONL布局；不可移植 |
| 叠加折扣  |  “批处理+缓存”  |  两者都应用时约为未缓存同步账单的10% |
