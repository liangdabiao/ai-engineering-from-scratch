# 面向大语言模型的 FinOps —— 单位经济性与多租户归因

> 传统的 FinOps 在 LLM 支出上失效。成本是令牌交易，而非资源运行时长。标签无法映射——API 调用是一笔交易，而非资产。工程决策（提示设计、上下文窗口、输出长度）即是财务决策。2026 年的行动手册包含三个从第一天起就应该配置的归因维度：按用户（`user_id`）用于座位定价和扩展，按任务（`user_id` + `task_id`）用于产品表面成本和优先级排序，按租户（`user_id`）用于单位经济性和续约。四个令牌层——提示、工具、记忆、响应——单一桶会隐藏支出。多租户产品的执行阶梯：按租户的速率限制（预期峰值的 2-3 倍，明确的 429 + retry-after）；每日支出上限（合同上限的 1.5-3 倍；触发速率收紧 + 警报）；当支出 z 分数 > 4 时的终止开关（自动暂停 + 呼叫值班人员）。归因模式：标签聚合、遥测连接器（trace-ID → 计费；最高精度）、采样外推、基于模型的分配、事件溯源、实时流。单位指标：每次解决查询的成本，每次生成工件的成本——而非 $/M 个令牌。追溯性标记总会遗漏；在请求创建时即进行配置。

**类型:** 学习
**语言:** Python（标准库，带终止开关的玩具成本归因模拟器）
**前提条件:** 阶段 17 · 13（可观测性），阶段 17 · 14（缓存）
**时间:** ~60 分钟

## 学习目标

- 解释为什么传统的 FinOps（标签 + 层级）在 LLM 支出上失效，并列举三个新的归因维度。
- 列举四个令牌层（提示、工具、记忆、响应），并说明为什么单桶计费会隐藏成本。
- 为多租户产品设计一个执行阶梯（速率 → 支出上限 → 终止开关）。
- 选择一个单位指标（每次解决查询 / 工件的成本）而非 $/M 个令牌。

## 问题

你的账单显示 $40,000。你不知道：
- 哪个租户花掉了这笔钱。
- 哪个产品功能驱动了这笔费用。
- 是否有任何单个用户滥用。
- 是提示膨胀、工具调用还是记忆放大是罪魁祸首。

在云资源（EC2、S3）上，提供端侧的标签聚合有效，因为标签会传播到行项目。LLM 的 API 调用不会自动打标签——你必须在调用位置标记用户/任务/租户并传递下去。追溯性归因总会遗漏边缘情况。

## 核心概念

### 三个归因维度

**按用户**（`user_id`）：谁花费了什么。驱动座位定价、扩展对话，识别高能用户。

**按任务**（`task_id` + `route`）：哪个产品表面花费了多少。驱动功能优先级排序、终止昂贵功能的决策。

**按租户**（`tenant_id`）：哪个客户盈利。驱动单位经济性、续约定价、层级阈值。

在调用位置从第一天起配置所有三个维度。回溯总是更差。

### 四个令牌层

|  层  |  示例  |  占总成本典型百分比  |
|-------|---------|---------------------|
|  提示  |  系统 + 用户输入  |  40-60%  |
|  工具  |  工具调用结果反馈  |  20-40%（代理工作负载）  |
|  记忆  |  先前对话 / 检索文档  |  10-30%  |
|  响应  |  模型输出  |  10-30%  |

将所有四个层放在同一个桶中会使优化变得盲目。在你的归因模式中分开它们。

### 执行阶梯

1. **速率限制** 按租户。预期峰值的 2-3 倍。返回 429 及 `Retry-After`。租户看到摩擦；没有意外账单。

2. **每日支出上限** 按租户。合同上限的 1.5-3 倍。触发：收紧速率限制 + 通知客户成功团队。

3. **终止开关** 当支出 z 分数 > 4（相对于租户基线）时。自动暂停租户；呼叫值班人员；升级到运维和客户成功团队。

### 归因模式

- **标签聚合**：在元数据头中打标签；后期聚合。简单；粗略。
- **遥测连接器**：通过 trace ID 将追踪连接到计费。最高精度。成熟团队的做法。
- **采样 + 外推**：采样 5-10%，乘以比例。对于粗略支出成本效益高；遗漏尾部。
- **基于模型的分配**：回归推断成本驱动因素。适用于没有标签的历史数据。
- **事件溯源**：将成本作为流中的事件（Kafka / Kinesis）。实时。
- **实时流**：仪表板亚秒级更新。

### 每 X 的成本是单位指标

$/M 个令牌是供应商的说法。产品指标：

- 每次解决支持工单的成本。
- 每篇生成文章的成本。
- 每个成功代理任务的成本。
- 每个用户会话分钟的成本。

将成本与产品成果挂钩。否则优化就会失去锚点。

### 成本归因追踪形状

```
trace_id: abc123
  user_id: u_42
  tenant_id: t_7
  task_id: task_classify_doc
  route: model_haiku
  layers:
    prompt_tokens: 1800
    tool_tokens: 600
    memory_tokens: 400
    response_tokens: 150
  cost_usd: 0.0135
  cached_input: true
  batch: false
```

在每次调用时发出。存储在数据湖中。按维度聚合。阶段 17 · 13 可观测性栈是它的归属地。

### 复利储蓄堆栈

堆栈：缓存 + 批处理 + 路由 + 网关。四项结合：
- 缓存L2（阶段17·14）：输入成本降低约10倍。
- 批处理（阶段17·15）：节省50%。
- 路由到廉价模型（阶段17·16）：成本减少60%。
- 网关效率（阶段17·19）：冗余+重试。

最佳情况叠加：约为基准的5-10%。大多数团队使用2-3个杠杆；很少有团队叠加全部四项。

### 你应该记住的数字

- 归因维度：按用户、按任务、按租户。
- 四个令牌层：提示词、工具、记忆、响应。
- 终止开关：支出z得分>4。
- 单位指标：每个已解决查询的成本，而非美元/百万令牌。
- 叠加优化：可达到基准的约5-10%。

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|----------------|------------------------|
|  按用户归因  |  "用户级成本"  |  `user_id` 标记在每次调用上  |
|  按任务归因  |  "功能成本"  |  `task_id` + `route` 标识产品表面  |
|  按租户归因  |  "客户成本"  |  `tenant_id`；驱动单位经济  |
|  四个令牌层  |  "成本层"  |  提示词 + 工具 + 记忆 + 响应  |
|  速率限制  |  "429防护"  |  按租户上限在网关实施  |
|  每日支出上限  |  "每日限额"  |  租户范围的预算，带警报  |
|  终止开关  |  "自动暂停"  |  支出z得分>4触发自动暂停  |
|  每已解决成本  |  "产品单位指标"  |  成本与产品结果挂钩，而非令牌  |
|  遥测连接器  |  "追踪到计费"  |  最高准确度的归因模式  |
|  叠加优化  |  "缓存+批处理+路由+网关"  |  复合节省至基准的约5-10%  |
