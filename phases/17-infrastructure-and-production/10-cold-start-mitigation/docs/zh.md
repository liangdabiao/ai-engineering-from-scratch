# 无服务器大语言模型的冷启动缓解

> 一个20GB的模型镜像从冷启动到提供服务需要5-10分钟（7B模型）到20分钟以上（70B模型）。在真正的无服务器世界中，这不是预热——这是中断。缓解措施分为五个层次：预种子节点镜像（AWS上的Bottlerocket，双卷架构）、模型流式加载（NVIDIA Run:ai Model Streamer，原生集成在vLLM中）、GPU内存快照（Modal检查点，重启速度提升高达10倍）、热池（`min_workers=1`）、分层加载（ServerlessLLM的NVMe→DRAM→HBM管道，延迟降低10-200倍），以及实时迁移（移动输入令牌（KB）而非KV缓存（GB））。Modal发布的最低冷启动时间为2-4秒；Baseten默认5-10秒，预热后可达到亚秒级。本节课程教你如何测量、预算和堆叠这五个层次。

**类型：** 学习
**语言：** Python（标准库，冷启动路径模拟玩具）
**先修知识：** 阶段17·02（推理平台经济学），阶段17·03（GPU自动伸缩）
**时间：** 约60分钟

## 学习目标

- 列举冷启动缓解的五个层次，并在每个层次命名一个工具或模式。
- 计算70B模型的冷启动总时间，作为（节点供应）+（权重下载）+（权重加载到HBM）+（引擎初始化）之和。
- 解释为什么实时迁移传输输入令牌（KB）而不是KV缓存（GB），以及代价是什么（重新计算）。
- 指出热池的权衡（为空闲GPU付费或接受冷启动长尾）以及`min_workers > 0`成为强制要求的SLA阈值。

## 问题

你的无服务器LLM端点夜间缩容到零。早上8点流量激增。第一个请求等待以下步骤：

1. Karpenter 供应一个GPU节点：45-60秒。
2. 容器拉取一个30GB的镜像及权重：120-300秒。
3. 引擎将权重加载到HBM：45-120秒，取决于模型大小和存储速度。
4. vLLM或TRT-LLM初始化CUDA图、KV缓存池、分词器：10-30秒。

总计：返回一个令牌之前需要220-510秒（大约3-8分钟）。你的SLA是2秒。你部署了一个热池（`min_workers=1`），问题似乎消失了——但现在你要为一个空闲GPU支付24x7的费用。如果你的服务有5个产品，每个有一个热副本，那么无论是否有用户调用，你每月都要支付5×24×30=3,600 GPU小时。

冷启动缓解是指在保持无服务器经济性的同时，逼近始终在线延迟的方法。

## 核心概念

### 第一层——预种子节点镜像（Bottlerocket）

在AWS上，Bottlerocket的双卷架构将操作系统与数据分离。预拉取容器镜像后对数据卷进行快照；在`EC2NodeClass`中引用快照ID。新节点启动时，本地NVMe上已有权重——步骤2和步骤3的一部分消失。原生支持Karpenter。典型节省：大型模型每次冷启动节省2-4分钟。

在GCP上等效：使用预置容器层的自定义VM镜像。在Azure上：采用相同模式的管理磁盘快照。

### 第二层——模型流式加载（Run:ai Model Streamer）

无需在响应第一个请求之前加载完整文件，而是逐层将权重流式加载到GPU内存，并在第一个Transformer块驻留后立即开始处理。NVIDIA Run:ai Model Streamer原生集成在vLLM 2026中。支持S3、GCS和本地NVMe。通过将I/O与计算设置重叠，大型模型的权重加载时间大约减半。

### 第三层——GPU内存快照（Modal）

Modal在首次加载后对GPU状态（权重、CUDA图、KV缓存区域）创建检查点。后续重启时直接反序列化到HBM——比重新初始化快10倍。这最接近于"在2秒内启动一个热GPU"。权衡：快照是针对每个GPU拓扑的，因此如果Karpenter将你迁移到不同的SKU，你需要重新创建检查点。

### 第四层——热池（min_workers=1）

最简单的缓解措施：始终保持一个副本就绪。成本是一个GPU的每小时费用，24x7。对于小型模型来说，这笔账很残酷（你支付$0.85-$1.50美元/小时来避免30秒的冷启动），但对于大型模型则较友好（支付4美元/小时来避免5分钟的冷启动）。热池成为强制要求的SLA阈值：通常在70B+模型上，TTFT P99 < 60秒。

### 第五层——分层加载（ServerlessLLM）

ServerlessLLM将存储视为层次结构：NVMe（快速但容量大）、DRAM（中等但分层次）、HBM（小但即时）。权重预加载到DRAM；按需加载到HBM。论文报告，与简单的磁盘到HBM相比，冷加载延迟降低10-200倍。生产环境采用尚早，但已存在与vLLM的集成。

### 第六层——实时迁移（附加模式）

当节点不可用时（抢占实例回收、节点腾空），传统模式是冷启动另一个副本并清空请求队列。实时迁移将输入令牌（KB）移动到已加载模型的目标节点，并在目标节点重新计算KV缓存。重新计算比通过网络传输GB级KV缓存更便宜。适用于分离式部署。

### 热池的账目

对于P99 TTFT SLA为2秒的服务，问题不是"是否使用热池"，而是"使用多少个热副本，以及哪些路径获得它们"。

- 高价值交互路径（实时聊天、语音代理）：`min_workers=1-2`。
- 后台批处理路径（夜间分类）：接受缩容到零，可容忍5-10分钟的冷启动。
- 高级套餐：每个租户`min_workers=1-2`，配备专用容量。

### 先测量，再优化

70B模型在新节点上的冷启动剖析（示意）：

|  阶段  |  时间  |  缓解措施  |
|-------|------|-----------|
|  节点供应  |  50s  |  Bottlerocket + 预种子镜像，热池  |
|  镜像拉取  |  180s  |  预种子数据卷（消除）  |
|  权重到HBM  |  75s  |  模型流式加载（减半）；GPU快照（消除）  |
|  引擎初始化  |  20s  |  持久化CUDA图缓存  |
| 首先前向传播 | 3s | 最小固有延迟 |
| **总冷启动** | **328s** |  |
| **采用缓解措施后总时间** | **~15s** | 减少22倍 |

### 你应该记住的数字

- Modal冷启动：2-4秒（使用GPU快照）。
- Baseten默认冷启动：5-10秒；预热后可亚秒级。
- 原始70B冷启动：3-8分钟。
- Run:ai模型流式加载：权重加载速度提升约2倍。
- ServerlessLLM分层加载：延迟降低10-200倍（论文数据）。

## 练习

1. 运行`code/main.py`。计算盈亏平衡请求率，超过该请求率，热副本比因在SLO处丢弃额外请求而支付冷启动税更便宜。
2. 你部署了一个13B模型，P99 TTFT SLA为3秒。选择实现该目标的最小缓解措施堆叠（层次最少）。
3. Bottlerocket预播种消除了镜像拉取，但权重仍从快照加载到HBM。如果快照支持的NVMe读取速度为7 GB/s，计算70B模型的挂钟时间。
4. 你的无服务器提供商提供GPU快照（Modal），你的团队拒绝，理由是“快照泄露PII”。从双方论证——实际风险是什么？缓解措施是什么（临时快照、加密、命名空间隔离）？
5. 设计一个分层热池策略：付费用户、试用用户和批处理工作负载各有多少热副本？展示数学计算。

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|----------------|------------------------|
| 冷启动 | "大停顿" | 从请求到在新副本上生成第一个令牌的时间 |
| 热池 | "始终在线的最小量" | `min_workers >= 1` 保持至少一个副本就绪 |
| 预播种镜像 | "烘焙好的AMI" | 节点镜像，其中容器权重预置 |
| Bottlerocket | "AWS节点操作系统" | 具有双卷快照支持的AWS容器优化操作系统 |
| 模型流式加载 | "流式加载" | 将权重I/O与计算设置重叠 |
| GPU快照 | "检查点到HBM" | 序列化加载后的GPU状态；重启时反序列化 |
| 分层加载 | "NVMe + DRAM + HBM" | 存储层级层次结构；按需加载 |
| 实时迁移 | "移动令牌" | 传输输入（KB），在目标端重新计算KV |
| `min_workers` | "热副本" | 无服务器最小保持存活数量 |
| 缩放到零 | "完全无服务器" | 空闲时无成本；接受完整的冷启动代价 |
