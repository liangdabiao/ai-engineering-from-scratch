# A/B 测试 LLM 功能 —— GrowthBook、Statsig 与主观感受问题

> 传统的A/B测试并非为不确定性LLM设计。关键区别：评估回答“模型能否完成任务？”A/B测试回答“用户在乎吗？”两者都必要；仅凭感觉上线已过时。2026年需要测试的内容：提示工程（措辞）、模型选择（GPT-4 vs GPT-3.5 vs 开源；准确率 vs 成本 vs 延迟）、生成参数（温度、top-p）。真实案例：某个聊天机器人奖励模型变体带来了+70%的对话长度和+30%的留存率；Nextdoor的AI主题行实验在奖励函数优化后带来了+1%的点击率；Khan Academy的Khanmigo在延迟与数学准确率之间迭代平衡。平台对比：**Statsig**（2025年9月被OpenAI以11亿美元收购）——顺序测试、CUPED、一站式。**GrowthBook**——开源、数据仓库原生、贝叶斯+频率派+顺序引擎、CUPED、SRM检查、Benjamini-Hochberg + Bonferroni校正。根据数据仓库SQL偏好以及“被OpenAI收购”对组织是否重要来选择。

**类型：** 学习
**语言：** Python（标准库，玩具顺序测试模拟器）
**前置要求：** 阶段17 · 13（可观测性），阶段17 · 20（渐进式部署）
**时间：** 约60分钟

## 学习目标

- 区分评估（“模型能否完成任务”）与A/B测试（“用户在乎吗”）。
- 列举三个可测试的维度（提示、模型、参数）并为每个维度选择指标。
- 解释CUPED、顺序测试和Benjamini-Hochberg多重比较校正。
- 根据数据仓库SQL姿势和公司收购立场选择Statsig或GrowthBook。

## 问题

你手动调优了一个系统提示。感觉更好了。你上线了。转化指标因噪声而变化。你责怪指标。或者你上线了一个新模型，转化没有变化——是模型退化了还是变化太小无法检测？你不知道，因为你没有做A/B测试就上线了。

评估回答的是模型能否在标记数据集上完成任务。它们不能回答用户是否喜欢输出。只有受控的在线实验能回答这个问题，而且只有当实验有足够的统计功效、控制不确定性并进行多重比较校正时才行。

## 核心概念

### 评估 vs A/B测试

**评估** — 离线、标记数据集、评判者（评分标准或LLM作为评判者或人类）。回答：“在这个固定分布上，输出是否正确/有用/安全？”

**A/B测试** — 在线、真实用户、随机化。回答：“新变体是否提升了用户层面的重要指标？”

两者都必要。评估在用户接触前发现回归；A/B测试确认产品影响。

### 测试什么

1. **提示工程** — 措辞、系统提示结构、示例。指标：任务成功、用户留存、每次请求成本。
2. **模型选择** — GPT-4 vs GPT-3.5-Turbo vs Llama开源。指标：准确率（任务）+ 每次请求成本 + P99延迟。多目标。
3. **生成参数** — 温度、top-p、最大令牌数。指标：任务特定（输出多样性 vs 确定性）。

### CUPED — 方差缩减

使用实验前数据控制实验。在比较实验后期间回归掉实验前期的方差。典型方差缩减：30-70%。有效样本量免费增加。

实现：Statsig和GrowthBook都实现了。

### 顺序测试

经典A/B测试假设固定样本量。顺序测试（“偷看并决定”）在多次查看下控制假阳性率。始终有效的顺序程序（mSPRT、Howard的置信序列）允许在明确获胜者时提前停止。

### 多重比较校正

以95%置信水平运行20个A/B测试，预期会有一个假阳性。Bonferroni校正收紧每个测试的α；Benjamini-Hochberg控制错误发现率。GrowthBook实现了两者。

### SRM — 样本比例不匹配

分配哈希将用户随机分配到变体。如果是50/50的分配，结果却是47/53，说明有问题——SRM检查会标记。两个平台都实现了。

### Statsig vs GrowthBook

**Statsig：**
- 被OpenAI以11亿美元收购（2025年9月）。托管、SaaS。
- 顺序测试、CUPED、保留样本。
- 一站式：功能标志 + 实验 + 可观测性。
- 最适合：团队已经想要捆绑产品，不在乎OpenAI所有权。

**GrowthBook：**
- 开源（MIT）；数据仓库原生（直接读取Snowflake/BigQuery/Redshift）。
- 多引擎：贝叶斯、频率派、顺序。
- CUPED、SRM、Bonferroni、BH校正。
- 自托管或托管云。
- 最适合：数据仓库SQL团队，数据团队控制指标层，想要开源。

### 不确定性使功效计算复杂化

相同提示产生不同输出。传统功效计算假设独立同分布观测。由于LLM的不确定性，有效样本量低于名义值。将所需样本量乘以约1.3-1.5倍作为安全裕度。

### 真实案例结果

- 聊天机器人奖励模型变体：+70%对话长度，+30%留存率。
- Nextdoor主题行：奖励函数优化后+1%点击率。
- Khan Academy的Khanmigo：迭代延迟与数学准确率的权衡。

### 反模式：凭感觉上线

每个资深工程师都能举出一个因为“感觉更好”而没有做A/B测试就发布的功能。其中大多数功能导致团队数月未注意到的产品指标恶化。A/B测试是强制机制。

### 你应该记住的数字

- Statsig被OpenAI收购：11亿美元，2025年9月。
- GrowthBook：开源MIT许可；贝叶斯+频率学派+序贯。
- CUPED方差缩减：30-70%。
- LLM非确定性→样本量缓冲增加30-50%。

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|----------------|------------------------|
|  评估  |  “离线测试”  |  对模型能力的标注集评估  |
|  A/B测试  |  “实验”  |  在用户上的实时随机化比较  |
|  CUPED  |  “方差缩减”  |  预处理期回归以降低方差  |
|  序贯检验  |  “可窥探检验”  |  总是有效的程序，允许提前停止  |
|  多重比较  |  “家庭错误”  |  运行多个测试会增加假阳性  |
|  邦费罗尼校正  |  “严格校正”  |  将α除以测试数量  |
|  Benjamini-Hochberg  |  “BH FDR”  |  错误发现率控制，不那么保守  |
|  SRM  |  “错误分组”  |  样本比例不匹配；分配错误  |
|  Statsig  |  “OpenAI所有”  |  商业一体化平台，2025年被收购  |
|  GrowthBook  |  “开源的那个”  |  MIT许可的仓库原生平台  |
|  mSPRT  |  “序贯概率比检验”  |  经典序贯程序  |
