# 提示缓存与语义缓存的经济学

> **定价快照日期为 2026-04。** 下面的数字索赔反映了本课程发布时的供应商价格表；在下游引用之前，请对照链接的文档进行验证。

> 缓存发生在两个层。L2（提供商级别）提示/前缀缓存会为重复的前缀重用注意力KV——Anthropic 的提示缓存文档宣称，对于长提示，成本可降低高达 90%，延迟降低高达 85%；对于 Claude 3.5 Sonnet，缓存读取为 $0.30/M vs $3.00/M（新鲜），TTL 为 5 分钟，对于 1 小时 TTL 选项，写入溢价为 2 倍（docs.anthropic.com，2026-04）。OpenAI 提示缓存自动应用于 ≥1024 个令牌的提示，缓存输入的价格大约是新鲜输入的 90% 折扣（platform.openai.com，2026-04）；每个模型的确切缓存费率取决于实时价格表。L1（应用级别）语义缓存在嵌入相似性命中时完全跳过 LLM。供应商声称的“95% 准确率”指的是匹配正确性，而非命中率——报告的生成命中率范围从 10%（开放式聊天）到 70%（结构化 FAQ）；任何提供商均未发布官方基线，因此请将这些视为社区遥测数据，而非保证。生产陷阱：并行化会破坏缓存（在第一次缓存写入之前发出的 N 个并行请求可能会使支出增加数倍），前缀内的动态内容会完全阻止缓存命中。ProjectDiscovery 报告称，通过将动态文本移出可缓存前缀，命中率从 7% 提高到 74%（2025-11）。

**类型：** 学习
**语言：** Python（标准库，玩具级双层缓存模拟器）
**先决条件：** 阶段 17 · 04（vLLM 服务内部机制），阶段 17 · 06（SGLang RadixAttention）
**时间：** 约 60 分钟

## 学习目标

- 区分 L2 提示/前缀缓存（提供商处的 KV 重用）和 L1 语义缓存（基于相似提示绕过 LLM）。
- 解释 Anthropic 的 `cache_control` 显式标记以及两种 TTL 选项（5 分钟与 1 小时）及其价格乘数。
- 根据命中率、提示/回答组合和令牌价格计算预期的月度节省。
- 指出导致账单膨胀 5-10 倍的并行化反模式和导致命中率骤降的动态内容反模式。

## 问题

您为 RAG 服务添加了提示缓存。账单保持不变。您测量了命中率；它是 7%。您的提示看起来是静态的，但实际上并非如此——系统提示包含格式化为分钟级别的当前日期、一个请求 ID 以及一个用于多样性的随机化示例重排。每个请求都会写入一个新的缓存条目，读取为零。

另外，您的代理针对每个用户问题运行十个并行工具调用。所有十个请求都在第一次缓存写入完成之前到达提供商。十次写入，零次读取。您的账单是“启用缓存”应该花费的 5-10 倍。

缓存是一种协议，而不是一个标志。两个层，两种不同的故障模式。

## 核心概念

### L2——提供商提示/前缀缓存

提供商存储可缓存前缀的注意力 KV，并在下一个匹配该前缀的请求中重用。您支付一次写入成本，读取几乎免费。

**Anthropic（Claude 3.5 / 3.7 / 4 系列）**：请求中的显式 `cache_control` 标记。您标记哪些块是可缓存的。TTL：5 分钟（写入成本为基础价格的 1.25 倍）或 1 小时（写入成本为基础价格的 2 倍）。缓存读取：$0.30/M on Claude 3.5 Sonnet vs $3.00/M 新鲜——便宜 10 倍（docs.anthropic.com，截至 2026-04）。不同模型的费率不同（Opus/Haiku 单独发布）；始终核对实时定价页面。

**OpenAI**：对于 ≥1024 个令牌的提示自动缓存（platform.openai.com，2026-04）。没有显式标记。在当前 gpt-4o/gpt-5 价格表上，缓存输入大约比新鲜输入便宜 10 倍。文档和发行说明均未发布官方命中率基线；社区报告在精心设计提示的情况下集中在 30–60%。通过监控 @@SKOP0000@@ 来衡量您自己的命中率。

**Google（Gemini）**：通过显式 API 进行上下文缓存；100 万个令牌的上下文意味着缓存节省更多。

**自托管（vLLM, SGLang）**：阶段 17 · 06 涵盖 RadixAttention——在您自己的计算资源上采用相同模式。

### L1——应用级别语义缓存

在调用 LLM 之前，对提示进行哈希处理，嵌入它，并查找相似的缓存请求（余弦相似度高于阈值，通常为 0.95+）。如果命中，返回缓存的响应。如果未命中，调用 LLM 并缓存结果。

开源：Redis 向量相似性、GPTCache、Qdrant。商业：Portkey Cache、Helicone Cache。

供应商声称的准确率指的是返回的缓存响应在语义上合适的频率——而不是命中频率。生产命中率：

- 开放式聊天：10-15%。
- 结构化 FAQ/支持：40-70%。
- 代码问题：20-30%（微小变体会破坏命中）。
- 重复提示的语音代理：50-80%（语音规范化固定集合）。

### 并行化反模式

您的代理并行发出 10 个工具调用。所有 10 个具有相同的 4K 令牌系统提示。Anthropic 缓存写入是每个请求进行的；第一次缓存写入在提供商看到提示后大约 300 毫秒完成。请求 2-10 在同一毫秒窗口内到达，每个都看到缓存未命中。您支付 10 次写入溢价，0 次读取折扣。

修复：使用顺序优先的批处理——先单独发出请求 1，然后在请求 1 的缓存填充后再发出请求 2-10。这会为第一个工具调用增加 300 毫秒；节省 5-10 倍的账单。

### 动态内容反模式

您的系统提示看起来像：

```
You are a helpful assistant. The current time is 14:32:17.
User ID: abc123. Today is Tuesday...
```

每个请求都是唯一的。每个请求都写入。零命中。

修复：将所有真正静态的内容移动到可缓存前缀；在缓存边界之后附加动态内容：

```
[cacheable]
You are a helpful assistant. [rules, examples, instructions]
[/cacheable]
[dynamic, not cached]
Current time: 14:32:17. User: abc123.
```

ProjectDiscovery 通过这种方式将缓存命中率从 7% 提高到 74%，并发布了详细分析。

### 将批处理与缓存叠加用于过夜工作负载

批处理 API（阶段 17 · 15）在 24 小时周转时间内提供 50% 折扣。在此基础上叠加缓存输入可再获得约 10 倍的折扣。通过叠加，过夜分类、标记和报告生成工作负载的成本可以降低到同步非缓存成本的约 10%。

### 你应该记住的数字

定价点截至 2026-04 从链接的供应商文档中捕获，每隔几个月就会变动——在依赖它们之前请重新检查。

- Anthropic 缓存读取：Claude 3.5 Sonnet 上为 $0.30/M，大约比新鲜输入便宜 10 倍（docs.anthropic.com）。
- Anthropic 缓存写入溢价：1.25 倍（5 分钟 TTL）或 2 倍（1 小时 TTL）。
- OpenAI 自动缓存：适用于 ≥1024 个令牌的提示；根据当前价格表，缓存输入的定价约为新鲜输入的 10%（platform.openai.com）。
- 语义缓存命中率（社区报告）：开放式聊天约 10%；结构化 FAQ 最高可达 70%。不是供应商记录的基线。
- ProjectDiscovery：通过将动态内容移出前缀，命中率从 7% 提高到 74%（项目博客，2025-11）。
- 并行化反模式：当 N 个并行请求错过第一次缓存写入时，通常报告导致账单膨胀 5-10 倍。

## 练习

1. 运行 `code/main.py`。切换并行化标志。费用变化多少？
2. 你的系统提示中包含日期。将其移出。展示前后命中率计算。
3. 给定请求到达率，计算1小时 TTL（2倍写入）与5分钟 TTL（1.25倍写入）的盈亏平衡点。
4. 语义缓存在0.95阈值下命中率为20%。在0.85下命中率为50%，但你会看到错误的缓存响应。选择合适的阈值并证明其合理性。
5. 你对每个用户问题批处理10个并行子查询。重写为缓存友好而不增加端到端延迟。

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|----------------|------------------------|
|  L2 提示缓存  |  "前缀缓存"  |  提供商为重复前缀存储 KV  |
|  `cache_control`  |  "Anthropic 缓存标记"  |  显式属性标记可缓存块  |
|  缓存写入溢价  |  "写入税"  |  首次未命中到缓存的额外成本（1.25倍或2倍）  |
|  L1 语义缓存  |  "嵌入缓存"  |  在调用 LLM 前的应用层哈希与嵌入  |
|  GPTCache  |  "LLM 缓存库"  |  流行的开源 L1 缓存库  |
|  缓存命中率  |  "命中数/总数"  |  从缓存服务的请求比例  |
|  并行化反模式  |  "N次写入陷阱"  |  N个并行请求导致N次缓存未命中  |
|  动态内容陷阱  |  "提示中时间陷阱"  |  前缀中的动态字节破坏命中率  |
|  RadixAttention  |  "副本内缓存"  |  SGLang 的前缀缓存实现  |
