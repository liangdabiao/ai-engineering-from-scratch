# AI 的 SRE —— 多智能体事件响应、运维手册、预测性检测

> AI SRE（AI站点可靠性工程）利用基于基础设施数据（日志、操作手册、服务拓扑）的LLM，通过RAG（检索增强生成）自动化调查、文档编制和协调阶段。2026年的架构模式是多智能体编排 —— 由监督者协调专门的智能体（日志、指标、操作手册）；AI提出假设和查询，人类批准决策。Datadog Bits AI和Azure SRE Agent以托管产品形式提供此功能。操作手册正在演变：NeuBird Hawkeye使用对抗性评估（两个模型分析同一事件；一致=置信，不一致=不确定）；操作记忆在团队变动中持续存在。自动修复保持谨慎：AI建议，人类批准。完全自主行动范围狭窄（重启Pod、回滚特定部署），并设有严格的防护栏 —— 任何推销“设置后即可遗忘”的都是过度宣传。新兴前沿：事件前预测。MIT研究报告称，一个在历史日志、GPU温度和API错误模式上训练的LLM提前10-15分钟预测了89%的中断。预测：到2026年底，95%的企业LLM将拥有自动故障转移功能。

**类型：** 学习
**语言：** Python（标准库，玩具多智能体事件分类模拟器）
**前置条件：** 阶段17·13（可观测性），阶段17·24（混沌工程）
**时间：** 约60分钟

## 学习目标

- 绘制多智能体AI SRE架构图：监督者 + 专门智能体（日志、指标、操作手册）+ 人类批准门。
- 解释为什么自动修复范围狭窄（重启Pod、回滚部署）而非广泛（重新架构服务）。
- 命名对抗性评估模式（NeuBird Hawkeye）：两个模型一致 = 置信；不一致 = 升级。
- 引用MIT 89%早期检测结果和操作约束：没有行动预测只是仪表盘。

## 问题

一位值班工程师在凌晨3点收到告警：“结账错误率高”。他们检查Datadog、Loki、三本操作手册、部署日志。30分钟后，他们意识到根本原因是vLLM OOM（显存溢出）源于KV缓存峰值。他们重启Pod；错误消失。

到2026年，调查的前20分钟可以自动化。按服务分组日志、关联最近部署、匹配操作手册——这些都是RAG + 工具使用。监督智能体可以在人类打开Datadog之前执行初步分类并呈现假设。

完全自主修复是另一个问题。重启Pod：安全。扩展GPU池：如果策略允许则安全。重新架构服务：绝对不行。纪律是划定狭窄界限。

## 核心概念

### 多智能体架构

```
          Incident
             │
             ▼
        Supervisor
        /    |    \
       ▼     ▼     ▼
  Log agent  Metric agent  Runbook agent
       │     │     │
       └─────┴─────┘
             │
             ▼
        Hypothesis + evidence
             │
             ▼
        Human approval
             │
             ▼
        Action (narrow set)
```

监督者将事件分解为子查询。专门智能体拥有工具访问权限（日志搜索、PromQL、文档检索）。监督者综合，向人类呈现假设和证据。人类批准或重定向。

### 自动修复范围

**安全（狭窄）**：重启Pod、回滚特定部署、在预先批准范围内扩展池、启用预先批准的功能标志。

**不安全（广泛）**：更改服务拓扑、修改资源限制、部署新代码、更改IAM、修改数据库。

任何推销“设置后即可遗忘”的都是过度宣传。随着AI SRE的发展，安全集合会扩大，但界限是真实的。

### 对抗性评估（NeuBird Hawkeye）

两个模型独立分析同一事件。如果它们对根本原因达成一致，置信度很高。如果不一致，则升级给人类，同时显示两个假设。简单模式，有效过滤幻觉根本原因。

### 操作记忆

团队更替是传统SRE的无声杀手——隐性知识流失。AI SRE将操作手册和事后复盘存储在向量数据库中；智能体在每次新事件时检索。当新工程师加入时，AI拥有完整历史。

### 事件前预测

MIT 2025年研究：在历史日志、GPU温度、API错误模式上训练的LLM在测试集上提前10-15分钟预测了89%的中断。

现实检验：没有行动的预测只是仪表盘。操作问题是“当我们预测时，我们做什么？”预先排空？告警？自动扩展？答案取决于策略。

### 2026年的产品

- **Datadog Bits AI** —— Datadog内部的托管SRE助手。
- **Azure SRE Agent** —— Azure原生。
- **NeuBird Hawkeye** —— 对抗性评估 + 操作记忆。
- **PagerDuty AIOps** —— 分类 + 去重。
- **Incident.io Autopilot** —— 事件指挥官 + 协调。

### 操作手册即代码

操作手册从Confluence页面演变为版本控制的Markdown，具有结构化部分（症状、假设、验证、行动）。结构化的操作手册有助于更好的RAG检索。任何AI SRE的部署都应从将非结构化操作手册变为结构化开始。

### 你应该记住的数字

- MIT早期检测：89%的中断，提前10-15分钟。
- 多智能体分类：监督者 + （日志、指标、操作手册）+ 人类。
- 安全自动修复集合：重启Pod、回滚部署、在范围内扩展。
- 对抗性评估：两个模型独立；一致 = 置信。

## 使用它

`code/main.py` 模拟多智能体分类：日志智能体发现错误，指标智能体发现CPU峰值，操作手册智能体匹配已知问题。监督者排序假设。

## 练习

1. 运行 `code/main.py`。如果日志和指标智能体意见不一致，监督者如何解决？
2. 为你的服务定义三个“安全”的自动修复操作。证明每个合理性。
3. 编写结构化操作手册模板：部分、必填字段、验证命令。
4. 预测检测在提前12分钟触发。你的策略是什么——告警、预先排空，还是两者兼有？
5. 论证一个3人团队是否应在2026年采用AI SRE还是等待。考虑成熟度、量级、风险。
