# 检查点与回滚

> 每个图状态转换都会持久化。当工作进程崩溃时，其租约过期，另一个工作进程会在最新的检查点处接续。Cloudflare Durable Objects 可跨数小时或数周保持状态。先提议后提交（第15课）为每个动作定义了回滚计划。动作后验证完成闭环。《欧盟人工智能法案》第14条要求高风险系统必须进行有效的人类监督——在实践中，这意味着检查点必须可查询，回滚必须经过演练，审计跟踪必须在部署后仍然存在。严峻的故障模式是：如果没有幂等键和前置条件检查，一次临时故障后的重试可能导致已批准的动作被重复执行。动作后验证正是用来捕获这种情况的。

**类型:** 学习
**语言:** Python（标准库，检查点和回滚状态机）
**先决条件:** 阶段15 · 12（持久化执行），阶段15 · 15（先提议后提交）
**时间:** 约60分钟

## 问题

持久化执行（第12课）使崩溃的代理可恢复。先提议后提交（第15课）使已批准的动作可审计。本课将两者结合：当一个已批准的动作部分执行、崩溃并恢复时会发生什么？回滚何时运行，以及针对什么状态运行？

实际系统采用不同的方式实现：

- **LangGraph** 将每个图状态转换检查点到 PostgreSQL。工作进程崩溃时，租约释放，另一个工作进程在最新检查点处恢复。工作流在 `interrupt()` 处暂停，该本身也会持久化。
- **Cloudflare Durable Objects** 跨数小时或数周保持每个键的状态。将计算与已批准动作的存储放在一起。
- **Microsoft Agent Framework** 在工作流 API 中暴露 `interrupt()` 原语；重放加幂等性覆盖重试。

在每种情况下，实际有效的方法是：幂等键（防止重复执行）+ 前置条件检查（状态仍与我们批准的一致）+ 动作后验证（副作用实际发生）+ 验证失败时回滚。

## 核心概念

### 每个转换都持久化

图状态转换是将工作流从一个命名状态移动到另一个命名状态的任何步骤。简单的实现只在特定的提交点持久化；生产级实现持久化每个转换。其成本（少量额外的写入）相对于可靠性增益（重放可定位到任意位置，租约恢复精确）很小。

### 租约恢复

当工作进程崩溃时，工作流不会丢失；租约（声明此工作进程正在执行此运行的短期声明）只是过期。另一个工作进程会获取最新检查点并恢复。租约机制正是让生产系统在滚动部署期间不丢失进行中工作的原因。

### 幂等性加前置条件

仅靠幂等性是不够的。考虑：一个工作流被批准“转账 $100 from A to B when balance > $1000”。工作流已提交，在中间执行时崩溃，然后恢复。如果只检查幂等键，然后执行恢复，则转账执行一次（正确）。但考虑在崩溃和恢复之间，另一个工作流将A的余额降至 $500。幂等性检查仍然通过，但前置条件不通过。没有前置条件检查，我们就会导致透支。

每个关键动作都需要两者：

- **幂等键**：防止重复执行。
- **前置条件检查**：确认状态仍与批准时一致。

### 执行后验证

"工具返回200" 不是验证。真正的验证会重新读取目标状态，并确认副作用确实发生了。模式如下：

- 数据库更新：`UPDATE ... RETURNING *` 然后断言返回的行与预期状态匹配。
- 发送邮件：提交后在已发送文件夹中查找消息ID。
- 文件写入：重新读取文件并计算哈希。
- API 调用：对目标资源进行后续 `UPDATE ... RETURNING *`。

如果验证失败，工作流处于已知的坏状态。回滚开始。

### 回滚计划

先提议后提交（第15课）中的每个关键动作都附带一个回滚计划。类型包括：

- **带内回滚**：直接逆转副作用（`DELETE` 后 `INSERT`，`Send-correction-email` 后 send）。
- **补偿事务**：一种新的动作，用于中和原始动作（标准 SAGA 模式）。
- **带外回滚**：向人类发出警报，暂停工作流，将坏状态留待调查。

空操作回滚（“我们无法撤销此操作”）必须在提议中明确说明。没有回滚的动作在提交时需要更强的人机协同（第15课挑战-响应）。

### 《欧盟人工智能法案》第14条的操作性解读

第14条要求对高风险系统进行“有效的人类监督”。在操作层面，实施者将其解读为：

- 检查点可供审计员查询。
- 回滚需要经过演练（至少端到端测试一次）。
- 审计跟踪在部署后仍然存在（检查点后端不是临时的）。
- 失败的验证会发出警报，而不是默默记录。

一个在提交中途崩溃、恢复并完成副作用，但没有验证加回滚路径的工作流，无法通过第14条测试。

### 严峻的故障模式：重复执行

该领域最常见的生产事件：

1. 动作已批准，幂等键 k。
2. 提交开始，执行，返回200。
3. 工作流在持久化“已提交”状态之前崩溃。
4. 工作流恢复；看到“已批准但未提交”；重新执行。
5. 副作用触发两次。

缓解措施：在执行前持久化一个“进行中”意图，使用幂等键执行，然后仅在动作后验证成功后才标记为“已提交”。如果动作触发但状态写入失败，你知道需要验证并（如有必要）重新触发。如果状态写入成功但动作失败，你则通过恢复路径验证并精确触发一次。

## 使用它

`code/main.py` 实现了一个带检查点的、具有幂等性、前置条件、验证和回滚的工作流。驱动模拟了四种场景：干净运行、崩溃后重试（幂等性捕获）、前置条件失败（工作流产而不触发）、验证失败（触发回滚）。

## 发布

`outputs/skill-rollback-rehearsal.md` 为提议的工作流设计了一个回滚排练测试，并审计检查点后端以确保审计追踪的持久性。

## 练习

1. 运行 `code/main.py`。验证四种场景。对于提交期间崩溃的情况，确认该操作在重试中恰好触发一次。

2. 修改“先标记完成，再执行”模式，使得状态写入在操作之后触发。重新运行崩溃场景。测量有多少重复操作被触发。

3. 为特定的生产操作（例如“发布到 Slack 频道”）设计一个回滚计划。分类为带内、补偿或带外。证明选择的合理性。

4. 以你熟悉的一个工作流为例。识别每个状态转换。为每个状态转换标记持久性要求（持久化/不持久化）。统计当前未持久化的数量。

5. 排练回滚测试：设计一个端到端测试，运行真实工作流，使其崩溃，并确认回滚路径被触发。该测试断言什么？
