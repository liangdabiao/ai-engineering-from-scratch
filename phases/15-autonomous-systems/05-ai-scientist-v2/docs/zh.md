# AI Scientist v2 — 工作坊级别的自主研究

> Sakana 的 AI Scientist v2（Yamada 等人，arXiv:2504.08066）运行完整的研究循环：假设、代码、实验、图表、论文撰写、提交。它是第一个有生成的论文通过 ICLR 2025 工作坊同行评审的系统。独立评估（Beel 等人）发现 42% 的实验因编码错误失败，文献综述经常将已有概念误标为新概念。Sakana 自己的文档警告代码库会执行 LLM 编写的代码，并建议使用 Docker 隔离。两种情况都是关键点。

**类型：** 学习
**语言：** Python（标准库，研究循环状态机玩具）
**前置条件：** 阶段 15 · 03（AlphaEvolve），阶段 15 · 04（DGM）
**时间：** 约 60 分钟

## 问题

研究是一项开放式任务。与 AlphaEvolve 的算法搜索或 DGM 的基准约束自我修改不同，研究结果没有机器可检查的正确性标准。论文由评审者判断，而非单元测试。这使得循环更难闭合——但如果闭合了，则更有价值，因为研究是复合进步的所在。

AI Scientist v1（Sakana，2024）通过从人类编写的模板开始来闭合循环。LLM 在固定框架内填充实验。AI Scientist v2（Yamada 等人，2025）通过使用基于代理的树搜索和视觉语言模型批评循环移除了模板需求。该系统生成想法、实施实验、制作图表、撰写论文，并根据评审反馈进行迭代。

同行评审结论：一篇 v2 生成的论文被 ICLR 2025 工作坊接受（已披露）。独立评估结论：该系统远非可靠。两者都是事实。

## 核心概念

### 架构

1. **想法生成。** LLM 根据主题和先前文献提出研究想法。v1 使用模板；v2 在假设空间上进行代理搜索。
2. **新颖性检查。** 文献检索步骤检查该想法是否已发表。这是 Beel 等人评估中发现误标的步骤——已有方法经常被分类为新方法。
3. **实验计划。** 代理起草实验协议并编写代码。
4. **执行。** 代码在沙箱中运行。失败会被反馈到重试循环中。在 Beel 等人的测量中，42% 的实验在此阶段因编码错误失败。
5. **图表生成。** 视觉语言模型读取生成的图表并重写以提高清晰度。这是 v2 的关键技术新增。
6. **论文撰写。** LLM 起草论文，并与内部评审者进行迭代。
7. **可选：提交。** 论文提交到会议。

### 工作坊接受结果的意义

一篇 v2 生成的论文通过了 ICLR 2025 工作坊的同行评审。作者向程序委员会披露了论文的起源。接受是一个数据点；它并不代表可以声称该系统“做研究”。

重要背景：工作坊论文的门槛低于主会议论文。同行评审存在噪声；任何一天都有一小部分提交被接受。一次成功是概念验证，而非可靠性声明。Nature 2026 论文记录了端到端循环，并且本身是由人类研究人员合著的；它并非“系统写了 Nature 论文”。

### 独立评估的发现

Beel 等人（arXiv:2502.14297）进行了外部评估。主要发现：

- **实验失败。** 42% 的实验因编码错误（错误导入、形状不匹配、未定义变量）失败。重试循环捕获了部分，而非全部。
- **新颖性误标。** 文献检索步骤经常将已有概念标记为新概念。这是研究领域的幻觉等效物。
- **呈现质量差距。** 视觉语言图表批评生成了可发表级别的可视化，掩盖了底层实验缺陷。

最后一点对于本阶段很重要。一个生成令人信服输出但不做令人信服研究的系统，比一个明显失败的系统更危险，而非更安全。评估必须深入底层主张，而非止步于图表。

### 沙箱逃逸风险

Sakana 自己的仓库 README 警告：

> 由于本软件的性质（执行 LLM 生成的代码），我们无法保证安全。存在危险包、不受控的网络访问和生成意外进程的风险。使用风险自负，并考虑 Docker 隔离。

这是自治在未验证领域中的操作形态。LLM 编写代码；代码运行；代码可以做进程被允许做的任何事情。如果没有硬性限制文件系统、网络和进程操作的沙箱，任何自我导向的研究代理都可以窃取数据、消耗计算资源或重写自身。

AlphaEvolve 的沙箱故事更容易，因为其评估器很严格。AI Scientist v2 的循环运行开放式代码和开放式目标。这就是它需要更强隔离（最低 Docker；优选 seccomp/gVisor）并在每次提交离开系统前进行人工审查的原因。

### v2 在前沿堆栈中的位置

|  系统  |  目标  |  输出类型  |  评估器  |  已知失败  |
|---|---|---|---|---|
|  AlphaEvolve  |  算法  |  代码  |  单元+基准  |  受评估器严格度限制  |
|  DGM  |  代理框架  |  代码  |  SWE-bench  |  奖励篡改  |
|  AI Scientist v2  |  研究论文  |  文本+代码+图表  |  同行评审（弱）  |  实验失败、误标、修饰掩盖缺陷  |

v2 在三个系统中拥有最弱的自动评估器、最宽的输出面以及最短的通往公开物件的路径。操作控制（沙箱、审查、披露）承担了大部分安全工作。

## 使用它

`code/main.py` 将 v2 循环模拟为状态机：想法 → 新颖性检查 → 实验 → 图表 → 撰写 → 评审 → 接受或迭代。每个状态都有可配置的失败概率，源自 Beel 等人的发现。运行模拟器 N 个循环并计数：

- 有多少想法到达提交阶段。
- 有多少提交存在经过修饰的论文所隐藏的关键实验缺陷。
- 重试预算如何在质量与产量之间权衡。

## 发布

`outputs/skill-ai-scientist-sandbox-review.md` 是一个双门审查清单，适用于研究循环代理生成的、在离开沙箱之前的任何内容。

## 练习

1. 使用默认参数运行 `code/main.py`。有多少比例的循环运行产生了“干净”的论文？有多少比例产生了带有实验失败缺陷（被图表批评修饰过）的论文？

2. 默认值已经使用 Beel 等人的 42% / 25%。分别用 `--experiment-failure 0.20 --novelty-mislabel 0.10` 和 `--experiment-failure 0.60 --novelty-mislabel 0.40` 重新运行。两次运行之间，修饰但有缺陷的论文比例如何变化？

3. 阅读Sakana的AI Scientist v2仓库README关于沙盒要求的部分。列出除了Docker之外，你会为多天自主运行施加的另外两个限制。

4. 阅读Beel等人第4节关于展示质量差距的内容。设计一个额外的评估器，能够捕捉看起来精美但实验上有缺陷的论文。

5. 提出一个针对研究代理输出的人工评审协议，其扩展性优于“让博士生阅读每篇论文”。识别瓶颈并围绕它设计。

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|---|---|---|
|  AI Scientist v1  |  "Sakana的模板化研究代理"  |  将实验填充到固定框架中  |
|  AI Scientist v2  |  "无模板研究代理"  |  结合VLM图形批评的代理式树搜索  |
|  代理式树搜索  |  "分支研究代理"  |  并行扩展多个实验计划；通过内部批评进行剪枝  |
|  视觉语言批评  |  "VLM图形润色"  |  多模态模型读取图形并重写以使其清晰  |
|  文献检索  |  "新颖性检查"  |  搜索先前工作以确认想法的新颖性——文献记载可能错误标记  |
|  润色掩盖  |  "漂亮的论文，破碎的研究"  |  展示质量超过实验质量；隐藏弱点  |
|  沙盒逃逸  |  "LLM代码逃逸"  |  代理执行的代码做出了循环设计者未预期的事情  |
