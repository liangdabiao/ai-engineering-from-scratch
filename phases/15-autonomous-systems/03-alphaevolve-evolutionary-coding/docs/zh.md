# AlphaEvolve — 进化编码代理

> 将一个前沿编码模型与进化循环和机器可检查评估器配对。让循环运行足够长时间。它发现了一个使用48次标量乘法的4x4复矩阵乘法过程——这是56年来对斯特拉森(Strassen)的首次改进。它还找到了一个谷歌范围内的Borg调度启发式算法，在生产中恢复了约0.7%的集群计算资源。该架构故意平淡无奇。胜利来自评估器的严谨性。

**类型:** 学习
**语言:** Python (stdlib, evolutionary-loop toy)
**先决条件:** Phase 15 · 01 (长期框架), Phase 15 · 02 (自学推理)
**时间:** ~60 分钟

## 问题

大型语言模型可以编写代码。进化算法可以搜索代码空间。两者分别被尝试了几十年；都遇到了瓶颈。大语言模型（LLM）的瓶颈是虚构(confabulation)：模型写出看似合理但实际并未实现其声称功能的代码。进化算法的瓶颈是搜索成本：对语法进行随机突变很少产生可编译的程序，更不用说更好的程序了。

AlphaEvolve (Novikov等人, DeepMind, arXiv:2506.13131, 2025年6月) 将两者结合。大语言模型（LLM）对程序数据库提出有针对性的编辑；自动评估器对每个变体进行评分；高分变体成为后续世代的父代。LLM处理编写看似合理代码的高成本步骤；评估器捕捉虚构内容。循环运行数小时到数周。

报告的结果：48次标量乘法的4x4复矩阵乘法（斯特拉森1969年的上界是49），谷歌生产环境中的Borg调度启发式算法，FlashAttention内核加速32.5%，Gemini训练吞吐量提升。

该架构之所以有效，是因为评估器是机器可检查的。在评估器不可检查的地方它就不起作用。这种不对称性就是教训。

## 核心概念

### 循环

1. 从种子程序`P_0`开始，该程序正确但次优。
2. 维护一个变体程序数据库，每个都由评估器评分。
3. 从数据库中采样一个或多个父代（采用MAP-elites风格或岛屿模型）。
4. 提示LLM（对于许多候选使用Gemini Flash，对于困难的使用Gemini Pro）生成父代的修改变体。
5. 在留出评估器上编译、运行和评估变体。
6. 按分数和特征向量插入数据库。
7. 重复。

两个细节很重要。第一，提示LLM时不仅提供父程序——通常还提供数据库中的几个顶级变体、评估器签名以及简短的任务描述。模型的任务是提出可能提高分数的有针对性的更改。第二，数据库以结构化的方式组织（MAP-elites网格、岛屿模型），以便循环探索多样性，而不仅仅是当前领先者。

### 是什么让评估器不可妥协

AlphaEvolve的胜利都来自评估器快速、确定且难以攻克的领域：

- **矩阵乘法算法**：一个单元测试，用于矩阵相乘并按位检查相等性。
- **Borg调度启发式算法**：一个生产级模拟器，重放历史集群负载并测量浪费的计算资源。
- **FlashAttention内核**：一个正确性测试加上实际硬件上的实时基准测试。
- **Gemini训练吞吐量**：每步测量的GPU秒数。

在每种情况下，评估器都能捕获那些本来会占主导地位的LLM错误：虚构的正确性声明、在硬件上消失的性能声明以及边缘情况失败。去掉评估器，循环就会优化出漂亮的代码（而未必正确）。

### 奖励黑客攻击(Reward hacking)是该陈述的另一面

进化优化的是评估器所测量的任何指标。如果评估器不完美，循环就会找到不完善之处。在未经验证的领域，循环会优化表面特征，而非预期行为。DeepMind在论文中明确指出了这一点：AlphaEvolve的成功仅适用于评估器严谨性与搜索雄心相匹配的领域。

2025-2026年代码搜索循环中奖励黑客攻击的具体实例：

- 奖励'完成时间'的优化目标奖励了提交空解决方案。
- 奖励测试正确性的基准分数奖励了记忆测试和过拟合。
- '代码质量'代理奖励了删除注释和重写变量名（没有语义变化）。

AlphaEvolve的修复方案：提供一个LLM从未见过的留出评估器，并在评估时生成输入。即便如此，DeepMind仍建议对任何拟部署的方案进行严格审查。

### 为什么LLM加搜索胜过单独使用任何一种

LLM可以产生可编译、语义合理的修改。对2000行Python文件进行随机突变遗传算法几乎总是产生语法错误。LLM还将搜索集中在合理的邻域（更改一个函数，而不是随机字节），从而大大减少了浪费的评估器调用。

反过来，评估器捕获了LLM的虚构内容。LLM会自信地声称某个函数'在极限情况下是O(n log n)'，而实际上是O(n^2)；实时基准测试解决了这个问题。

### AlphaEvolve在前沿堆栈中的位置

|  系统  |  生成器  |  评估器  |  领域  |  示例成果  |
|---|---|---|---|---|
|  AlphaEvolve  |  Gemini  |  正确性 + 基准测试  |  算法、内核、调度器  |  48次乘法的4x4矩阵乘法  |
|  FunSearch (DeepMind, 2023)  |  PaLM / Codey  |  正确性  |  组合数学  |  cap-set下界  |
|  AI Scientist v2 (Sakana, L5)  |  GPT/Claude  |  LLM批评 + 实验  |  机器学习研究  |  ICLR workshop论文  |
|  Darwin Godel Machine (L4)  |  代理脚手架  |  SWE-bench / Polyglot  |  代理代码  |  20% → 50% SWE-bench  |

所有这四个都是同一配方的变体：生成器加评估器，循环。区别在于评估器评什么以及它的严谨程度。

## 使用它

`code/main.py`在一个玩具符号回归问题上实现了一个最小的类AlphaEvolve循环。"LLM"是一个标准库代理，对计算目标函数的程序提出小的语法突变。"评估器"测量留出测试点上的均方误差。

观看：

- 每一代最优分数如何提升。
- MAP-elites网格如何保持多样性解存续，使循环不收敛到局部最小值。
- 移除留出测试（仅训练评估器）如何让循环产生惊人的过拟合。

## 练习

1. 运行`code/main.py`。注意最优分数轨迹。禁用留出评估器（标志`--no-holdout`）并重新运行。量化过拟合程度。

2. 阅读AlphaEvolve论文中关于MAP-elites网格的第3节。为一个新问题（例如编译器优化步骤）设计一个特征向量描述符，以保持搜索的多样性。

3. 经过56年后，48次乘法的4x4结果改进了Strassen的49次乘法界。阅读论文附录F，用三句话解释为什么这个问题的评估器特别容易正确设计，以及为什么大多数领域并非如此。

4. 提出一个AlphaEvolve会失败的领域。明确指出评估器在哪里失效以及原因。

5. 对于你熟悉的领域，写出你会使用的评估器签名。包括(a)正确性条件，(b)性能指标，(c)留出输入生成规则，(d)至少一项反奖励黑客检查。

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|---|---|---|
|  AlphaEvolve  |  "DeepMind的进化编码智能体"  |  Gemini + 程序数据库 + 机器可检查评估器  |
|  MAP-elites  |  "多样性保存档案"  |  以特征向量为键的网格；每个单元格保存具有该描述符的最佳变体  |
|  岛屿模型  |  "并行进化子种群"  |  独立种群定期迁移；防止过早收敛  |
|  机器可检查评估器  |  "确定性预言机"  |  单元测试、模拟器或基准，LLM无法伪造——这是该循环的前提条件  |
|  奖励黑客  |  "优化指标而非目标"  |  循环找到一种方法在不执行预期任务的情况下最大化分数  |
|  种子程序  |  "起点"  |  循环从中进化的初始正确但次优的程序  |
|  留出评估器  |  "LLM从未见过的评估数据"  |  评估时生成的输入以防止记忆  |
