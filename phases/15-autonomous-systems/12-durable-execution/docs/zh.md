# 长时间运行的后台代理：持久执行

> 生产环境中的长时间跨度代理并不在`while True`中运行。每次LLM调用成为一个具有检查点、重试和回放的活动。Temporal的OpenAI Agents SDK集成于2026年3月正式发布。Claude Code Routines（Anthropic）在没有持久本地进程的情况下调度Claude Code调用。会话在人工输入时暂停，部署后存活，并根据`thread_id`键从最新检查点恢复。新的易用性设计背后是一个旧模式——工作流编排——加上一个新输入：LLM调用作为非确定性活动，在恢复时必须确定性地回放。

**类型:** 学习
**语言:** Python (标准库，最小持久执行状态机)
**先修条件:** 阶段15 · 10 (权限模式)，阶段15 · 01 (长时间跨度代理)
**时间:** 约60分钟

## 问题

考虑一个运行四个小时的代理。它调用了三个工具，提示了用户两次，并进行了四十次LLM调用。运行到一半时，它所运行的主机重启了。会发生什么？

- 在一个简单的`while True`循环中：一切都会丢失。运行从头开始。三个工具调用（具有实际副作用）再次执行。用户再次被提示他们已经批准的事情。四十次LLM调用被重复计费。
- 使用持久执行：运行从最近的检查点恢复。已经完成的活动不会重新执行；其结果从持久日志中回放。用户不会重新批准他们已经批准的事情。已经进行的LLM调用不会重复计费。

这与工作流引擎已经推出十年的模式相同（Temporal、Cadence、Uber的Cherami）。新的是LLM调用现在成为一种活动——非确定性、昂贵、有副作用——并且它们完美地适配这种模式。

本课的贯穿主题：长时间跨度的可靠性会衰减（METR观察到一种“35分钟退化”——成功率大致随跨度二次下降）。持久执行使得运行可以超过可靠性曲线所支持的时间，这是一种新的方式：如果设计正确，可以安全地失败；如果设计错误，则不安全地失败。

## 核心概念

### 活动、工作流与回放

- **工作流**：确定性编排代码。定义活动的顺序、分支和等待。必须是确定性的，以便能够从事件日志中回放而不会出现意外的差异。
- **活动**：一个非确定性、可能失败的工作单元。LLM调用、工具调用、文件写入、HTTP请求。每个活动都会记录其输入和（完成后）输出。
- **事件日志**：持久化后端存储。记录每个活动的开始、完成、失败、重试以及每个工作流决策。
- **回放**：恢复时，工作流代码从头重新运行；每个已经完成的活动返回其记录的结果而不重新执行。只有尚未完成的活动才会实际运行。

这与React针对虚拟DOM重新渲染，或Git从提交重建工作树是同样的形态。编排器中的确定性使得持久性变得廉价。

### 为什么LLM调用适配这种模式

LLM调用是：
- 非确定性的（温度>0；即使温度0也会因模型版本不同而偏移）。
- 昂贵的（金钱和延迟）。
- 可能失败的（速率限制、超时）。
- 有副作用的（如果它们调用工具）。

这正是活动的特征。将每次LLM调用包装为一个活动，可以带指数退避重试、跨重启检查点以及用于调试的可回放跟踪。

### 由`thread_id`键标识的检查点

LangGraph、Microsoft Agent Framework、Cloudflare Durable Objects和Claude Code Routines都收敛于相同的API形态：一个`thread_id`（或等效项）标识会话；每个状态转换持久化到后端（默认PostgreSQL，开发用SQLite，缓存用Redis）；恢复读取最新检查点。

后端选择很重要：

- **PostgreSQL**: 持久、可查询、部署后存活。LangGraph的默认选择。
- **SQLite**: 仅本地开发；跨主机时丢失数据。
- **Redis**: 快速但临时，除非配置了AOF/快照。
- **Cloudflare Durable Objects**: 透明分布式；由唯一键限定范围；存活数小时到数周。

### 人工输入作为一等状态

先提议后提交（第15课）需要一个持久的“等待人工”状态。工作流暂停，外部队列持有待处理请求，批准从该点精确恢复。没有持久性，这是尽力而为；有了它，隔夜的批准到达后，工作流在早上恢复。

### 35分钟退化

METR观察到，每类被测量的代理在连续运行约35分钟后都会出现可靠性衰减。任务时长加倍大致使失败率翻两番。持久执行并不能解决这个问题；它允许你运行得比可靠性曲线支持的时间更长。安全的模式是将持久性与需要在重新进入时进行新的HITL的检查点结合，以及与限制总计算量（无论挂钟时间如何）的预算终止开关（第13课）结合。

### 何时持久执行是错误的答案

- 运行时间短于几分钟且无人工输入。开销 > 收益。
- 严格只读的信息检索。
- 正确性要求在一个上下文窗口内端到端完成的任务（一些推理任务；一些一次性生成）。

## 使用它

`code/main.py`在标准库Python中实现了一个最小的持久执行引擎。它支持：

- `@activity`装饰器，将输入和输出记录到JSON事件日志。
- 一个对活动进行排序的工作流函数。
- 一个`@activity`函数，回放已完成的活动而不重新执行它们。

驱动程序模拟一个三活动工作流，中途崩溃，并显示(a)一个简单的重试重新执行所有内容，与(b)一个回放仅运行缺失的活动。

## 练习

1. 运行`code/main.py`。观察简单重试和回放之间活动执行次数的差异。改变崩溃点并显示回放次数相应地改变。

2. 将玩具引擎转换为显式使用`thread_id`。模拟两个并发会话共享引擎，并确认它们的事件日志不会冲突。

3. 以玩具引擎中的一个活动为例。引入非确定性（工作流决策中的挂钟时间戳）。展示重放时的分歧。解释真实引擎如何处理此问题（副作用注册、`Workflow.now()` API）。

4. 阅读LangChain的“深度代理的运行时”文章。列出运行时持久化的每个状态，并指出每个状态覆盖了哪种故障模式。

5. 为6小时自主编码任务设计检查点策略。你在哪里设置检查点？崩溃后恢复是什么样子？什么情况下需要全新的人工参与？

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|---|---|---|
| 工作流  |  “代理脚本”  |  确定性编排代码；可从事件日志重放 |
| 活动  |  “一个步骤”  |  非确定性单元（LLM调用、工具调用）；在前后记录日志 |
| 事件日志  |  “后备存储”  |  每个状态转换的持久记录 |
| 重放  |  “恢复”  |  重新运行工作流；已完成的活动返回记录的结果而不重新执行 |
| 检查点  |  “保存点”  |  持久化状态，以thread_id为键；恢复时取最新 |
| thread_id  |  “会话键”  |  限定持久状态的标识符 |
| 35分钟退化  |  “可靠性衰减”  |  METR：成功率随时长近似二次下降 |
| 非确定性  |  “重放漂移”  |  挂钟时间、随机数、LLM输出；必须注册为副作用 |
