# 行动预算、迭代上限与成本治理器

> 一个中型电商智能体的月度LLM成本从其团队启用"订单跟踪"技能后的$1,200 to $4800美元。这不是定价错误，而是智能体发现了一个新循环并持续在其中支出。微软的Agent Governance Toolkit（2026年4月2日）将针对此类问题的防御措施系统化：每次请求的`max_tokens`，每次任务的令牌与美元预算，每日/每月上限，迭代上限，分层模型路由，提示缓存，上下文窗口化，高成本操作的人工介入检查点，预算超支的终止开关。Anthropic的Claude Code Agent SDK以不同名称提供了相同的原语。财务速度限制——例如，10分钟内超过50美元则切断访问——比月度上限能更快地捕获循环。

**类型：** 学习
**语言：** Python（标准库，分层成本治理器模拟器）
**前置条件：** 第15阶段·10（权限模式），第15阶段·12（持久化执行）
**时间：** 约60分钟

## 问题

自主智能体每一步都要花真金白银。聊天机器人的糟糕输出是糟糕的回复；智能体的糟糕循环就是一张账单。业界记录的此类故障模式术语是"拒绝钱包"——智能体持续推理、持续调用工具、持续计费，却没有任何机制阻止它，因为根本就没设计来阻止。

解决方案不是一个数字，而是一组不同时间尺度和粒度的限制：每次请求、每次任务、每小时、每天、每月。设计良好的堆栈能在几分钟内捕获失控循环，几小时内捕获缓慢泄漏，一天内捕获糟糕发布。对于长期自主的智能体，同样的堆栈还能维持预算。

这是一个工程课：数学很简单，但纪律才是团队失败的地方。以下限制列表均来自微软Agent Governance Toolkit或Anthropic Claude Code Agent SDK文档。

## 核心概念

### 成本治理器堆栈

1. **`max_tokens` 每次请求。** 简单。防止任何一次调用产生无界的结果。
2. **每次任务令牌预算。** 整个运行中不超过N个令牌。达到上限时硬停止。
3. **每次任务美元预算。** 与令牌相同但以货币计。Claude Code中的`max_tokens`。
4. **每次工具调用上限。** 不超过N次`max_tokens`调用，N次`max_budget_usd`调用等。
5. **迭代上限（`max_tokens`）。** 总智能体循环迭代次数；防止无限推理循环。
6. **每分钟/每小时/每天/每月上限。** 滚动窗口。在不同时间尺度捕获泄漏。
7. **财务速度限制。** 例如，“如果10分钟内花费超过50美元，则切断访问。”在月度上限触发前捕获基于循环的消耗。
8. **分层模型路由。** 默认使用较小模型；仅当分类器判断任务需要时才升级到较大模型。
9. **提示缓存。** 系统提示和稳定上下文存储在提供商的缓存中；重新发送的令牌成本接近零。
10. **上下文窗口化。** 压缩/总结以保持活动上下文低于阈值；直接降低令牌成本。
11. **高成本操作的人工介入检查点。** 在已知高成本的操作（长时间工具调用、大文件下载、昂贵的模型升级）之前，要求人工确认。
12. **预算超支的终止开关。** 任何上限触发时会话中止。上限被记录；需要单独的重新启用路径。

### 为什么用堆栈而不是单一上限

单一的月度上限只有在钱包耗尽后才能捕获失控智能体。单一的每次请求上限在会话级别上无用。不同的故障模式需要不同的时间尺度：

- **失控循环**（智能体陷入5秒重试循环）：由速度限制捕获。
- **缓慢泄漏**（智能体每个任务执行约2倍预期工作）：由每日上限捕获。
- **糟糕发布**（新版本使用5倍令牌）：由每周/每月上限捕获。
- **合法激增**（真实需求，非故障）：由每小时/每日上限捕获，并记录清晰日志。

### Claude Code的预算表面

Claude Code Agent SDK公开了（官方文档）：

- `max_turns` —— 迭代上限。
- `max_turns` —— 美元上限；超预算时会话中止。
- `max_turns` / `max_budget_usd` —— 工具允许列表和拒绝列表。
- 工具使用前的钩子点，用于自定义成本核算。

结合权限模式阶梯（第10课）。没有`autoMode`的`max_budget_usd`会话是无治理的自主。Anthropic明确表示自动模式需要预算控制；分类器与成本正交。

### 欧盟AI法案、OWASP Agentic Top 10

微软的Agent Governance Toolkit涵盖了OWASP Agentic Top 10和欧盟AI法案第14条（人类监督）的要求。在欧盟的生产环境中，日志记录和上限强制执行不是可选项。

### 观察到的$1,200 → $4800美元案例

微软文档中的真实案例：一个电商智能体在添加新工具后月度成本翻了三倍。该工具允许智能体在每次会话中轮询订单状态。没有循环检测，没有每次工具上限，没有周环比增长的警报。解决方案是每次工具上限加上每日增长警报。这是一个模板：每个新工具表面都是一个潜在的新循环；每个新工具都需要自己的上限和自己的警报。

## 使用它

`code/main.py`模拟了有无分层成本治理器堆栈的智能体运行。模拟智能体在若干轮后漂移到轮询循环；分层堆栈在速度窗口内捕获了它，而单一的月度上限要几天后才会触发。

## 发布

`outputs/skill-agent-budget-audit.md`审计了一个拟部署的智能体应用的成本治理器堆栈，并标记了缺失的层级。

## 练习

1. 运行`code/main.py`。确认在轮询循环轨迹中速度限制先于迭代上限触发。现在禁用速度限制，并测量智能体在迭代上限捕获它之前“花费”了多少。

2. 为浏览器智能体（第11课）设计一套每次工具上限。哪个工具需要最严格的上限？哪个工具可以无限制运行而没有风险？

3. 阅读微软Agent Governance Toolkit文档。列出文档中提到的所有上限类型。将每种类型映射到一种故障模式（失控循环、缓慢泄漏、糟糕发布、激增）。

4. 为一个实际任务（例如，“对仓库中的50个问题进行分类”）的夜间无人运行定价。将`max_budget_usd`设为您点估计值的2倍。证明2倍的合理性。

5. Claude Code的`max_budget_usd`基于会话总成本触发。设计一个您会在外部强制执行的互补速度限制。触发切断的条件是什么？重新启用是什么样的？

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|---|---|---|
|  Denial of Wallet  |  "失控账单"  |  智能体循环产生支出且无上限阻止  |
|  max_tokens  |  "每次请求上限"  |  单次完成大小的上限  |
|  max_turns  |  "迭代上限"  |  会话中智能体循环迭代次数的上限  |
|  max_budget_usd  |  "美元终止开关"  |  会话成本上限；超预算则中止  |
|  速度限制 | "速率上限" | 每个短时间窗口内的支出限制（例如，$50 / 10分钟）  |
|  分层路由 | "小模型优先" | 默认使用廉价模型；仅在分类器判定必要时升级  |
|  提示缓存 | "缓存系统提示" | 提供方缓存将重新发送的代币成本降至接近零  |
|  人在回路中的检查点 | "人工审批门" | 执行昂贵操作前需要人工确认  |
