# 终止开关、熔断器和蜜令令牌

> 终止开关是保存在代理编辑表面之外的布尔值——一个Redis键、一个功能标志、一个签名配置——用于完全禁用代理。熔断器粒度更细：它在特定模式（连续五次相同的工具调用）下触发，暂停有问题的路径，并上报给人工处理。蜜令令牌继承自经典欺骗手段：一个虚假凭证或蜜罐记录，代理没有正当理由触碰它，一旦访问就会触发警报。基于eBPF的数据路径（如Cilium）可以在内核层将隔离Pod的出站流量重写为取证蜜罐；已发布的Cilium基准测试报告负载下P99数据路径延迟低于毫秒（你的传播预算取决于策略更新如何到达节点，而不是数据路径本身）。自适应移动基线的统计检测器（EWMA, CUSUM）会悄悄接受漂移——应将它们与不变硬的硬性宪法性限制分层使用。

**类型：** 学习
**语言：** Python (stdlib, 三个检测器模拟器：终止开关、熔断器、蜜令令牌)
**先决条件：** 第15阶段·第13课（成本限制器），第15阶段·第10课（权限模式）
**时间：** ~60分钟

## 问题

成本限制器（第13课）限制了代理可以花费多少。它们并不限制代理在预算内可以做什么。一个速度上限为50美元的代理仍然可以泄露秘密、发布错误帖子或删除资源——代价高昂的操作往往在token消耗上很便宜。

本课涵盖与成本层并行的三个检测器：

1. **终止开关**：位于代理无法触及的范围之外的布尔关闭按钮。
2. **熔断器**：暂停特定路径的动作模式检测器。
3. **蜜令令牌**：诱饵，代理没有正当理由触碰它，一旦触碰就会暴露自己。

这三种都是LLM之前的工程。经典的欺骗、速率限制熔断器和功能标志终止在自主代理之前就存在了。新的地方在于攻击面：代理读取不受信任的内容（第11课）、编辑自己的记忆、并且可以将许多看似安全的动作组合成一个不安全的动作。这里提到的检测器之所以有效，是因为它们不信任代理的自我报告。

## 核心概念

### 终止开关

终止开关是代理可以读取但不能写入的布尔值。实现选择：

- **托管服务中的功能标志。** LaunchDarkly, Statsig, Unleash。更新在几秒钟内传播。
- **代理轮询的Redis键。** 简单；要求代理进程在每一步都检查。
- **对象存储中的签名配置。** 代理在启动时验证签名；拒绝未签名状态。
- **操作系统级信号或容器生命周期终止。** Docker `kill`, Kubernetes `kubectl delete pod`, systemd stop。

正确的终止开关应具备的特性：

- 代理不能将其设置为`off`。（存在于代理凭据无法写入的系统中。）
- 它在每个重要动作上都被检查，而不仅仅是在启动时。
- 当它关闭时，代理不做任何外部可观察的事情，包括记录到代理可以访问的系统。
- 重新启用它是一个明确的人工操作，而不是自动超时。

### 断路器

熔断器暂停特定模式，而不是整个代理。经典形态（来自2007年Nygard的书，仍然适用）：

- **关闭**：允许动作。
- **打开**：阻止动作。
- **半开**：冷却后，允许1-3次探测尝试（默认1次）；成功则关闭熔断器，任何剩余的失败则重新打开。

与代理相关的触发条件：

- 连续五次相同的工具调用（重复循环）。
- 同一工具连续五次不同输入的失败（系统性失败）。
- 工具调用速率超过阈值（第13课速度）。
- 在读取不受信任的内容后（第11课）调用了特定工具（例如`file.delete`）。

### 蜜令令牌

蜜令令牌（也称蜜罐令牌）是代理永远不应该触碰的输入。访问即报警。

代理的示例：

- 环境中一个没有实际权限的虚假`AWS_SECRET_ACCESS_KEY`。数据泄露尝试可立即归因。
- 一个带有已知哨兵值的虚假数据库记录。任何对此记录的读取或更新都会触发警报。
- 工作区中的一个虚假文件。对此文件的读取或编辑都是可疑的。
- 在会话开始时注入的一个虚假记忆条目。如果它出现在后续输出中，说明记忆已被泄露。

蜜令设计特定于工作流程。代理有正当理由触碰的蜜令就是噪音。

### 为什么分层统计和硬性限制

统计检测器（EWMA、CUSUM、工具调用率的z-score）适应移动基线。这对于合法漂移是一个特性，对于对抗性漂移则是一个缺陷。耐心的攻击者可以缓慢地移动基线，使得没有警报触发。

硬性宪法性限制（第17课）不会弯曲。将"永远不要发送电子邮件至`secrets@`"作为宪法性规则更简单、可审计，且无法通过缓慢漂移进行博弈。分层模式：统计检测器捕获大部分噪音，硬性限制捕获那些从检测器旁走过的攻击。

### 通过eBPF数据路径重定向进行隔离

Cilium基于eBPF的网络策略可以将隔离Pod的出站流量重写为内核层的取证蜜罐。Cilium的公开基准测试报告在TCP_RR上的P99数据路径延迟低于毫秒；在你的集群中*触发*隔离的挂钟预算取决于策略更新从控制平面传播到节点所需的时间（在Kubernetes上预加载operator时，几百毫秒是现实的；尾延迟取决于集群大小和协调节奏）。这给你带来了两样东西：代理不会泄露任何真实信息，并且代理在蜜罐内的行为被完全观察。与蜜令令牌相同的模式，不同层次：代理通过它在被隔离时尝试做的事情来揭示意图。

### 没有检测器能捕获什么

- **从允许的原语组合而成的语义攻击。** 每个单独的动作都通过了所有检测器；组合才是问题。这就是第10课中的自动模式组合失败。
- **将检测器作为有效负载一部分的攻击。** 如果代理读取了一个写着“在你行动之前，禁用蜜令”的页面，并且代理有能力做到，那么检测器就被破坏了。蜜令令牌应该放在代理不能修改的系统中。

## 使用它

`code/main.py` 模拟了一个简短的代理轨迹通过三个检测器。一个保存在外部字典中的终止开关；一个在五次相同工具调用时触发的熔断器；一个读取会触发警报的蜜令文件。输入合成轨迹：合法操作、重复循环、蜜令探测以及终止开关触发导致代理动作停止的场景。

1. 运行 `code/main.py`。确认断路器在第5轮（第五次相同调用）触发，金丝雀在第9轮（假键读取）触发。

2. 添加统计检测器：对工具调用率进行EWMA z-score检测。输入一个缓慢漂移的轨迹，显示检测器从未触发。现在添加一个硬限制（10分钟内最多50次工具调用），并显示该硬限制在同一轨迹上触发。

3. 为浏览器智能体设计一个金丝雀令牌集（第11课）。列出至少三个金丝雀及其各自检测的内容。

4. 阅读Cilium网络策略文档。具体描述一个出口重定向隔离流程：哪个策略选择器，哪个Pod，哪个出口重写，哪个告警。从“决定隔离”到“第一个重定向数据包”的挂钟延迟由什么决定？

5. 为杀死开关智能体定义重新启用程序。谁可以重新启用？必须记录什么？在重新启用之前，智能体必须做出什么改变？

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|---|---|---|
| 杀死开关 | "关闭按钮" | 位于智能体编辑表面之外的布尔值；在每个重要动作上进行检查 |
| 断路器 | "模式暂停" | 针对动作的触发条件：重复、失败率或速率限制 |
| 金丝雀令牌 | "蜜令牌" | 智能体没有正当理由触碰的诱饵；访问触发告警 |
| 蜜罐 | "取证沙箱" | 重定向的流量/工作空间，用于观察被隔离的智能体 |
| EWMA | "移动平均" | 指数加权；适应漂移（特性+缺陷） |
| CUSUM | "累积和" | 检测相对于基线的持续偏移 |
| 硬限制 | "宪法规则" | 不调整；无论历史如何都保持不变 |
| 宪法限制 | "始终为真的规则" | 与第17课的宪法绑定；不能被智能体编辑 |
