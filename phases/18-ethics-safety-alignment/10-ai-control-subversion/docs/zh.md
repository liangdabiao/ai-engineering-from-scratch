# AI Control — 即使存在颠覆也能确保安全

> Greenblatt, Shlegeris, Sachan, Roger (Redwood Research, arXiv:2312.06942, ICML 2024)。控制(Control)重新定义了安全问题：给定一个不受信任的强模型U，它可能以对抗方式针对你进行优化，我们应使用何种协议，在限制不可接受结果概率的同时，提取有用工作？资源包括一个受信任的弱模型T、一个受信任的人力预算H，以及编辑U输出的能力。在此框架内，对于代码后门任务，受信任编辑(Trusted Editing)帕累托优于不受信任监控(Untrusted Monitoring)。该框架继承了安全工程中的最坏情况假设：安全是指在红队压力(作为U的替代)下，对抗性选择的U无法造成具体不可接受结果的概率。

**类型：** 学习
**语言：** Python (stdlib, 代码后门任务的安全-效用前沿)
**先决条件：** 阶段18 · 07 (休眠智能体)，阶段18 · 08 (诡计)，阶段18 · 09 (对齐伪装)
**时间：** 约75分钟

## 学习目标

- 定义AI控制威胁模型：不受信任的强模型U，受信任的弱模型T，受信任的人力预算H。
- 描述受信任编辑协议，并解释为何它在ICML 2024代码后门任务上帕累托优于不受信任监控。
- 解释对齐(alignment)（U可信）与控制(control)（U不可信但我们仍然安全）之间的区别。
- 说明控制的三个限制：人力无法扩展，受信任弱模型最终落后，控制不会产生可信的强模型。

## 问题

课程7-9确立了前沿模型可能具有欺骗性对齐、施行诡计或伪装对齐。安全问题被重新定义：我们能否从一个不信任的模型中提取有用工作，而不让它有机会导致不可接受的结果？这就是控制问题，其结构类似于经典安全：假设最坏情况的对手，设计协议，在红队压力下衡量对手的成功率。

## 核心概念

### 威胁模型

控制假设：
- U — 不受信任的强模型。能力强。可能具有对抗性（评估将其视为对抗性）。
- T — 受信任的弱模型。能力较弱，但已知非对抗性。
- H — 受信任的人力。预算小，无法扩展。

系统必须使用U产生有用输出。对手试图在保持一定比例合法有用工作以达到效用阈值的同时，造成具体不可接受的结果（例如，在发布代码中植入后门）。

控制放弃对U的信任。它询问何种协议能在缺乏信任的情况下维持安全。

### ICML 2024后门任务

提示是竞赛级别的编程问题。U产生候选解决方案。隐藏的oracle标记解决方案是否包含后门（使代码产生错误输出的特定输入）。红队控制U，并允许查看所有协议内部。协议必须在保持后门发布率低于阈值的同时，最大化解决问题率。

### 四种协议比较

- **基线：** 直接输出U的结果。效用最大化，安全性最小化。
- **提前审计：** 对所有输出使用H。安全性最大化，预算迅速耗尽。
- **不受信任监控：** 第二个U实例对第一个实例的输出评分；H审计被标记的输出。风险：两个U实例可能串通。
- **受信任编辑：** T编辑每个查询的U输出。如果T不确定，转给H。

Greenblatt等人发现受信任编辑帕累托优于不受信任监控：在每个安全水平上都有更高的效用。

### 为什么受信任编辑胜出

不受信任监控要求第二个U实例不能与第一个协调。串通可能通过隐写线索或共享训练先验实现。协议必须投入预算用于防串通（改写、混淆），这会降低监控器的信号。受信任编辑完全避免了串通，因为T是可信的——任何T提出的编辑，假设都不具对抗性。代价是T能力较弱，因此可能过度拒绝或遗漏微妙的后门。

### 四个协议轴

1. U扮演的角色：仅生产者、监控者或两者兼具。
2. T扮演的角色：编辑者、监控者或重采样者。
3. H的分配方式：统一审计、随机审计或标记导向审计。
4. 协议花费多少计算资源用于防串通。

### 对齐vs控制

- 对齐：“U是否安全可信？”课程2、4、5、6、7、8、9均与此相关。
- 控制：“能否在不信任U的情况下安全使用U？”课程10。

对齐是模型的性质。控制是部署协议的性质。控制不会产生对齐的模型；它产生对可能未对齐模型的安全使用。两个议程都是必需的。

### 三个基本限制

- 人力无法扩展。H预算受限于愿意审计困难AI输出的人数。
- 受信任弱模型最终落后。当T远弱于U时，T的编辑信号变成噪声。
- 控制不是对齐。在ASL-3（课程18）起作用的协议可能无法在ASL-4起作用——如果U能操纵T或推断H的审计分布，控制就会失败。

### 2025年发展

UK AISI和Redwood合作开展控制安全案例（metr.org / redwoodresearch.org 2025）：控制安全案例论证，在关于U的最坏情况假设下，协议限制了不可接受结果的概率。安全案例（课程18）继承了控制框架。

### 这在阶段18中的位置

课程6-9描述了威胁（欺骗是可能的）。课程10描述了假设威胁存在时的防御范式。课程11描述了互补的进攻范式（通过可扩展监督使U可信）。

## 使用它

`code/main.py` 在玩具后门任务上模拟四种协议。U产生输出；一部分被植入后门（红队策略）。T批准或编辑。在预算内调用H。你可以改变预算和串通参数，以复现Greenblatt等人2024年图2中的帕累托前沿（形状，而非数值）。

## 发布

本课程产生`outputs/skill-control-protocol-audit.md`。给定部署协议描述，它能识别U/T/H角色，命名串通向量，并估计该协议在红队压力下在效用-安全前沿上的位置。

## 练习

1. 运行`code/main.py`。复现受信任编辑与不受信任监控的效用-安全前沿。确定不受信任监控的串通假设成为关键因素的预算区间。

2. 新增一个协议："T 以低温度重新采样，然后 U 编辑 T 的输出。" 衡量它在前沿上的位置。为什么这比可信编辑失败得更灾难性？

3. 2024年的论文假设U可以看到协议。设计一个协议，隐藏被审计的输出，不让U知道。这个协议对部署基础设施有什么要求？

4. 阅读 Greenblatt 等人 2024 年第 7 节（合谋）。提出一种防合谋措施，并解释其预期的效用成本。

5. 为一个使用 U = GPT-4类、T = GPT-3.5类、H = 审计1%输出的部署阐明控制安全案例。指出该案例所依赖的唯一最强假设。

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|-----------------|------------------------|
|  AI控制  |  "颠覆下的安全"  |  在最坏情况U下限制不可接受结果的协议设计 |
|  不可信模型(U)  |  "强模型"  |  我们想从中获取工作但不可信的有能力模型 |
|  可信模型(T)  |  "弱模型"  |  一个较弱的、已知不具有对抗性的模型，用作编辑器/监控器 |
|  可信人类(H)  |  "审计预算"  |  有限的专家劳动；昂贵；真实信号 |
|  可信编辑  |  "T编辑U"  |  T作为U输出门的协议；在模糊输出时调用H |
|  不可信监控  |  "U监控U"  |  第二个U实例对输出打分的协议；只有额外投入才能抵抗合谋 |
|  控制安全案例  |  "正式论证"  |  论证协议在最坏情况U下限制攻击者成功率的书面论证 |
