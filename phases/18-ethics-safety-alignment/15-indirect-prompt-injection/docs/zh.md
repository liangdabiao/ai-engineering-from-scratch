# 间接提示注入(Indirect Prompt Injection, IPI) — 生产环境攻击面

> 间接提示注入(IPI)将指令嵌入外部内容中——网页、电子邮件、共享文档、支持工单——由智能体系统在没有用户显式操作的情况下消费。IPI是2026年生产环境的主要威胁：它绕过了用户输入过滤器，因为攻击者从未接触用户；它随着智能体处理更多外部内容而悄无声息地扩展；它针对的是自动化工作流，其中没有人阅读提示词。MDPI Information 17(1):54 (2026年1月) 总结了2023-2025年的研究。NDSS 2026的IPI防御论文提出了核心挑战：注入的指令在语义上可能是良性的（如"请打印Yes"），因此检测需要超出关键词过滤。"攻击者后发制人"（Nasr等，OpenAI/Anthropic/DeepMind联合，2025年10月）：自适应攻击（梯度、强化学习、随机搜索、人工红队）攻破了12种已发表的防御中的>90%，而这些防御最初报告的攻破率接近零。

## 学习目标

- 定义间接提示注入并描述三种常见的传递向量。
- 解释为什么用户输入过滤器完全无法检测IPI。
- 描述"信息流控制"框架作为2026年防御范式。
- 陈述Nasr等人（2025年10月）关于自适应攻击对已发表的IPI防御成功率的发现。

## 问题

直接提示注入需要攻击者接触用户或其提示词。IPI则不需要：攻击者将载荷放置在任何智能体可能读取的内容中——网页、收件箱中的电子邮件、GitHub议题、产品评论。智能体在正常操作过程中拾取并执行指令。用户是信使，而非意图的发出者。

## 核心概念

### 三种传递向量

- **检索增强生成(RAG)。** 攻击者发布文档；检索步骤获取该文档；提示词将其拼接在用户问题之前；模型执行攻击者的指令。
- **收件箱/文档工作流。** 攻击者向用户发送电子邮件；智能体读取邮件；提示词包含邮件正文；模型遵循邮件的指令。
- **工具输出。** 攻击者控制智能体使用的工具（例如，返回攻击者控制结果的网络搜索）；工具输出包含指令；智能体的控制流遵循这些指令。

这三种向量共享一个结构特性：攻击者控制提示词的一个片段，而无需接触面向用户的输入。

### 为什么用户输入过滤器无法检测

IPI载荷不会出现在用户的输入中。它出现在检索到的内容中。如果过滤器仅针对用户输入，则载荷会绕过它。如果过滤器针对所有到达模型的內容，则必须适用于任意检索到的文本——这代价高昂，并且会对恰好包含祈使语气语言的合法内容产生误报。

### 人工智能的信息流控制(IFC)

2026年的防御范式借鉴了经典操作系统安全。将每个内容来源视为一个安全标签。将用户查询标记为"可信"。将检索到的内容标记为"不可信"。将模型的控制流视为信息流：由不可信内容触发的操作必须在执行前得到可信输入的批准。

CaMeL（微软2025）、ConfAIde（斯坦福2024）和NDSS 2026的IPI防御论文以不同方式实现了IFC。共同原则是：只要代码和数据共享相同的上下文窗口，遏制就是目标，而非预防。

### "攻击者后发制人"

Nasr等人（2025年10月）使用自适应攻击（梯度搜索、强化学习策略、随机搜索、72小时人工红队）测试了12种已发表的IPI防御。每种最初报告接近零攻破率的防御都被攻破至>90%的攻破率。

方法论上的教训：只有经过自适应攻击评估的防御才能发表。静态攻击基准并非鲁棒性的证据；攻击者会知道防御手段。

### 真实事件

第25课涵盖EchoLeak（CVE-2025-32711，CVSS 9.3）——微软365 Copilot中首个公开记录的零点击IPI。GitHub Copilot Chat中的CamoLeak（CVSS 9.6）。GitHub Copilot中的CVE-2025-53773。生产部署正在被实际环境中的IPI攻陷，而不仅仅是基准测试。

### OWASP和NIST框架

OWASP LLM Top 10（2025年）将提示注入（直接+间接）列为LLM01，即排名第一的应用层威胁。NIST AI SPD 2024年将间接提示注入称为"生成式AI最大的安全缺陷"。

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|-----------------|------------------------|
|  IPI  |  "间接提示注入"  |  通过用户未编写的内容注入，智能体在正常操作期间消费 |
|  RAG注入  |  "中毒检索"  |  攻击者发布检索步骤获取的内容；提示词包含载荷 |
| 零点击（Zero-click）  |  "无用户操作（no user action）"  |  攻击在代理操作期间自动触发；用户无需任何操作 |
| IFC  |  "信息流控制（information flow control）"  |  基于标签的方法：来自不可信内容的操作需要可信的批准 |
| 自适应攻击（Adaptive attack）  |  "梯度/强化学习红队（gradient / RL red-team）"  |  了解防御并针对其优化的攻击；诚实评估所必需 |
| 良性指令（Benign instruction）  |  "请打印Yes（please print Yes）"  |  语义上良性的IPI载荷；无关键字过滤器能捕获它 |
| 范围违规（Scope violation）  |  "跨信任泄露（cross-trust exfiltration）"  |  代理从一个信任上下文访问数据并输出到另一个信任上下文 |
