# 文档与图表理解

> 文档不是照片。PDF、科学论文、发票或手写表单具有布局、表格、图表、脚注、页眉和语义结构，这些是纯图像理解无法捕捉的。前VLM时代的技术栈是一个流水线：Tesseract OCR + LayoutLMv3 + 表格提取启发式方法。VLM浪潮用无OCR模型取代了它——Donut（2022）、Nougat（2023）、DocLLM（2023）——这些模型直接输出结构化标记。到2026年，前沿技术就是“将页面图像以2576像素原生分辨率输入Claude Opus 4.7”，然后结构化标记输出就自然得到了。本课解读文档AI的三个时代演进。

**类型：** 构建
**语言：** Python（标准库，布局感知文档解析器骨架）
**前置条件：** 阶段12·05（LLaVA），阶段5（NLP）
**时长：** 约180分钟

## 学习目标

- 解释文档AI的三个时代：OCR流水线、无OCR、VLM原生。
- 描述LayoutLMv3的三个输入流：文本、布局（边界框）、图像块，以及统一掩码。
- 比较Donut（无OCR，图像→标记）、Nougat（科学论文→LaTeX）、DocLLM（布局感知生成）、PaliGemma 2（VLM原生）。
- 为新任务（发票、科学论文、手写表单、中文收据）选择一个文档模型。

## 问题

“理解这个PDF”出奇地困难。信息分布在：

- 文本内容（90%的信号）。
- 布局（页眉、脚注、侧边栏、双栏格式）。
- 表格（行、列、合并单元格）。
- 图形和图表。
- 手写批注。
- 字体和排版（标题与正文）。

原始OCR提取文本但丢失了其他信息。一个关心发票的系统需要知道“总计：1,245美元”来自右下角，而不是脚注。

## 核心概念

### 时代1——OCR流水线（2021年前）

经典技术栈：

1. PDF → 每页图像。
2. Tesseract（或商用OCR）提取文本及每个词的边界框。
3. 布局分析器识别区块（页眉、表格、段落）。
4. 表格结构识别器解析表格。
5. 领域规则+正则表达式提取字段。

适用于清晰的印刷文本。在手写体、歪斜扫描件、复杂表格、非英语文本上会失效。每种失败模式都需要自定义异常路径。

### TrOCR（2021）

TrOCR（Li等人，arXiv:2109.10282）用基于合成+真实文本图像训练的Transformer编码器-解码器取代了Tesseract经典的CNN-CTC。在手写和多语言文本上表现优异。但仍是一个流水线（检测器→TrOCR→布局），不过OCR步骤得到了显著改进。

### 时代2——无OCR（2022-2023）

第一批无OCR模型说：完全跳过检测，直接将图像像素映射到结构化输出。

Donut（Kim等人，arXiv:2111.15664）：
- 编码器-解码器Transformer，编码器为Swin-B。
- 输出为表单理解的JSON、摘要的markdown或任何任务特定模式。
- 无OCR，无布局，无检测。

Nougat（Blecher等人，arXiv:2308.13418）：
- 专门在科学论文上训练。
- 输出为LaTeX/markdown。
- 处理公式、多栏布局、图表。
- 每个arXiv解析器都调用的模型。

这些是专才，不是通才。Donut处理科学论文失败；Nougat处理发票失败。

### LayoutLMv3（2022）

另一条路线。LayoutLMv3（Huang等人，arXiv:2204.08387）保留OCR但增加了布局理解：

- 三个输入流：OCR文本令牌、每个令牌的2D边界框、图像块。
- 跨三种模态的掩码训练目标（掩码文本、掩码块、掩码布局）。
- 下游任务：分类、实体提取、表格问答。

LayoutLMv3是基于OCR的文档理解的巅峰。在表单和发票上表现强劲。需要上游OCR。在标准化文档基准测试中达到前VLM时代的最佳准确率。

### DocLLM（2023）

DocLLM（Wang等人，arXiv:2401.00908）是LayoutLM的生成式兄弟。基于布局令牌条件生成自由形式的回答。更适合文档问答；仍依赖OCR输入。

### 时代3——VLM原生（2024+）

2024年的VLM已经足够好，可以完全取代整个流水线。将全页图像以高分辨率输入VLM，提问，得到答案。

- LLaVA-NeXT 336分块AnyRes适用于小文档。
- Qwen2.5-VL动态分辨率原生支持2048+像素。
- Claude Opus 4.7支持2576像素文档。
- PaliGemma 2（2025年4月）专门针对文档+手写体训练。

VLM原生与OCR流水线之间的差距迅速缩小。到2026年，VLM原生在以下方面胜出：

- 场景文本（手写+印刷，混合文字）。
- 带有合并单元格的复杂表格。
- 嵌入文本中的数学公式。
- 带有文本注释的图表。

OCR 流程在以下方面仍然获胜：

- 大规模纯扫描工作负载，其中每页延迟至关重要。
- 流程可靠性（确定性故障与 VLM 幻觉）。
- 需要可审计 OCR 输出的受监管环境。

### Claude 4.7 / GPT-5 前沿

在 2576 像素原生输入下，前沿视觉语言模型以接近人类的准确度进行文档理解。以下为 2026 年初的基准数据：

- DocVQA：Claude 4.7 ~95.1，PaliGemma 2 ~88.4，Nougat ~77.3，流水线 LayoutLMv3 ~83。
- ChartQA：Claude 4.7 ~92.2，GPT-4V ~78。
- VisualMRC：Claude 4.7 ~94。

闭源模型的差距主要在于分辨率和基础大语言模型规模。7B 参数的开源模型落后几个百分点，但正在追赶。

### 数学公式与 LaTeX 输出

科学论文需要公式的精确 LaTeX 输出。Nougat 正是在此训练而成。使用 LaTeX 目标训练的视觉语言模型（如 Qwen2.5-VL-Math、Nougat 衍生模型）能生成可用的 LaTeX。没有经过显式 LaTeX 训练的模型会产生可读但不精确的转录。

对于 2026 年的科学论文流程：在 PDF 上链式使用 Nougat，然后对棘手页面使用 VLM。

### 手写文字

仍然是最难的子任务。混合印刷+手写（医生笔记、填写表格）的场景中，OCR 流程在成本上仍优于 VLM。纯手写 VLM 正在改进（Claude 4.7、PaliGemma 2）。

### 2026 年方案

对于新的文档 AI 项目：

- 大规模纯印刷发票：LayoutLMv3 + 规则，成本高效。
- 混合文档（科学+手写+表格）：VLM 原生方法（PaliGemma 2 或 Qwen2.5-VL）。
- 完整 arXiv 摄入：Nougat 处理数学公式，VLM 处理图表。
- 监管场景：OCR 流程 + VLM 验证器进行交叉检查。

## 使用它

`code/main.py`：

- 一个简易的布局感知分词器：给定（文本，边界框）对，生成 LayoutLMv3 风格的输入。
- 一个 Donut 风格的任务模式生成器：用于表格的 JSON 模板。
- 跨 OCR 流程、Donut、Nougat 和 VLM 原生方法的每页 token 预算比较。

## 发布

本课产生 `outputs/skill-document-ai-stack-picker.md`。给定一个文档 AI 项目（领域、规模、质量、监管），在 OCR 流程、免 OCR 专用模型和 VLM 原生方法之间进行选择。

## 练习

1. 你的项目每天处理 1000 万张发票。哪种技术栈能在不损失准确率的情况下最小化每页成本？

2. 为什么 LayoutLMv3 在表单问答上优于纯 CLIP 视觉语言模型，但在场景文本上表现不佳？边界框流（bbox stream）牺牲了什么？

3. Nougat 生成 LaTeX。提出一个测试用例，其中 VLM 原生输出在 LaTeX 保真度上击败 Nougat，以及一个 Nougat 获胜的用例。

4. 阅读 PaliGemma 2 论文（Google，2024）。与 PaliGemma 1 相比，关键的训练数据添加是什么提升了文档准确率？

5. 设计一个安全的监管混合方案：OCR 流程为主，VLM 为辅助交叉检查。如何解决不一致？

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|-----------------|------------------------|
|  OCR 流程  |  "Tesseract 风格"  |  分阶段堆栈：检测 -> OCR -> 布局 -> 规则；确定性，脆弱  |
|  免 OCR  |  "Donut 风格"  |  跳过显式 OCR 的图像到输出变换器；单一模型  |
|  布局感知  |  "LayoutLM"  |  输入包含每个 token 的边界框坐标；跨模态统一掩码  |
|  VLM 原生  |  "前沿 VLM"  |  将页面图像直接以高分辨率输入 Claude/GPT/Qwen 视觉语言模型；无流程  |
|  DocVQA  |  "文档基准"  |  文档视觉问答标准；被引用最多的分数  |
|  标记输出  |  "LaTeX / Markdown"  |  结构化输出格式而非自由文本；支持下游自动化  |
