# 任意分辨率视觉：Patch-n'-Pack 与 NaFlex

> 真实图像并非224×224的正方形。收据是9:16，图表是16:9，医学扫描可能是4096×4096，手机截图是9:19.5。2024年之前的VLM答案——将所有图像缩放到固定正方形——抛弃了使OCR、文档理解和高分辨率场景解析得以工作的信号。NaViT（Google，2023）表明，你可以将可变分辨率的面片(patch)打包到单个Transformer批次中，并使用块对角掩码(block-diagonal masking)。Qwen2-VL的M-RoPE（2024）完全丢弃了绝对位置表。LLaVA-NeXT的AnyRes将高分辨率图像平铺为基图+子图像。SigLIP 2的NaFlex变体（2025）现在是开放VLM的默认编码器，这些VLM希望单个检查点服务于所有宽高比。本课将端到端实现patch-n'-pack。

**类型：** 构建
**语言：** Python (stdlib, patch packer + block-diagonal mask)
**前置条件：** 阶段12 · 01 (ViT patches), 阶段12 · 05 (LLaVA)
**时间：** ~120分钟

## 学习目标

- 将一批可变分辨率图像的面片打包成一个序列，并构建块对角注意力掩码(block-diagonal attention mask)。
- 根据给定任务在AnyRes平铺（LLaVA-NeXT）、NaFlex（SigLIP 2）和M-RoPE（Qwen2-VL）之间进行选择。
- 计算OCR、图表和摄影的令牌预算(token budget)而不调整图像大小。
- 命名方形缩放的三种失败模式：文本挤压、内容裁剪、填充浪费令牌。

## 问题

Transformer期望一个序列。一个批次是相同长度序列的堆叠。如果你的图像是224×224，每次得到196个面片令牌，无需填充，任务完成。在224上训练，在224上推理，再也不需要考虑分辨率。

世界并不配合。文档是纵向的（8.5×11英寸，约2:3）。图表截图是横向的（16:9）。收据又高又窄（1:3）。医学成像为2048×2048或更大。移动设备截图是1170×2532（0.46:1）。

三个2024年前的选项及其失败原因：

1. 缩放到固定正方形（224×224或336×336）。挤压扭曲了文本和面部。缩小破坏了图表标签和OCR内容。这是LLaVA-1.5之前的标准做法。
2. 裁剪到固定宽高比。你丢弃了大部分图像，而选择裁剪位置本身就是一个视觉问题。
3. 填充到最长边。解决了变形问题，但对于纵向图像，浪费了50%以上的令牌用于填充。所有这些填充令牌导致二次注意力成本。

2024-2025年的答案：让Transformer在图像原始分辨率下处理面片，并弄清楚如何将异质批次打包成一个序列而不浪费计算。

## 核心概念

### NaViT与patch-n'-pack

NaViT（Dehghani等人，2023）是证明这在规模上有效的论文。其思路是机械性的：

1. 对于批次中的每张图像，以选定的面片大小（例如14）计算其原始面片网格。
2. 将每张图像的面片展平成自己的变长序列。
3. 将所有图像的面片拼接成一个批次的长序列。
4. 构建块对角注意力掩码，使得图像A的面片只关注图像A内部。
5. 携带每个面片的位置信息（2D RoPE或分数位置嵌入）。

一个包含三张图像（336×336（576个令牌），224×224（256个令牌），448×336（768个令牌））的批次变成单个1600令牌的序列，带有1600×1600的块对角掩码。没有填充。没有浪费计算。Transformer处理任意宽高比。

NaViT还引入了训练期间的分数面片丢弃(fractional patch dropping)——在批次中随机丢弃50%的面片——这既正则化又加速训练。SigLIP 2继承了这一点。

### AnyRes（LLaVA-NeXT）

LLaVA-NeXT的AnyRes是实用的替代方案。给定高分辨率图像和固定编码器（336的CLIP或SigLIP），平铺图像：

1. 从预定义集合中选择一个网格布局——(1x1)、(1x2)、(2x1)、(1x3)、(3x1)、(2x2)等——以最佳匹配图像的宽高比。
2. 将完整图像平铺到网格中；每个瓦片变为336×336的裁剪。
3. 同时生成缩略图：将整个图像缩放到336×336作为全局上下文令牌。
4. 通过冻结的336编码器编码每个瓦片。拼接瓦片令牌和缩略图令牌。

对于2×2网格加上缩略图的672×672图像：4 * 576 + 576 = 2880个视觉令牌。昂贵但有效——LLM同时看到局部细节和全局上下文。

当你的编码器被冻结且只支持一种分辨率时，AnyRes是首选路线。它会大幅增加大图像的令牌数量（4×4网格的1344×1344图像为9216 + 576 ≈ 9800令牌，几乎填满8k LLM上下文）。

### M-RoPE（Qwen2-VL）

Qwen2-VL引入了多模态旋转位置嵌入(Multimodal Rotary Position Embedding)。与NaViT的分数位置或AnyRes的平铺加缩略图不同，每个面片携带3D位置（时间、高度、宽度）。查询/键旋转处理任意的H、W和时间长度。

M-RoPE无需重新训练即可提供原生动态分辨率。在推理时，你输入任意H×W图像，面片嵌入器生成H/14 × W/14个令牌，每个令牌获取其（t=0，r=行，c=列）位置，RoPE以正确的频率旋转注意力，完成。Qwen2.5-VL和Qwen3-VL继续采用此方法。InternVL3的V2PE是相同思路，每个模态具有可变编码。

与AnyRes不同，M-RoPE在原始分辨率下是O(H×W/P²)个令牌——没有乘法平铺开销。与NaViT不同，它仍然期望每次前向传播一张图像。跨分辨率批处理仍然需要patch-n'-pack。

### NaFlex（SigLIP 2）

NaFlex是SigLIP 2检查点的原生灵活模式。单个模型在推理时提供多种序列长度（256、729、1024个令牌）。内部使用NaViT风格的patch-n'-pack进行训练，以及每个面片的绝对分数位置。卖点：一个检查点，在推理时根据任务选择你的令牌预算。

对于语义任务（分类、检索），256个令牌。对于OCR或图表理解，1024个令牌。无需重新训练。

### 打包掩码

块对角掩码是大多数实现容易出错的地方。对于长度为`N_total`的打包序列，涵盖长度为`i=0..B-1`的图像`n_i`，形状为`M`的掩码`(N_total, N_total)`中，如果两个索引落在同一图像块中则为1，否则为0。你可以从累积长度列表构建它：

```
offsets = [0, n_0, n_0+n_1, ..., N_total]
M[i, j] = 1 iff there exists b where offsets[b] <= i < offsets[b+1] and offsets[b] <= j < offsets[b+1]
```

这可以通过PyTorch中的`torch.block_diag`或显式gather一行完成。FlashAttention的可变长度路径（`cu_seqlens`）完全跳过掩码，直接使用累积长度张量在序列内部进行注意力——对于典型批次，比密集掩码快约10倍。

### 令牌预算

根据任务选择策略：

- OCR / 文档：1024-4096个令牌。使用SigLIP 2 NaFlex在1024处，或AnyRes 3x3 + 缩略图。
- 图表和UI：在384-448原生分辨率下729-1024个令牌。Qwen2.5-VL动态分辨率，带有最大像素上限。
- 自然照片：256-576个令牌即可。下游LLM能看到足够信息。在内容密度高的地方为令牌付费。
- 视频：空间池化后每帧64-128个令牌，2-8 FPS。第12.17课涵盖此内容。

2026年的生产规则：为每个任务选择一个最大像素上限，以原生宽高比编码至该上限，打包批次，并跳过填充。Qwen2.5-VL暴露了`min_pixels`和`max_pixels`来控制这个旋钮。

## 使用它

`code/main.py`实现了对具有整数像素坐标的异构图像批次进行分块打包。它：

- 接收一个(H, W)图像尺寸列表。
- 计算每张图像在分块大小14下的分块序列长度。
- 将它们打包成一个总长度为`sum(n_i)`的序列。
- 构建块对角注意力掩码（为清晰起见，使用密集掩码）。
- 比较打包成本与方形缩放和AnyRes平铺。
- 打印混合批次（收据、图表、截图、照片）的令牌预算表。

运行它。得到的数字就是为什么每个2026年的开源视觉语言模型都使用分块打包的原因。

## 发布

本节产生`outputs/skill-resolution-budget-planner.md`。给定一个混合宽高比的工作负载（OCR、图表、照片、视频帧）和一个总令牌预算，它选择正确的策略（NaFlex、AnyRes、M-RoPE或固定方形），并输出每个请求的配置。当你为产品定型视觉语言模型时使用此技能——它能防止悄无声息的10倍令牌爆炸，从而破坏延迟预算。

## 练习

1. 一张收据是600x1500（1:2.5）。在分块大小14下，有多少原生分辨率令牌？在方形缩放到336后有多少？实践中哪种会损失更多的OCR准确性？

2. 为四个长度分别为256、576、729、1024的图像批次构建块对角掩码。验证注意力矩阵为2585×2585，并且恰好有`256^2 + 576^2 + 729^2 + 1024^2`个非零项。

3. 对于一张1792x896的图像，分块大小14，比较：(a) 方形缩放到336然后编码，(b) AnyRes 2x1 + 缩略图，(c) 原生M-RoPE。哪种使用的令牌最少？哪种保留的细节最多？

4. 实现部分分块丢弃：给定一个打包序列，均匀随机丢弃50%的令牌，并相应地更新块对角掩码。测量掩码的稀疏度变化。

5. 阅读Qwen2-VL论文（arXiv:2409.12191）第3.2节。用两句话描述`min_pixels`和`max_pixels`控制什么，以及为什么两个边界都很重要。

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|-----------------|------------------------|
|  分块打包  |  "NaViT式打包"  |  将来自不同图像的可变长度分块序列连接到一个批次维度  |
|  块对角掩码  |  "打包掩码"  |  限制每张图像的分块仅关注自身，而不是打包中相邻分块的注意力掩码  |
|  AnyRes  |  "LLaVA-NeXT平铺"  |  将高分辨率图像分割为固定大小图块的网格加上全局缩略图；使用固定编码器编码每个图块  |
|  NaFlex  |  "SigLIP 2原生灵活"  |  单个SigLIP 2检查点，在推理时无需重新训练即可支持256/729/1024令牌预算  |
|  M-RoPE  |  "多模态RoPE"  |  处理任意H、W、T而无需位置表的3D旋转位置编码（时间、行、列）  |
|  cu_seqlens  |  "FlashAttention打包"  |  FlashAttention变长路径使用的累积长度张量，替代密集块对角掩码  |
|  min_pixels / max_pixels  |  "分辨率边界"  |  Qwen2.5-VL每个请求的控制旋钮，限制非常小或非常大输入上的令牌数量  |
|  视觉令牌预算  |  "每张图像多少令牌"  |  每张图像发出的分块令牌的大致数量；设定了LLM的提示预算和注意力成本  |
