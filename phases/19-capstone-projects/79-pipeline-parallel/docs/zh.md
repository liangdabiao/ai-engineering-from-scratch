# 流水线并行(Pipeline Parallel)与空泡(Bubble)分析

> 张量并行(Tensor Parallelism)将矩阵乘法拆分到多个节点上。流水线并行(Pipeline Parallel)将模型拆分到多个节点上，每个节点一个阶段。微批次(Microbatch)在流水线中流动。开始和结束时的空闲时间就是空泡；最小化空泡是全部技巧所在。

**类型：** 构建
**语言：** Python
**先修知识：** 第19阶段C课程第42-49课
**时间：** ~90分钟

## 学习目标

- 将一个顺序模型拆分为N个阶段，并模拟在N个节点上的前向流水线。
- 使用GPipe调度（先只进行前向填充，再进行后向）调度M个微批次通过流水线，并计算空泡比例。
- 将空泡与Megatron-LM和PipeDream中使用的交错式1F1B调度进行比较。
- 论证阶段分配：每个阶段计算量相等比每个阶段参数量相等更重要。

## 问题

一个fp16的70B参数模型仅参数就需要140GB。没有消费级GPU能容纳它。ZeRO-3将参数分片到各个节点上，但仍需要每个节点在每个前向步骤中对整个层执行allgather操作，每层付出log(N)跳的代价。流水线并行采取了不同路径：将模型切成N个阶段，每个阶段放在一个节点上。第1层的前向在节点0完成后将激活张量传递给节点1；节点1运行第2层并传递给节点2；依此类推。后向按相反方向流动。内存线性下降，因为每个节点只保存一个阶段；计算是顺序的，这就是空泡问题。

空泡是流水线开始时的空闲时间（等待第一个微批次到达最后阶段）和结束时的空闲时间（等待最后一个微批次反向流回）。对于M个微批次和N个阶段，每个阶段的空泡比例为(N-1)/(M+N-1)。当M=8, N=4时为27%。当M=64, N=4时为4.5%。当每一步有更多微批次时，空泡缩小，这意味着每个微批次的批次大小较小，这是驱动微批次设计的约束。

## 核心概念

```mermaid

  R0[rank 0: stage 0 / layer 0] --> R1[rank 1: stage 1 / layer 1]
  R1 --> R2[rank 2: stage 2 / layer 2]
  R2 --> R3[rank 3: stage 3 / loss]
  R3 -.backward.-> R2
  R2 -.backward.-> R1
  R1 -.backward.-> R0
```

### GPipe调度

在开始任何后向之前，先将所有M个微批次前向填充流水线；然后按相反方向排空后向。每个微批次的激活必须保存到其后向完成，因此内存随M线性增长。前向需要M+N-1个周期，后向同样需要M+N-1个周期。每个阶段的有效工作量为2M个周期；每个阶段的空泡为2(N-1)个周期。当每个前向和后向各占一个时间单位时，空泡比例为(N-1)/(M+N-1)。选择M远大于N可以隐藏空泡。

### 1F1B调度

交错：一旦微批次的前向到达最后阶段，立即开始其后向并让其反向流回。调度在每个阶段交替进行一次前向和一次后向。空泡仍然是N-1，但激活内存受限于流水线深度，而不是微批次数量。生产环境中的流水线使用1F1B（Megatron, PipeDream）。本课先实现更简单的GPipe，1F1B作为练习。

### 为什么每个阶段的计算量要均衡

如果阶段0需要50ms，阶段1需要100ms，那么每个周期都受限于阶段1。其他阶段在每个周期中空闲50ms等待阶段1释放。参数量相等是错误的衡量维度：一个Transformer的计算量主要由每层的注意力(Attention)和MLP主导，而嵌入层参数多但计算量少。阶段分配应该均衡每个阶段的FLOPs，而不是每个阶段的权重数。

### 微批次与批次

一个流水线运行M个大小为B的微批次。有效批次大小为M*B。流水线步骤结束时的梯度是组合后的M*B个样本的梯度。空泡比例取决于M；优化器看到的是M*B。调优M意味着在空泡（高M低空泡）与每个微批次的内存（GPipe中高M导致更高激活内存）之间权衡。

## 实际中的生产模式

三种模式使流水线并行足够健壮以投入生产。

**激活检查点(Activation Checkpointing)与流水线配合使用。** 在GPipe中有M个微批次在飞，激活内存是一个微批次的M倍。激活检查点在后向时会重新计算前向，以计算换内存；这种组合使得流水线对于长序列变得可行。

**阶段均衡应基于测量，而非假设。** 生产团队运行一个性能分析过程，测量目标硬件上每层的实际计算量（FLOPs和墙上时间），然后根据测量结果进行划分。Megatron-LM的`--num-layers-per-stage`标志接受一个列表，允许在各阶段具有不同每层成本时分配不均匀的层数。

**发送-接收调度必须避免死锁。** 如果流水线中每个阶段先发送后接收，则会在线上死锁。标准修复是交错：偶数序号节点先发送后接收，奇数序号节点先接收后发送。本课显式调度节点，使模式可见。

## 使用它

生产模式：

- **Megatron-LM.** 大规模流水线并行的参考。使用1F1B，并支持张量+流水线+数据并行组合。
- **DeepSpeed Pipeline.** 与ZeRO集成；ZeRO-1+流水线是最大开源模型的常见组合。
- **PyTorch Pipe.** PyTorch原生的流水线封装器，构建在`torch.distributed.pipeline.sync.Pipe`之上。

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|----------------|------------------------|
|  Pipeline  |  "沿深度方向的模型并行"  |  每节点一个阶段，激活逐阶段流动  |
|  Bubble  |  "流水线空闲时间"  |  开始和结束时各(N-1)步，部分阶段无工作  |
|  Microbatch  |  "批次切片"  |  一个前向/后向单元；空泡随M增大而缩小  |
|  GPipe  |  "先填充后排空"  |  所有M个前向先执行，再执行任何后向；高激活内存  |
|  1F1B  |  "交错调度"  |  每阶段一次前向一次后向；有界激活内存  |
