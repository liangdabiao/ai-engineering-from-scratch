# 分块策略对比

> 分块决定了检索器能够触及的范围。分界选错了，任何嵌入模型、重排序器、大语言模型都无法修复下游的损害。

**类型：** 构建
**语言：** Python
**前置知识：** 阶段11课程04（嵌入）、06（RAG）、07（高级RAG）；阶段19跟踪B基础（课程20-29）
**时间：** 约90分钟

## 学习目标
- 从零实现五种分块策略：固定窗口、句子、递归切分、语义聚类和结构化Markdown标题。
- 在带有标签答案片段的固定语料库上测量召回率@k，并解释为什么一种策略在散文中胜出，而另一种策略在技术文档中胜出。
- 阅读块长度分布，并识别每种策略引入的失败模式：孤立句子、中间符号截断、仅标题块、语义漂移。
- 无需运行基准测试，通过检查三个属性为新的语料库选择默认策略：文档类型、平均段落长度以及格式是否带有显式结构。

## 问题

每个RAG管道都从将源文档切割成足够小的块开始，以便嵌入模型能够容纳，同时每个块又足够大，能包含一个完整的思想。切割位置的选择不是一个超参数，而是检索器能返回内容的上限。

一个查询问“预算中止阈值是什么样子的”，只有当包含中止阈值的块可被检索到时才能成功。如果固定窗口切分器将阈值值与上下文切分开，嵌入会移动到不同的聚类，BM25分数下降，重排序器看到噪声，LLM生成的答案就是错误的。2024年的论文“LongRAG: Enhancing Retrieval-Augmented Generation with Long-context LLMs”测量到，仅因分块选择，检索召回率就有35个百分点的绝对波动。2025年关于上下文块标题的后续工作缩小了差距，但并未完全消除。

本课程并行构建五种策略，在带有标签答案片段的固定语料库上运行它们，并让你亲自查看召回率数据。

### 固定窗口

最暴力的基准。每隔N个字符切分。可选择重叠，这样在位置N被切断的句子在从位置N - overlap开始的块中保持完整。快速、确定、边界效果差。用作对照，而非默认。

### 句子

使用正则表达式或简单状态机在句子边界处切分。将一句话或多句话打包成一个块，直到达到目标字符预算。避免在单词中间截断，但仍会在段落中间和章节中间截断。许多早期RAG管道的默认选项，对没有其他结构的散文是合理选择。

### 递归切分

由2023年时代的库推广的层次策略。首先尝试在最强分隔符（双换行、段落）处切分，然后退而求其次（单换行），然后是句子，最后是字符。当块符合预算时递归终止。对结构不一致的文档表现良好，因为它能按区域自适应。

### 语义聚类

对每个句子进行嵌入。将共享主题中心的连续句子聚在一起。当到中心的运行相似度低于阈值时切分。边界反映含义，而非字符。构建较慢且依赖嵌入模型，但对段落内切换主题的文档具有鲁棒性。

### 结构化Markdown标题

对于带有显式结构的文档（Markdown、reStructuredText、RFC风格编号章节），在标题边界处切分。每个块包含标题及其下方直到下一个同级或更高级别标题的所有内容。每个主题的块最小，但仅当语料库格式良好时可用。

### 召回率@k如何衡量边界选择

带有标签的查询携带源文档中答案片段的确切字符偏移量。分块后，你问：检索器返回的前k个块中是否有任何一个与标签片段重叠？如果是，则该查询的召回率@k为1；否则为0。对查询集取平均。对每种策略运行相同的评估，差异显示了哪种边界策略能在你拥有的语料库中存活。

## 动手构建

`code/main.py` 实现：

- `fixed_window(text, size, overlap)` - 基准。
- `fixed_window(text, size, overlap)` - 简单句子打包器。
- `fixed_window(text, size, overlap)` - 层次递归。
- `fixed_window(text, size, overlap)` - 基于确定性模拟嵌入的质心聚类。
- `fixed_window(text, size, overlap)` - 标题感知切分器。
- `fixed_window(text, size, overlap)` - 基于哈希的嵌入，以便循环离线运行。
- `fixed_window(text, size, overlap)` - 与阶段19跟踪B的混合检索课程相同的形状。
- `fixed_window(text, size, overlap)` - 比较循环。
- 一个`fixed_window(text, size, overlap)`在固定语料库上运行每种策略并打印召回率@k表。

运行它：

```bash
python3 code/main.py
```

输出是一个小表格，每行对应一种策略，每列对应一个k值。句子策略在结构化语料库上失败。结构化Markdown在Markdown语料库上获胜。递归在混合语料库上表现稳定，因为递归自适应。语义聚类在散文语料库上获胜，因为没有有用的结构线索。

## 表格不会隐藏的失败模式

**孤立句子。** 句子打包产生缺少主题句的块。嵌入因此指向错误的聚类。

**中间符号截断。** 在代码或YAML中使用固定窗口会将标识符一分为二。两半嵌入成噪声。

**仅标题块。** 结构化Markdown生成只包含`## Title`的块。过滤掉它们或附加下一个块的第一个段落。

**语义漂移。** 当语料库主题统一时，语义聚类会过度切分。一个5000字符的块将许多具体答案打包成一个分散的嵌入。将语义聚类与硬字符上限结合。

**过时嵌入。** 语义聚类使用嵌入模型。如果更改模型，块也会变化。将分块模型与检索模型分开固定，或一起重建索引。

## 无需运行基准测试选择默认策略

三个属性决定新语料库的默认分块器。

|  属性  |  值  |  默认  |
|----------|-------|---------|
|  文档类型 | 无结构散文 | 递归拆分，目标800  |
|  文档类型 | Markdown / RFC / API文档 | 结构化的Markdown  |
|  文档类型 | 代码 | 识别AST（超出范围；参见第19阶段第02课）  |
|  段落长度 | 长文本，单一主题 | 按句子拆分，目标500  |
|  段落长度 | 短文本，混合主题 | 语义拆分，阈值0.6  |

如有疑问，选择递归拆分。它是最强的单一策略基线。

## 使用它

生产模式：

- 在部署新管道之前运行评估；不要信任库默认的策略。
- 每当你更改嵌入模型或语料库组合时，重新运行评估；优胜者取决于语料库。
- 将策略名称持久化到每个分块的元数据中，以便后续归因回归。

## 练习

1. 添加第六种策略：使用`tiktoken`的token窗口代替字符计数。与同一夹具上的固定窗口进行比较。
2. 向散文夹具中注入30%的代码块。重新运行表格。解释为什么除了结构化的Markdown之外的所有策略都丢失了召回率。
3. 将确定性嵌入替换为项目实际提供商的嵌入。测量语义聚类召回率的变化。报告策略之间的差异是扩大还是缩小。
4. 为每个分块添加一个`tiktoken`字段：一句话的中心描述。将摘要附加到分块主体后重新运行评估。测量召回率的提升。

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|-----------------|------------------------|
|  召回率@k | "我们得到了正确的分块吗？" | 任何前k个分块与黄金答案片段重叠的查询比例  |
|  分块重叠 | "滑动窗口" | 将上一个分块的最后N个字符重新包含到下一个分块中  |
|  结构拆分器 | "标题感知的分块" | 在H1/H2/H3边界处切割；标题文本是分块的一部分  |
|  语义分块器 | "主题感知的分块" | 嵌入句子，按质心相似度聚类，在漂移处切割  |
|  质心漂移 | "主题转移" | 运行均值与下一个句子之间的余弦相似度下降到超过阈值  |
