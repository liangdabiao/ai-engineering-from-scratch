# 工具接口——为什么智能体需要结构化输入输出

> 语言模型生成标记(token)。程序执行动作。两者之间的鸿沟正是工具接口(Interface)：一个合约(Contract)，让模型请求动作而宿主执行它。每个2026年的技术栈——OpenAI、Anthropic和Gemini上的函数调用(function calling)；MCP的`tools/call`；A2A的任务部分(task parts)——都是同一四步循环的不同编码。本课命名该循环并展示运行它的最小机制。

**类型：** 学习
**语言：** Python (标准库，无LLM)
**前置要求：** 阶段11 (LLM补全API)
**时间：** 约45分钟

## 学习目标

- 解释为什么只能生成文本的LLM自身无法对现实世界采取行动。
- 画出四步工具调用循环（描述→决定→执行→观察）并指出每一步的拥有者。
- 将工具描述写为三部分：名称、JSON Schema输入和确定性执行函数。
- 区分纯工具(pure tools)和副作用工具(side-effecting tools)，说明为何这种区分对安全至关重要。

## 问题

LLM输出下一个标记的概率分布。这就是全部输出面。如果你问一个聊天模型“班加罗尔现在天气如何”，它可以写出一个合理的句子，但无法调用天气API。那个句子可能碰巧正确，也可能是三天前的。

弥合这一鸿沟正是工具接口的目的。宿主机程序——你的智能体运行时、Claude Desktop、ChatGPT、Cursor或自定义脚本——向模型通告一个可调用工具列表。当模型决定需要某个动作时，它发出一个结构化载荷(payload)，指明工具及其参数。宿主机解析该载荷，真正运行该工具，并将结果反馈回去。循环持续，直到模型认为不再需要调用。

这个合约的第一个版本于2023年6月作为OpenAI的"functions"参数发布。Anthropic随后在Claude 2.1中加入了`tool_use`块。Gemini几个月后添加了`functionDeclarations`。现在每个提供商都暴露了相同的形式：一个JSON Schema类型化的工具列表输入，一个JSON载荷的工具调用输出。模型上下文协议(Model Context Protocol, 2024年11月)将该合约泛化，使得一个工具注册表可以服务所有模型。A2A (2026年4月, v1.0)为智能体间的委托(delegation)添加了相同的原语。

四步循环是所有这一切之下的不变量。阶段13中的其他内容都是对其的阐述。

## 核心概念

### 第一步：描述(Describe)

宿主用三个字段声明每个工具。

- **名称。** 一个稳定的、机器可读的标识符。`get_weather`，而不是“天气的东西”。
- **描述。** 一段自然语言简述。“当用户询问某个具体城市的当前条件时使用。不要用于历史数据。”
- **输入模式。** 一个JSON Schema对象(draft 2020-12)，描述工具的参数。

模型接收列表。现代提供商会使用提供者特定的模板将这些声明序列化到系统提示(system prompt)中，因此作为调用者你只需处理结构化形式。

### 第二步：决定(Decide)

给定用户消息和可用工具，模型选择三种行为之一。

1. **直接回答**文本。不调用工具。
2. **调用一个或多个工具。** 发出结构化调用对象。在`parallel_tool_calls: true`下（OpenAI和Gemini默认，Anthropic选择加入），模型可以在一次轮次中发出多个调用。
3. **拒绝。** 严格模式的结构化输出可以产生一个类型化的`parallel_tool_calls: true`块而不是调用。

一个工具调用载荷有三个稳定字段：一个调用`id`，一个工具`name`和一个JSON `arguments`对象。id的存在使得宿主可以将后续结果与特定调用关联起来，这在并行调用乱序返回时很重要。

### 第三步：执行(Execute)

宿主接收调用，根据声明的模式验证参数，并运行执行器。无效参数意味着模型幻觉了一个字段或使用了错误的类型——这是弱模型上非常常见的失败模式。生产宿主对无效参数会做三件事之一：快速失败并将错误反馈给模型，使用约束解析器修复JSON，或者将验证错误包含在提示中让模型重试。

执行器本身是普通代码。Python、TypeScript、shell命令、数据库查询。它产生一个结果，通常是一个字符串，但可以是任何JSON值或结构化内容块（MCP中的文本、图像或资源引用）。结果必须是可序列化的。

### 第四步：观察(Observe)

宿主将工具结果附加到对话中（作为`tool`角色消息，带有匹配的`id`），并重新调用模型。模型现在上下文中有了工具输出，可以产生最终答案或请求更多调用。这一直持续到模型停止发出调用或宿主达到迭代次数的安全限制。

### 信任分割(Trust Split)

工具有两种类型，对安全至关重要。

- **纯(Pure)。** 只读、确定性、无副作用。`get_weather`、`search_docs`、`get_current_time`。可安全推测性调用。
- **后果性(Consequential)。** 改变状态、花费金钱、接触用户数据。`get_weather`、`search_docs`、`get_current_time`。必须设门控。

Meta 2026年关于智能体安全的“双重规则(Rule of Two)”指出，单次轮次最多可组合如下两项：不可信输入、敏感数据、后果性动作。工具接口正是你实施该规则的地方——通过拒绝调用、要求用户确认或升级作用域。详见阶段13·15安全章节和阶段14·09智能体级权限策略。

### 循环的所在

| 上下文 | 谁描述 | 谁决定 | 谁执行 |
|---------|---------------|-------------|--------------|
| 单轮函数调用 (OpenAI/Anthropic/Gemini) | 应用开发者 | LLM | 应用开发者 |
| MCP | MCP服务器 | 通过MCP客户端的LLM | MCP服务器 |
| A2A | 智能体卡片发布者 | 调用智能体 | 被调用智能体 |
|  网页浏览器（函数调用代理）  |  浏览器扩展 / WebMCP  |  LLM  |  浏览器运行时  |

到处都是一样的四个步骤。列名变了，结构没变。

### 为什么不直接提示模型输出JSON？

"让模型以JSON格式回复"是函数调用之前的模式。它在最先进的模型上约有5%到15%的失败率，在较小的模型上更高。失败模式包括缺少花括号、尾随逗号、幻觉字段和错误类型。这时你需要一个JSON修复步骤、重试或约束解码器。

原生函数调用之所以更好，有三个原因。第一，提供商在确切的调用形式上对模型进行端到端训练，因此严格模式下有效JSON率提升到98%到99%。第二，调用负载位于自己的协议槽中，而不是在自由文本内——因此工具调用永远不会泄露到用户可见的回复中。第三，提供商通过约束解码（OpenAI的严格模式、Anthropic的`tool_use`、Gemini的`responseSchema`）强制符合模式。输出保证有效。

### 断路器

当模型停止发出调用或主机达到最大轮次计数时，循环终止。生产环境的主机将此设置为5到20轮之间。超过这个数量，你几乎肯定陷入了模型无法退出的循环。Claude Code默认为20；OpenAI Assistants为10；Cursor的代理模式为25。

另一种选择——无界循环——每半年就会出现一次"代理一夜之间在API调用上花费了400美元"的事后分析。没有界限就不要发布。

## 使用它

`code/main.py`在没有LLM的情况下运行四步循环。一个假的"决策者"函数通过对用户消息进行模式匹配来模拟模型；执行器、模式验证器和观察步骤测试工具是真实的。运行它以查看完整的请求/响应编排及可打印的中间状态，然后在后面的课程中用任何真实的提供商替换这个假的决策者。

需要关注的内容：

- 工具注册表为每个工具包含四个字段：名称、描述、模式和执行器引用。
- 验证器是一个仅用标准库编写的最小JSON Schema子集（类型、必需、枚举、最小/最大）。第13阶段·04提供了一个更完整的版本。
- 循环将迭代次数限制为五次。生产代理正需要这种断路器。

## 练习

1. 向`code/main.py`添加第四个工具，名为`get_stock_price(ticker)`。将其描述写为"当用户通过股票代码询问当前股价时使用。不要用于历史价格或市场摘要。"运行测试工具，确认假决策者将提到股票代码的查询路由到新工具。

2. 破坏模式验证器。传入一个缺少必需字段的`arguments`对象的调用，确认主机在执行前拒绝它。然后传入一个带有额外未知字段的调用。决定：主机应该拒绝还是忽略？用安全性论证来证明你的选择。

3. 将测试工具中的每个工具分类为纯工具或效果性工具。向需要它的注册表条目添加`consequential: true`标志，并更改循环，使其在每次选择效果性工具时打印一行"将向用户确认"。这就是每个生产主机需要的确认门控的形状。

4. 在纸上画出四步循环，并用上面提供的提供商列表表格为你最喜欢的客户端（Claude Desktop、Cursor、ChatGPT或自定义栈）填写内容。与第13阶段·06中的MCP特定变体进行交叉参考。

5. 从头到尾阅读OpenAI的函数调用指南。找出一个在请求中但不在此处呈现的四步循环中的字段。解释它增加了什么，以及为什么它是方便的而不是必不可少的。

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|----------------|------------------------|
|  工具  |  "模型可以调用的东西"  |  名称 + JSON Schema 类型化输入 + 执行器函数的三元组  |
|  函数调用  |  "原生工具使用"  |  提供商级API支持，用于发出结构化工具调用而不是文本  |
|  工具调用  |  "模型的行动请求"  |  一个JSON负载，包含`id`、`name`、`arguments`，由模型发出  |
|  工具结果  |  "工具返回的内容"  |  执行器的输出，包装在具有匹配ID的`tool`角色消息中  |
|  并行工具调用  |  "一次多个调用"  |  在一个模型轮次中的多个调用对象，独立且可按ID排序  |
|  严格模式  |  "保证JSON"  |  约束解码，强制模型的输出对声明的模式进行验证  |
|  纯工具  |  "只读工具"  |  无副作用；可安全重新运行  |
|  效果性工具  |  "动作工具"  |  改变外部状态；需要门控、审计或用户确认  |
|  四步循环  |  "工具调用循环"  |  描述→决策→执行→观察  |
|  主机  |  "智能体运行时"  |  持有工具注册表、调用模型并运行执行器的程序  |
