# 并行/群组/网络化架构

> 与监督者模式对比：没有中央决策者。代理从共享的事件总线读取任务，异步获取工作，并将结果写回。LangGraph 明确支持"群组架构（Swarm Architecture）"用于去中心化、动态的环境。Matrix (arXiv:2511.21686) 将控制流和数据流表示为通过分布式队列传递的序列化消息，以消除编排者瓶颈。这种权衡是明确的：用确定性和可追溯性换取可扩展性。群组架构适用于包含许多独立子问题的任务；不适用于需要单一连贯计划的任务。

**类型:** 学习 + 构建
**语言:** Python (标准库, `threading`, `queue`)
**前置条件:** 阶段16 · 05 (监督者模式), 阶段16 · 04 (原始模型)
**时间:** 约75分钟

## 问题

监督者模式可扩展到几个工作节点。但如果是成百上千呢？监督者本身就成了瓶颈：每个关于谁做什么的决策都集中在一个代理上。一个慢的计划步骤就会拖慢整个系统。

群组架构翻转了设计。工作节点从共享队列中领取工作，而不是由中央规划器分配工作。"协调"被内置于事件总线语义中。没有编排者；系统会扩展直到队列成为瓶颈。

## 概念

### 形状

```
                ┌──── shared queue ────┐
                │                      │
       ┌────────┼────────┐  ◄──────┬───┘
       ▼        ▼        ▼         │
     Worker  Worker  Worker   Worker
      A       B       C        D
       │        │        │         │
       └────────┴────────┴─────────┘
                 │
                 ▼
            results pool
```

没有编排者。每个工作节点重复：拉取任务、处理、写入结果（并可选择将后续任务加入队列）。

### 何时使用群组架构

- **许多独立任务。** 爬取、转换、分类。任务之间互不依赖。
- **时长可变的工作。** 如果某些任务耗时100毫秒，而其他任务耗时10秒，群组架构会自动平衡负载——快速的工作节点会领取下一个任务。监督者必须预先估计时长。
- **吞吐量优先于确定性。** 你关心的是总完成时间，而不是严格的顺序。

### 何时不适合使用群组架构

- **有顺序的工作流。** 如果步骤3需要步骤2的输出，群组架构可能导致步骤3在步骤2完成之前触发。
- **全局计划任务。** 复杂的研究问题受益于规划器。一群研究者产生独立的事实，而不是连贯的报告。
- **调试。** 由于没有中央日志和异步工作，重现错误代价高昂。

### Matrix (arXiv:2511.21686)

Matrix是2025年的论文，它将群组架构推向其自然结论：控制流和数据流都是分布式队列上的序列化消息。没有中央协调器。容错性来自于消息持久化。可扩展性是消息代理的问题，而不是系统的问题。

贡献：一种编程模型，其中多代理协调变成了"这个代理订阅哪个消息主题？"而不是"监督者接下来选择哪个代理？"这使系统看起来像一个发布/订阅事件网格。

### LangGraph的群组架构

LangGraph 2025文档明确将"群组架构"描述为多代理模式之一：代理是节点，但边形成有环的有向图，任何节点都可以从池中被激活。工作节点根据条件从可用工作中选择，而不是由监督者分配。

### 失败模式：饥饿和热点

如果所有工作节点都拉取最快可用的任务，长时间运行的任务永远不会被选中，直到它们成为仅剩的任务。经典的队列饥饿。

缓解措施：
- 具有显式老化机制的优先级队列（等待时间越长优先级越高）。
- 工作节点专业化：某些工作节点只承担"长"任务。
- 背压：限制进入队列的快速任务数量。

### 基于内容的路由关联

群组架构与基于内容的路由（第22课）自然结合。不要使用通用队列，而是为每种消息类型设置一个队列。专业工作节点只订阅其所属类型。这是可扩展到数千个代理的消息总线架构的基础。

## 动手构建

`code/main.py`实现了一个由4个工作线程组成的群组，从共享的`queue.Queue`中拉取任务。任务具有可变的持续时间（有的快，有的慢）。该演示对比了：

- **顺序基线：** 一个工作节点串行处理所有任务。
- **固定分配：** 每个任务预先分配给特定工作节点（监督者风格）。
- **群组：** 工作节点从共享队列中拉取。

群组架构自动平衡负载；固定分配则会使快速工作节点在其分配任务较慢时空闲。

运行：

```
python3 code/main.py
```

输出显示每个工作节点的任务计数（群组分配不均衡但最优）和挂钟时间。

## 使用它

`outputs/skill-swarm-fit.md`评估任务应使用群组架构还是监督者模式。输入：任务独立性、持续时间方差、排序要求、调试需求。

## 发布

检查清单：

- **具有老化机制的优先级队列。** 防止长任务饥饿。
- **工作节点幂等性。** 如果工作节点在运行中崩溃，任务可能被多次拉取。工作节点必须是幂等的。
- **持久化队列。** 在生产环境中使用Kafka、Redis Streams或数据库支持的队列。`queue.Queue`仅存在于内存中。
- **每个任务的可观测性。** 每个任务都有一个跟踪ID；每个工作节点记录其开始/结束。
- **背压。** 如果队列增长速度超过工作节点消耗速度，则减慢生产者。

## 练习

1. 运行`code/main.py`。群组架构在可变时长工作负载上比顺序执行快多少？比固定分配快多少？
2. 添加优先级队列变体（使用`code/main.py`）。根据任务的"重要性"字段分配优先级。观察低优先级任务在持续负载下是否出现饥饿。
3. 实现热点检测器：当任何工作节点处理的任务数量是最慢工作节点的3倍以上时记录日志。这表明任务时长分布有何问题？
4. 阅读Matrix论文(arXiv:2511.21686)的摘要和第三节。找出Matrix接受的一个具体权衡（可扩展性增益）和放弃的一个权衡（可追溯性、确定性）。
5. 将群组演示转换为使用(task_type, payload)元组的`code/main.py`，工作节点只订阅特定类型。当任务异构时，什么路由规则有意义？

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|----------------|------------------------|
| 群组架构 | "去中心化代理" | 工作节点从共享队列中拉取；没有中央编排者。 |
| 事件总线 | "代理订阅主题" | 按类型或内容将任务路由到工作节点的消息代理。 |
| 饥饿  |  "任务永远不运行"  |  低优先级任务因持续有更高优先级的工作到达而永远不会被选中。 |
| 热点  |  "某一个工作器不堪重负"  |  负载不平衡，导致某一个工作器承担了大部分任务。 |
| 背压  |  "让生产者慢下来"  |  一种机制，当队列填满时向上游发出信号，使其停止生产。 |
| 幂等工作器  |  "可以安全地重新运行"  |  一个任务处理两次会产生相同的结果。这是必需的，因为工作器可能在运行中途崩溃。 |
| 持久队列  |  "崩溃后仍能存活"  |  由磁盘或复制存储支持的队列；工作器崩溃时任务不会丢失。 |
| 矩阵框架  |  "全消息传递的群体"  |  数据和控制流都以序列化消息的形式在分布式队列上传递。 |
