# 角色专业化——规划者、批评者、执行者、验证者

> 2026年最常见的多智能体分解：一个智能体规划，一个执行，一个批评或验证。MetaGPT（arXiv:2308.00352）将其形式化为编码到角色提示中的SOP——产品经理、架构师、项目经理、工程师、QA工程师——遵循`Code = SOP(Team)`。ChatDev（arXiv:2307.07924）通过带有“沟通去幻觉”（智能体明确请求缺失细节）的“聊天链”将设计师、程序员、评审员、测试员串联起来。验证者是承重角色：Cemri等人（MAST，arXiv:2503.13657）表明每个多智能体故障都可以追溯到缺失或失效的验证。普华永道报告在CrewAI中通过结构化验证循环获得了7倍的精度提升（10% → 70%）。

**类型：** 学习+构建
**语言：** Python (stdlib)
**先决条件：** 阶段16·04（原始模型），阶段16·05（监督器）
**时间：** 约60分钟

## 问题

通用的多智能体系统产生通用的输出。群聊中的三个程序员写出同一平庸代码的三种变体。你可以添加更多智能体，增加更多轮次，但仍然无法跨越质量门槛。

解决办法不是增加智能体——而是*不同*的智能体。分配不同的角色。给予批评者规划者没有的工具。给予验证者客观的测试套件。现在系统有了内部的分歧和基于实地的修正，而不仅仅是平行猜测。

## 概念

### 四个经典角色

**规划者。** 阅读目标，生成步骤列表或规范。工具：知识检索，文档。输出：结构化计划。

**执行者。** 每次读取一个计划步骤，生成工件。工具：实际工作工具（代码编译器、shell、API客户端）。输出：工件。

**批评者。** 根据规划者的意图阅读执行者的输出。工具：对工件的只读访问，静态分析。输出：接受/拒绝并附理由。

**验证者。** 阅读工件并运行确定性检查。工具：测试运行器、类型检查器、模式验证器。输出：通过/失败并附证据。

批评者是主观的、有主见的，通常基于LLM。验证者是客观的、确定性的，通常基于代码。它们不是同一个角色。

### MetaGPT的SOP模式

MetaGPT（arXiv:2308.00352）将软件工程SOP编码为角色提示：

- **产品经理**编写PRD。
- **架构师**生成系统设计。
- **项目经理**拆分任务。
- **工程师**实现。
- **QA工程师**运行测试。

每个角色都有严格的输入/输出模式。角色提示说明角色*是*什么以及*必须产生*什么。`Code = SOP(Team)`的表述——确定性SOP将LLM团队转变为可预测的流水线。

### ChatDev的沟通去幻觉

ChatDev增加了一个关键举措：当执行者需要计划中没有的特定细节时，它在继续之前明确询问设计师。这防止了LLM经典失败——看似合理地编造细节。

实现：角色提示包含“当你需要未提供给你的特定信息时，在生成输出前按名称询问相关角色。”

### 为什么验证者最重要

Cemri等人（MAST）追踪了1642个多智能体执行失败案例。21.3%是验证缺口——系统发布了未经任何人检查的答案。其余的79%通常可追溯到“存在一个检查，但静默失败或从未运行”。验证是承重角色。

普华永道报告（CrewAI部署，2025年）添加结构化验证循环将准确率从10%提升到70%。一个角色带来7倍增益。

### 批评者 vs 验证者

- 批评者是一个LLM，评审工件质量。主观的。可能被看似合理的文字所欺骗。
- 验证者是一个在工件上运行的确定性程序。客观的。给出通过/失败并附证据。

两者并用。批评者捕捉验证者无法表达的品味问题。验证者捕捉批评者看不到的bug，因为它们只在运行时出现。

### 反模式

你系统中的每个角色都是LLM，每个角色的输出都是“看起来不错”。经典MAST失败模式。至少添加一个验证者，其通过/失败由代码决定，而不是由LLM决定。

### 框架映射

- **CrewAI** — `Agent(role, goal, backstory)`是教科书式的专业化界面。
- **LangGraph** — 节点可以有专门的提示；边强制执行流水线。
- **AutoGen** — 在群聊中具有单字名称的角色特定ConversableAgent。
- **OpenAI Agents SDK** — 角色专属智能体之间的交接工具。

## 动手构建

`code/main.py`实现了一个构建简单Python函数的4角色流水线：

- **规划者**生成规范。
- **执行者**生成代码字符串。
- **批评者**（LLM模拟）标记明显问题。
- **验证者**在沙箱（`exec`）中针对测试用例运行生成的代码。

演示运行两次：一次执行器生成正确代码（评审器与验证器均通过），一次执行器生成不符合规范的代码（评审器因代码看起来合理而遗漏错误，验证器因测试失败而捕获错误）。

运行：

```
python3 code/main.py
```

## 使用它

`outputs/skill-role-designer.md` 接受一个任务并生成角色分工表（3-5个角色）、每个角色的输入/输出模式以及验证器检查。在将代理接入框架之前使用此功能。

## 发布

检查清单：

- **至少一个确定性验证器。** 永远不要全LLM。
- **每个角色明确的I/O模式。** 规划器返回规范，而非散文；执行器读取该模式。
- **沟通性去幻觉。** 执行器在信息缺失时必须询问规划器；切勿自行编造。
- **评审器/验证器顺序。** 先运行评审器（廉价，捕获设计问题），再运行验证器（慢，捕获错误）。
- **循环预算。** 在升级到人类之前，最多进行2轮评审器-执行器修订。

## 练习

1. 运行 `code/main.py` 并观察验证器如何捕获评审器遗漏的错误。添加一个静态分析检查（统计 `return` 的出现次数）作为额外的验证器。它能捕获运行时测试遗漏的什么错误？
2. 添加第5个角色：“需求分析师”，将用户愿望转化为规划器可用的规范。哪些沟通性去幻觉请求应向上传递？
3. 阅读 MetaGPT 第3节（“代理”）。列出 MetaGPT 5个角色中每个角色的输入/输出模式。
4. 阅读 ChatDev 的对话链图示（arXiv:2307.07924 图3）。识别沟通性去幻觉在何处打破了一个否则会无限循环的循环。
5. 普华永道7倍准确率提升来自于验证循环。假设三个添加验证器无济于事的任务——其中对正确性的确定性检查不可能或成本过高。

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|----------------|------------------------|
|  角色专业化 | “不同代理，不同工作” | 针对规划器/执行器/评审器/验证器角色调整的独特系统提示。 |
|  标准操作流程模式 | “编码化标准操作流程” | MetaGPT的框架：每个角色严格的I/O模式将团队转变为流水线。 |
|  沟通性去幻觉 | “先问再编” | ChatDev模式：执行器在细节缺失时询问规划器，而非自行编造。 |
|  评审器 | “LLM审查者” | 主观、有观点的审查者。捕获品味问题。可能会被看似合理的散文愚弄。 |
|  验证器 | “确定性检查” | 基于代码的通过/失败。测试运行器、类型检查器、模式验证器。无法被愚弄。 |
|  验证缺口 | “无人检查” | 占 MAST 失败的 21.3%。答案在未经过本可捕获错误的检查的情况下交付。 |
|  修订循环 | “评审器打回” | 评审器拒绝触发执行器根据反馈重新运行。需要预算。 |
|  全LLM反模式 | “看起来不错” | 每个角色都是LLM，没有确定性检查。经典的 MAST 失败。 |
