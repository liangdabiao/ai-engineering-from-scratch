# 生成式代理与涌现模拟

> Park 等人 2023 年论文（UIST '23, arXiv:2304.03442）使用三部分架构构建了**Smallville**——一个包含25个代理的沙盒环境：**记忆流**（自然语言日志）、**反思**（代理对其自身流生成的高级综合）和**计划**（天级别行为，然后是子计划）。标志性结果是情人节派对涌现现象：一个代理被植入“想要举办情人节派对”的种子，无需进一步脚本，邀请函就在人群中传播开来，各方协调日期，派对最终举行——而其他24个代理最初对此一无所知。消融实验表明，所有三个组件对可信度都是必需的。记录在案的错误包括空间规范错误（进入关闭的商店、共用单人浴室）。该架构是2026年代理模拟和多代理社会评估的参考架构。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置条件：** 阶段16 · 04（原始模型），阶段16 · 13（共享内存）
**时间：** ~75分钟

## 问题

大多数多代理系统都是严格脚本化的团队：规划者规划，编码者编码，审查者审查。这对于定义明确的任务有效。但它无法捕捉当代理拥有记忆、优先级和开放世界时涌现出的无脚本行为。研究、社会模拟以及日益增长的游戏AI需要这第二种类型。

Smallville架构是这方面的基准。在Park 2023之前，最好的代理模拟是浅层脚本追随者；之后，该模式成为开放世界中生成式代理的默认选择。如果你在2026年构建一个代理模拟，你要么使用Smallville的三个组件，要么明确论证为何不使用。

## 概念

### 三个组件

**记忆流。** 一个仅追加的日志，记录观察、动作、反思和计划。每条记录包含时间戳、类型、描述（自然语言）以及派生元数据：**近因性**、**重要性**（代理自评1-10）和**相关性**（与当前查询的余弦相似度）。

```
[2026-02-14 09:12:03] observation: Isabella Rodriguez asked me if I like jazz
[2026-02-14 09:14:22] reflection:   I enjoy long conversations about music
[2026-02-14 10:05:00] plan:         Attend Isabella's Valentine's Day party tonight
```

记忆检索结合三个分数：`score = w_recency * e^(-decay * age) + w_importance * importance + w_relevance * cos_sim`。Top-k条记录进入当前提示。

**反思。** 定期（每N条记忆或遇到重要事件时），代理从近期记忆中生成更高层次综合。反思条目回到记忆流中，并像其他记忆一样被检索。这是代理构建“理解”的方式——该架构中相当于长期信念的部分。

**计划。** 自上而下的分解。首先，一个粗略的天级别计划（“去上班，与Klaus共进晚餐”）。然后是小时级别计划。最后是动作级别计划。计划可修改：当观察与计划矛盾时，代理重新规划受影响的部分。

### 为何三者都重要（消融实验）

Park等人进行了消融实验，分别移除观察、反思和计划。每种消融都会损害可信度：

- 没有**观察**，代理会错过上下文并依据过时的信念行动。
- 没有**反思**，代理无法形成更高层次的信念；交互保持肤浅。
- 没有**计划**，行为变成反应性噪音；目标消散。

人类评分者的可信度评分在三者齐备时最高；移除任何一个都会带来可测量的退步。

### 情人节派对涌现现象

一个名为Isabella Rodriguez的代理被植入了目标“想要在2月14日下午5点在Hobbs Cafe举办情人节派对”。其他24个代理未收到此类种子。在模拟的天数中：

1. Isabella的计划包括邀请他人。
2. 每个邀请成为邻居记忆流中的一条观察。
3. 该邻居的反思生成信念：“Isabella要举办派对。”
4. 该邻居的计划纳入“2月14日参加派对”。
5. 邻居告诉其他邻居。邀请在没有中央协调的情况下传播开来。
6. 2月14日下午5点，多个代理聚集在Hobbs Cafe。

这就是技术意义上的涌现：系统级行为（派对）源于局部交互（双边邀请+个体计划）而没有中央协调者。

### 记录在案的失败模式

Park等人明确记录了：

- **空间规范错误。** 代理走进已关闭的商店。代理试图使用同一个单人浴室。代理在非用餐区域进食。模型无法仅从环境推断社会-物理规范。
- **记忆溢出。** 深度模拟运行导致记忆检索成本增长。实际补救措施：定期记忆压缩（总结并修剪）和对低重要性条目进行衰减。
- **反思幻觉。** 反思可能发明出记忆中不存在的关联。缓解措施：在反思提示中包含源记忆ID，并在检索时验证。

这些是与生产相关的失败模式：任何2026年的代理模拟都会继承它们。

### 三组件实现规则

1. **记忆是仅追加的。** 切勿修改记忆条目。更正通过新条目进行。
2. **重要性评分成本低廉。** 在写入时调用LLM评分重要性1-10。缓存该分数。
3. **检索是排序而非过滤的。** 按组合分数取top-k；不要使用硬性过滤（会丢失上下文）。
4. **反思定期运行。** 当未处理记忆的重要性总和超过阈值（例如150）时触发。
5. **计划可修改。** 当新观察与计划矛盾时，仅重新生成受影响的部分，而非整个计划。

### 超越Smallville的生成式代理

2024-2026年的后续文献扩展了该架构：

- **用于政策/市场研究的多代理社会模拟。** 类似Smallville的群体模拟用户对功能的行为反应。比A/B测试更快；准确性有争议。
- **游戏中的NPC AI。** 使用Smallville代理的RPG游戏产生涌现式故事情节，而非脚本化任务。
- **生成式代理评估基准。** 评估指标不再是任务准确性，而是长期运行中的可信度+行为连贯性。

该架构是参考。扩展组件（向量存储用于记忆、检索增强反思、神经符号计划）但保留三部分结构。

### 为何这对多代理工程重要

Smallville是一个概念验证：当组件正确时，多代理涌现是廉价的。该架构现在已在开源模型上复现（较小的LLM会优雅地损失可信度，而非急剧下降）。任何需要**涌现社会行为**的生产系统都采用这种形式。任何需要**严格任务执行**的系统则使用本阶段早期的监督者/角色/原语模式。

## 动手构建

`code/main.py`使用脚本化代理策略（无真实LLM）在标准库Python中实现了三个组件。该演示以小规模复现了情人节派对涌现现象：

- `MemoryStream` — 仅追加日志，支持按新旧/重要性/相关性检索。
- `MemoryStream` — 对近期高重要性记忆进行脚本化反思。
- `MemoryStream` — 基于当前信念的日级和小时级计划。
- 场景：5个智能体。智能体1以"下午5点举办派对"开始。在模拟的时间步中，邀请传播，智能体们汇聚。

运行：

```
python3 code/main.py
```

预期输出：逐时间步追踪。到最后一个时间步，5个智能体中至少3个的计划显示派对，并且他们在派对地点汇聚。单个种子产生了协调到达，无需任何协调者。

## 使用它

`outputs/skill-simulation-designer.md` 设计一个生成式智能体模拟：智能体数量、记忆模式、反思频率、计划视野和评估指标。

## 发布

生产模拟的规则：

- **记忆就是数据库。** 选择真实的存储（向量数据库、Postgres）进行规模化。内存标准库仅用于原型。
- **记录检索轨迹。** 对于每个动作，记录驱动该动作的top-k记忆。这是你的调试能力。
- **为每个智能体预算令牌。** 每个智能体每个时间步的检索+反思+计划调用O(k)次LLM。N个智能体×T个时间步×每次调用的令牌数可能会让你的预算相形见绌。
- **定期压缩记忆。** 总结并修剪低重要性条目。保留策略是一个设计决策，而非细节。
- **显式检测空间/社会规范违反。** 架构不会自行学会它们。

## 练习

1. 运行`code/main.py`。确认3个以上智能体在派对处汇聚。将智能体增加到10个——涌现行为是否仍然发生？
2. 移除反思步骤。行为看起来如何？映射到Park 2023中的消融发现。
3. 引入一个竞争性种子目标（"Klaus想要在下午5点做一个研究演讲"）。智能体会分裂，还是其中一个目标占主导？由什么决定？
4. 添加空间约束：Hobbs Cafe最多容纳4个智能体。模拟是否优雅地处理了溢出，还是遇到了"单人洗手间"失败模式？
5. 阅读Park等人 (arXiv:2304.03442) 第6节（涌现行为实验）。找出一个在你的微型模拟中无法重现的行为。你需要增强架构的哪个组件？
