# 故障模式 — 多智能体系统(MAST)、群体思维、单一文化、级联错误

> 2026年的参考分类法是 **MAST** (Cemri et al., NeurIPS 2025, arXiv:2503.13657)，该分类法源自7个最先进的开源多智能体系统(MAS)的1642条执行轨迹，显示 **41–86.7%的故障率**。三个根本类别：**规范问题** (41.77%) — 角色模糊、任务定义不清晰；**协调失败** (36.94%) — 沟通中断、状态不同步；**验证缺口** (21.30%) — 缺少验证、缺乏质量检查。**群体思维** 系列 (arXiv:2508.05687) 增加了：单一文化崩溃（相同基础模型 → 相关故障）、从众偏差（智能体相互强化错误）、心智理论缺陷、混合动机动态、级联可靠性故障。级联示例：重试风暴，支付失败触发订单重试，进而触发库存重试，最终压垮库存服务（数秒内负载增长10倍 — 需要断路器）。记忆中毒：一个智能体的幻觉进入共享内存，下游智能体将其视为事实；准确率逐渐下降，使根本原因诊断变得痛苦。**STRATUS** (NeurIPS 2025) 报告通过专门的检测/诊断/验证智能体，故障缓解成功率提升1.5倍。本课将故障模式视为一流工程目标。

**类型:** 学习
**语言:** Python (stdlib)
**前置条件:** 阶段16 · 13 (共享内存), 阶段16 · 14 (共识与拜占庭容错), 阶段16 · 15 (投票与辩论拓扑)
**时间:** ~75分钟

## 问题

多智能体系统在实际任务中失败的概率为41-86.7%（Cemri等人2025年在7个开源MAS中测得）。这不能通过“再加一些智能体”来调试。故障具有结构性原因。MAST分类法给出了类别。本课将每个类别映射到具体的检测、诊断和缓解模式，使这些数字不再显得随意。

2026年的生产实践是将故障模式作为设计输入。你的架构在针对每个MAST类别指出你所部署的缓解措施之前，都不能算“足够好”。

## 概念

### MAST类别

**规范问题 (占故障的41.77%)**。智能体的任务定义不够严格。示例：

- 角色模糊：两个智能体都认为自己是评审者。
- 任务描述不充分：用户想要特定角度时，只写了“总结一下”。
- 成功标准隐含：智能体无法判断是否成功。

缓解措施：
- 编写明确角色契约。每个智能体的提示词要说明它做什么 *以及不做什么*。
- 为每个任务设置验收测试。在智能体开始前，定义“完成的样子是X”。
- 前置规范检查：在任务分配前，由一个独立智能体审查任务定义。

**协调失败 (36.94%)**。通信或状态崩溃。

示例：
- 两个智能体无同步地更新共享状态。
- 智能体间消息丢失（队列故障、超时）。
- 状态漂移：智能体A认为任务完成；智能体B仍在执行。

缓解措施：
- 版本化共享状态，乐观并发控制。
- 关键消息需显式确认（重试直到确认）。
- 定期状态同步检查点；及早检测漂移。

**验证缺口 (21.30%)**。对输出缺乏独立检查。

示例：
- 一个智能体声称成功；无人验证。
- 智能体链中的每个智能体都信任前一个的输出。
- 对涌现的复合行为缺少测试覆盖。

缓解措施：
- 独立验证智能体（第13课）。只读、独立数据源访问。
- 显式交接契约：“A的输出必须在B开始前通过检查器C”。
- 结果日志记录供事后分析。

### 群体思维系列 (arXiv:2508.05687)

当智能体同质化或相互模仿时，五种相关故障：

**单一文化崩溃**。相同基础模型或训练数据 → 相关错误。当三个智能体共享同一个LLM时，它们共享其幻觉。

**从众偏差**。智能体向最响亮或最自信的同伴调整，即使对方错误。

**心智理论缺陷**。智能体无法彼此建模信念；协调失败（第18课）。

**混合动机动态**。具有部分一致激励的智能体向折中方案漂移，而折中方案不满足任何一方。

**级联可靠性故障**。一个组件的错误模式触发依赖组件中的错误模式。

### 级联示例 — 重试风暴

一个经典的2026年故障模式：

```
payment service fails 10% of requests
   ↓
order agent retries payment (exponential backoff but naive)
   ↓
each retry is a new order-inventory check
   ↓
inventory service sees 2x normal load
   ↓
inventory service starts timing out
   ↓
every order retries inventory check
   ↓
inventory service sees 10x normal load
   ↓
cluster goes down
```

修复方案是经典的：**断路器**。当下游错误率超过阈值时，用缓存或默认结果短路。再加上每个请求的重试预算上限。

断路器是少数几个你直接从分布式系统借用到多智能体故障缓解中而无需修改的方法之一。

### 记忆中毒（重访）

来自第13课：一个智能体的幻觉成为共享内存中的事实；下游智能体基于中毒事实进行推理。用MAST术语来说，这是共享内存层的验证缺口。

准确率逐渐下降是症状。你不会得到崩溃，而是缓慢的漂移，难以根本原因排查。

缓解措施：仅追加日志、溯源、不可写验证器。已在第13课中涵盖。

### STRATUS — 用于故障检测的专门化智能体(Agent)

STRATUS（NeurIPS 2025）报告称，当你部署以下内容时，缓解成功率提升1.5倍：

- **检测智能体(Detection agent)**：监测症状模式（高分歧、重试峰值、准确率漂移）。
- **诊断智能体(Diagnosis agent)**：根据症状，从MAST分类中推断可能的根本原因。
- **验证智能体(Validation agent)**：在应用缓解措施后，检查症状是否消除。

这是SRE风格的事件响应，应用于智能体系统。这三个角色都可以是配备专门提示词(Prompt)的LLM智能体。

### 故障模式审计

2026年的一项最佳实践是进行年度（或每主要版本）故障模式审计：

1. **追踪样本(Trace sample)**：收集大约1000个真实执行追踪。
2. **分类(Categorize)**：对于每个追踪的故障，映射到MAST + Groupthink类别。
3. **计算按类别的故障率(Compute failure-by-category rate)**：哪些类别在你的系统中占主导？
4. **对缓解措施排序(Rank mitigations)**：哪种修复能消除最多的故障？
5. **选择2-3个缓解措施(Pick 2-3 mitigations)**：实施；下季度重新审计。

纪律比具体选择更重要。没有审计，故障就会融入噪声，永远得不到系统性的处理。

### 当系统无声地失效时

最危险的故障类别是无声正确性故障。一个大声失效（崩溃、异常、告警）的系统可以被监控。一个产生看似合理但错误输出的系统无法通过异常日志检测到。这就是为什么验证缺口(Verification gap)是每个故障中最昂贵的类别，尽管按数量计算仅占21.30%。

投资于：
- 基于样本的人工审查。
- 黄金数据集回归测试。
- 对重要输出进行跨智能体交叉检查。

### 瞬时故障与缓慢故障

有些故障是瞬时的，有些是缓慢的。瞬时故障（超时、模式不匹配、认证错误）检测成本低。缓慢故障（记忆中毒、单一文化漂移、角色模糊）检测和预防成本高。

2026年的工程举措：为缓慢故障设置代理指标(Proxy)，以便在漂移变成可见错误之前捕捉到它。一致率、重试率、输出长度分布以及连续智能体版本之间的编辑距离(Edit distance)都是有用的代理指标。

## 发布

生产环境中的故障模式纪律：

- **每季度MAST审计**。不是每年。随着系统增长，类别会变化。
- **无处不在的断路器**。每个对依赖服务的出站调用。默认打开阈值为5-10%错误率。
- **黄金数据集**。小型、高质量、人工审计。每周进行回归测试。
- **STRATUS三件套**。检测 + 诊断 + 验证智能体监控生产环境。仅从检测智能体开始；当症状嘈杂时添加诊断。
- **故障预算**。按类别的明确SLO（服务级别目标）。超出预算会触发停止发布讨论。

## 练习

1. 运行`code/main.py`。确认断路器限制了重试风暴。改变故障阈值并观察权衡。
2. 实现一个**缓慢故障代理指标**：三个并行智能体之间的一致率。当它突然下降时，触发警报。通过逐渐关联智能体输出来模拟单一文化漂移。
3. 阅读Cemri等人（arXiv:2503.13657）。选择他们的7个MAS系统之一，映射其前3个故障类别。这些与MAST预测的相比如何？
4. 阅读Groupthink论文（arXiv:2508.05687）。识别哪五种模式在生产中最难检测。提出一个代理指标。
5. 为你熟悉的具体多智能体系统设计一个STRATUS风格的检测-诊断-验证三件套。检测关注哪些症状？诊断推荐哪些缓解措施？验证如何确认它们有效？

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|----------------|------------------------|
|  MAST  |  "2026年分类法"  |  Cemri 2025；3个根类别+14个子类型。  |
|  规范问题(Specification Problem)  |  "角色模糊(Role ambiguity)"  |  任务或角色定义不足；智能体不知道要做什么。  |
|  协调失败(Coordination Failure)  |  "状态漂移(State drift)"  |  智能体之间的通信或同步中断。  |
|  验证缺口(Verification Gap)  |  "无人检查(No one checked)"  |  输出在没有独立验证的情况下被接受。  |
|  群体思维家族(Groupthink family)  |  "同质化故障(Homogeneity failures)"  |  单一文化(Monoculture)、从众(Conformity)、心智理论缺陷(Deficient Theory of Mind)、混合动机(Mixed-motive)、级联(Cascading)。  |
|  单一文化崩溃(Monoculture collapse)  |  "相同模型，相同幻觉"  |  来自共享基础模型或训练数据的相关错误。  |
|  重试风暴(Retry storm)  |  "级联错误放大"  |  一个故障触发重试，从而放大下游负载。  |
|  断路器(Circuit breaker)  |  "在错误率上快速失败"  |  当错误率超过阈值时打开；使用默认值短路。  |
|  STRATUS  |  "事件响应三件套"  |  检测+诊断+验证智能体。1.5倍缓解成功率。  |
| 记忆污染 | "幻觉传播" | 共享内存事实被污染；下游智能体基于毒化信息推理。 |
