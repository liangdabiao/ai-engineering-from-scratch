# 共享内存与黑板模式

> 2026年的多智能体系统中并存两种方法：**消息池**（所有智能体都能看到彼此的消息，如AutoGen GroupChat或MetaGPT）和**带订阅的黑板**（智能体订阅相关事件，如上下文感知MCP或Matrix框架）。两者都是多智能体系统中唯一有状态的部分——这意味着它们也是有趣bug的常居地。参考故障模式是**记忆毒化**：一个智能体幻觉出一个"事实"，其他智能体视其为已验证，准确率逐渐衰减，这种问题比立即崩溃更难以调试。本课从标准库构建这两种结构，注入毒化攻击，并展示三种在生产中实际有效的缓解措施。

**类型：** 学习 + 构建
**语言：** Python（标准库，`threading`）
**先修要求：** 阶段16·04（原始模型），阶段16·09（并行群体网络）
**时间：** ~75分钟

## 问题

多智能体系统需要一个地方供智能体共享事实。一种字面做法是"在消息中传递一切"——但这用额外的复制重新发明了共享状态。另一种是"给每个人一个全局日志"——但全局日志无界增长且易被毒化。第三种是"为每个智能体投影一个视图"——可扩展但模式繁重。

当一个智能体产生幻觉并将幻觉写入共享状态时，每个读取该状态的下游智能体都会将该幻觉视为事实。等到人类注意到时，推理链已经深入了五步，而根本原因是第三个写入的消息。调试多智能体准确率衰减比调试崩溃更难。

这就是记忆毒化。它是MAST分类学（Cemri等人，arXiv:2503.13657）中记录次数第二多的故障家族，并且是结构性的：任何没有来源追踪和不可写验证器的共享内存设计最终都会出现此问题。

## 概念

### 两种主要拓扑结构

**全消息池。** 每个智能体读取每条消息。AutoGen GroupChat和MetaGPT使用此方法。简单、透明、可检查，但无法扩展到约10个智能体以上，因为每个智能体的上下文会被其他智能体的工作填满。

```
agent-A ──write──▶ ┌────────────────┐ ◀──read── agent-D
                   │ message pool   │
agent-B ──write──▶ │                │ ◀──read── agent-E
                   │ (global log)   │
agent-C ──write──▶ └────────────────┘ ◀──read── agent-F
```

**带订阅的黑板。** 智能体声明对主题的兴趣；底层只路由相关消息。CA-MCP（arXiv:2601.11595）和Matrix去中心化框架（arXiv:2511.21686）使用此方法。可扩展性更好，但需要预先设计模式以使订阅有意义。

```
                   ┌─ topic: prices ──┐
agent-A ──pub────▶ │                  │ ──▶ agent-D (subscribed)
                   ├─ topic: orders ──┤
agent-B ──pub────▶ │                  │ ──▶ agent-E (subscribed)
                   ├─ topic: alerts ──┤
agent-C ──pub────▶ │                  │ ──▶ agent-F (subscribed)
                   └──────────────────┘
```

### 各自优势的场合

- **全池**在智能体数量少（<10）、异构且对话短时获胜。当所有人都能看到一切时，推理谁说了什么变得简单。
- **黑板**在智能体数量多、角色同质但实例众多（群体）、且对话长时间运行时获胜。路由节省了token成本和上下文污染。

生产系统通常混合使用：顶层（规划层）使用小型全池，下层（工作层）使用黑板。

### 一个场景中的记忆毒化

三个智能体协作完成一项研究任务。智能体A是检索智能体。智能体B是摘要生成器。智能体C是分析智能体。

1. A获取一个页面并向共享状态写入一条消息："该研究报告了42%的准确率提升。"
2. 实际获取的页面写的是"4.2%的提升。" A把小数点幻觉了。
3. B读取共享状态后写道："报告大幅42%准确率提升（来源：A）。"
4. C读取共享状态后写道："建议采纳——42%的提升是变革性的。"
5. 最终报告引用了一个从未存在过的42%的数字。

没有智能体崩溃。没有测试失败。系统"正常工作"。幻觉通过共享状态从一个智能体的上下文传播到了每个下游智能体的推理中。

### 为什么这是结构性的

没有共享状态时，智能体A的幻觉只停留在A的上下文中。下游智能体会重新获取或重新推导，可能会发现错误。使用朴素的共享状态，A的上下文变成了所有人的上下文，幻觉被洗白为事实。

问题不在于共享状态本身——而在于共享状态**没有来源追踪和独立验证器**。三种缓解措施解决此问题：

1. **每次写入标注来源。** 共享状态中的每条记录记载了谁写的、何时写的、在什么提示下写的，以及（如适用）智能体引用了什么来源。下游智能体根据来源保持怀疑地读取。
2. **版本化写入；视为仅追加。** 修正是一条新记录覆盖旧记录，而非原地更新。审计轨迹得以保留。
3. **至少保留一个不能写入共享状态的智能体。** 只读验证器智能体采样条目，重新获取来源，并标记不一致之处。因为它不能写入消息池，所以不会被消息池毒化。

### 黑板先例（Hayes-Roth，1985）

黑板模式比LLM智能体早四十年。Hayes-Roth（1985，《用于控制的黑板架构》）描述了专业化的知识源，它们观察全局黑板，贡献部分解决方案，并触发其他知识源。2026年的黑板（CA-MCP，Matrix）是相同的模式，只不过知识源换成了LLM智能体，部分解决方案换成了JSON数据块。旧文献已记录了写入争用、机会主义控制和一致性问题的解决方案，现代系统正在重新发现它们。

### 投影视图与全视图

纯黑板为每个订阅者提供相同的投影（限定主题范围）。一种更具侵略性的设计是**每个智能体投影**：每个智能体获得一个根据其角色定制的视图。LangGraph的状态缩减器是2026年的典型实现——缩减器函数将全局状态折叠成角色特定的切片。

每个智能体投影可扩展性更好，但需要模式。没有模式，你会在每个智能体的提示中重新构建即席投影。

### 写入争用模式

多个智能体同时写入是一个并发问题，而不仅仅是LLM问题。三种有效模式：

- **顺序写入器（单一生产者）。** 所有写入通过一个协调者智能体串行处理。简单，但形成瓶颈。
- **带版本号的乐观并发。** 每条记录有版本号；写入者在版本不匹配时失败并重试。经典数据库技术。
- **主题分区。** 不同智能体拥有不同主题。无跨主题争用。需要设计分区边界。

大多数2026年框架默认使用顺序写入器，因为LLM调用速度足够慢，争用很少发生，瓶颈不会造成太大影响。

### 不可写入的验证器

最核心的缓解措施是只读验证器。实现规则：

- 验证器与团队共享状态（读取黑板或池）。
- 验证器没有对共享状态的写入句柄——仅能写入一个独立的验证通道。
- 验证器独立获取写入中引用的来源。标记不一致之处。
- 验证器的自身输出被路由到人类或独立的决策代理，绝不会反馈回池中。

没有这种分离，验证器的输出会成为池中的新条目，这意味着被污染的池会污染验证器，进而污染其验证结果。

## 动手构建

`code/main.py` 在 stdlib Python 中实现了两种拓扑结构、一个玩具投毒攻击以及三种缓解措施。

- `MessagePool` —— 线程安全的追加日志，支持完全读出。
- `MessagePool` —— 基于主题的发布/订阅，支持按代理订阅。
- `MessagePool` —— 每次写入记录（写入者、时间戳、提示哈希、来源URI）。
- `MessagePool` —— 运行一个三代理研究任务，其中代理A虚构一个小数。输出最终报告。
- `MessagePool` —— 一个只读代理，重新获取来源并标记不一致之处。在验证器存在的情况下运行相同场景。

运行：

```
python3 code/main.py
```

预期输出：
- 运行1（无验证器）：虚构的42%传播到最终报告。
- 运行2（有验证器）：验证器标记不一致之处，池被标记为“flagged”，最终报告包含撤回。

## 使用它

`outputs/skill-memory-auditor.md` 是一个评估任何多代理系统共享内存设计（包括来源追踪、版本控制和验证器分离）的技能。在投产前对新多代理架构运行此技能。

## 发布

对于任何共享内存设计：

- 每次写入记录来源：`(writer, timestamp, prompt_hash, tool_calls_cited, source_uri)`。
- 使日志仅追加。更正是引用被取代条目的新条目。
- 部署至少一个具有独立来源访问权限的只读验证器代理。
- 将验证器输出路由到独立通道，而不是返回共享池。
- 记录被取代写入的比例——上升比例是幻觉模式（Hallucination Pattern）的早期证据。

## 练习

1. 运行 `code/main.py`。确认运行1传播了幻觉，运行2捕获了它。
2. 添加第二个幻觉：代理B虚构一个数据集大小。验证器应能同时捕获两者，而无需为任一手动调整。
3. 将完整池切换为带有主题分区的黑板（`code/main.py`，`prices`，`summaries`）。主题分区使哪些投毒场景更难实施，对哪些场景无帮助？
4. 阅读 Hayes-Roth（1985，《A Blackboard Architecture for Control》）。找出论文中本课未讨论的两个控制模式，这些模式在2026年的系统中将受益匪浅。
5. 阅读 CA-MCP（arXiv:2601.11595）。将其共享上下文存储映射到 `code/main.py` 中的 MessagePool 或 Blackboard 类。CA-MCP 在此基础上添加了哪些原语？

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|----------------|------------------------|
|  消息池  |  "共享聊天历史"  |  每个代理读取的仅追加日志。完全透明，扩展性差。  |
|  黑板  |  "共享工作空间"  |  基于主题的发布/订阅。代理订阅相关主题。扩展性更好。  |
|  来源追踪  |  "谁写了什么"  |  每次写入的元数据：写入者、时间戳、提示、来源。  |
|  记忆投毒  |  "幻觉传播"  |  一个代理的错误进入共享状态，下游代理将其视为事实。  |
|  仅追加  |  "无原地更新"  |  更正是引用被取代条目的新条目。保留审计追踪。  |
|  不可写验证器  |  "独立审计者"  |  只读代理，重新获取来源并标记不一致之处。  |
|  投影  |  "限界视图"  |  从全局状态计算的每代理视图。LangGraph 化简器是典型范例。  |
|  知识源  |  "专家代理"  |  Hayes-Roth 1985年对黑板参与者的术语。  |
