# 层级架构及其失败模式

> 层级结构就是嵌套的管理者。管理者代理管理子管理者，子管理者管理工作者。CrewAI `Process.hierarchical` 是教科书式的版本：一个 `manager_llm` 动态分配任务并验证输出。LangGraph 的等价物是 `create_supervisor(create_supervisor(...))`。当任务本身就是一个真实的组织架构时，这是自然的模式。但它也是最容易陷入管理循环的模式——管理者代理分配工作不当、错误解释子输出、或无法达成共识。顺序结构通常优于它。

**类型：** 学习 + 构建
**语言：** Python（stdlib）
**前置要求：** 阶段 16 · 05（管理者模式）
**时间：** 约 60 分钟

## 问题

一旦理解了管理者模式，自然的下一个问题就是“如果工作者本身也是管理者会怎样？”团队有子团队；公司有部门的部门。层级架构反映了这一点。

问题在于：LLM 管理者与人类管理者不同。人类管理者对其下属的知识有稳定的先验认知。而 LLM 管理者每次都会根据上下文重新推理整个组织。上下文中微小的漂移，就会导致整个树形结构错误分配工作。

## 概念

### 形状

```
                 Manager
                 ┌─────┐
                 └──┬──┘
           ┌────────┴────────┐
           ▼                 ▼
       Sub-Mgr A         Sub-Mgr B
       ┌─────┐           ┌─────┐
       └──┬──┘           └──┬──┘
         ┌┴──┬──┐          ┌┴──┐
         ▼   ▼  ▼          ▼   ▼
       W1  W2  W3         W4  W5
```

每个内部节点负责规划、委派和综合。只有叶子节点执行工作。

### 优势之处

- **清晰的组织映射。** 如果真实任务是部门式的（“法务审阅文档、财务审阅文档、工程审阅文档，然后为执行层总结”），那么层级结构是明确的。
- **局部总结。** 每个子管理者在顶层管理者看到之前，先综合其团队输出。顶层管理者看到三个子管理者的总结，而不是十五个工作者的原始输出。

### 失效之处

2026 年的事后分析中反复发现的三种失败模式：

1. **任务分配错误。** 管理者阅读目标后，想象出一个分解方案，并委派给错误的子管理者。由于子管理者忠实地执行了分配的任务，错误直到顶层综合时才显现——距离人类可以发现错误的位置隔了一层。
2. **输出误解。** 子管理者返回“无法验证声明 X”。顶层管理者总结为“声明 X 未确认”。每一层都会导致含义漂移。
3. **共识循环。** 两个子管理者意见不一致；顶层管理者要求他们协调；他们向下重新委派；工作者重新运行；子管理者返回略有不同的答案；循环。CrewAI 的 `Process.hierarchical` 通过步数限制来防止这种情况，但限制本身现在成了一个超参数。

### 决定性判断

顺序（线性流水线）vs 层级：你的任务真的有独立的子团队，还是只是一个线性流程伪装成树形结构？如果是后者，使用顺序。如果是前者，使用层级但要设定明确的协调规则预算。

### CrewAI 的实现

`Process.hierarchical` 将一个管理者 LLM 连接到专业团队组上。管理者：

- 接收顶层任务，
- 将子任务分配给团队组，
- 评估团队组输出，
- 决定是接受、重新委派还是迭代。

文档：https://docs.crewai.com/en/introduction（查看核心概念下的“层级过程”）。

### LangGraph 的实现

LangGraph 使用嵌套的 `create_supervisor` 调用。内部管理者有自己的图；外部管理者将内部图视为一个不透明的节点。这在调试时比 CrewAI 更清晰（你可以分别逐步执行每个图），但更难以表达树形结构的动态重塑。

参考：https://reference.langchain.com/python/langgraph-supervisor.

## 动手构建

`code/main.py` 运行一个 3 层层级结构：

- 顶层管理者：将任务拆分为“工程”和“法务”分支，
- 工程子管理者：拆分为“前端”和“后端”工作者，
- 法务子管理者：一个工作者。

演示对比了理想路径（所有人意见一致）和一个**扰动路径**，其中顶层管理者的分解错误地将“法务”标记为“财务”，并观察错误级联——子管理者忠实地执行了财务工作，顶层综合器报告了财务结果，而原始的法务问题没有得到回答。

运行：

```
python3 code/main.py
```

输出显示了两个路径，并清晰并列了“被问的是什么”与“实际交付了什么”。

## 使用它

`outputs/skill-hierarchy-fitness.md` 评估给定任务应该使用层级、顺序还是扁平管理者。输入：任务描述、组织架构、协调预算。输出：模式推荐以及需要防范的具体失败模式。

## 发布

如果你部署层级架构：

- **限制树形深度为 2。** 三层已经会隐藏大多数错误，难以观察。
- **明确的协调预算。** 设置顶层管理者必须做出承诺的最大轮数。通常为 2。
- **每次综合都要有出处。** 每个节点的总结必须引用产生它的叶子输出。
- **分解漂移警报。** 记录管理者每一步的分解；与用户查询进行对比。如果分解不再覆盖查询，触发警报。

## 练习

1. 运行 `code/main.py` 并比较理想路径和扰动路径。经过多少层管理者交接后，顶层输出才完全偏离用户的问题？
2. 添加第三层（顶层 → 子 → 子子 → 工作者）。测量随着深度增加，扰动路径纠正自身与完全偏离的频率。
3. 在每个子管理者处实现一个“金丝雀”工作者，该工作者始终被问及原始用户问题（未更改）。使用金丝雀的回答来检测分解漂移。当金丝雀与综合答案不一致时，管理者应如何反应？
4. 阅读 CrewAI 的 `code/main.py` 文档。找出 CrewAI 应用的一个具体防护措施（步数限制、manager_llm 约束），并描述它针对哪种失败模式。
5. 比较嵌套的 LangGraph 管理者与 CrewAI 层级架构。哪种架构更容易检测协调循环？

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|----------------|------------------------|
|  层级  |  "组织架构模式"  |  管理者之上还有管理者；只有叶子节点执行工作。 |
| 管理者LLM  |  "老板"  |  在内部节点上分解、分配和验证的LLM。  |
| 分解漂移  |  "老板失去了方向"  |  顶层管理者的拆分不再覆盖原始问题。  |
| 协调循环  |  "无休止的会议"  |  下级管理者意见不合；顶层重新委派；工人重新运行；循环直到预算耗尽。  |
| 深度2上限  |  "不要超过2层深度"  |  经验护栏：3层以上会导致可观测性崩溃。  |
| 金丝雀问题  |  "每一层的真实答案"  |  始终询问未修改的原始查询的工作器，以检测漂移。  |
| 溯源链  |  "谁说了什么"  |  从每次综合回溯到产生它的叶子输出。  |
