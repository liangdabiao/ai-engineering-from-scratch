# Agent Observability（代理可观测性）: Langfuse、Phoenix、Opik

> 2026年三大开源代理可观测平台主导市场。Langfuse（MIT许可证）——每月600万+ SDK安装量，提供追踪、提示管理、评估和会话回放。Arize Phoenix（Elastic 2.0许可证）——深度代理特定评估、RAG相关性、OpenInference自动检测。Comet Opik（Apache 2.0许可证）——自动提示优化、护栏、LLM作为裁判的幻觉检测。

**类型:** 学习
**语言:** Python（标准库）
**先修条件:** 第14阶段·23（OTel GenAI）
**时间:** ~45分钟

## 学习目标

- 列举三大顶级开源代理可观测平台及其许可证。
- 区分每个平台的最强之处：Langfuse（提示管理 + 会话）、Phoenix（RAG + 自动检测）、Opik（优化 + 护栏）。
- 解释为什么到2026年89%的组织报告已部署代理可观测性。
- 实现一个基于标准库的追踪到仪表盘管道，并包含LLM作为裁判的评估。

## 问题

OTel GenAI（第23课）提供了模式。你仍然需要一个平台来接收跨度、运行评估、存储提示版本并发现回归。三个候选者各自强调生命周期中的不同部分。

## 核心概念

### Langfuse（MIT许可证）

- 每月600万+ SDK安装量，19000+ GitHub星标。
- 特性：追踪、支持版本管理和实验环境的提示管理、评估（LLM作为裁判、用户反馈、自定义）、会话回放。
- 2025年6月：原商业模块（LLM作为裁判、注释队列、提示实验、实验环境）在MIT许可证下开源。
- 最强项：端到端可观测性与紧密的提示管理循环。

### Arize Phoenix（Elastic License 2.0）

- 更深入的代理特定评估：追踪聚类、异常检测、RAG的检索相关性。
- 原生OpenInference自动检测。
- 与托管版Arize AX配合用于生产环境。
- 没有提示版本管理——定位为漂移/行为回归工具，与更广泛的平台配合使用。
- 最强项：RAG相关性、行为漂移、异常检测。

### Comet Opik（Apache 2.0）

- 通过A/B实验自动优化提示。
- 护栏（PII编辑、主题限制）。
- LLM作为裁判的幻觉检测。
- Comet自身测量的基准：Opik日志加评估耗时23.44秒，而Langfuse为327.15秒（约14倍差距）——请将厂商基准视为方向性参考。
- 最强项：优化循环、自动实验、护栏实施。

### 行业数据

根据Maxim（2026年领域分析）：89%的组织已部署代理可观测性；质量问题是最主要的生产障碍（32%的受访者提及）。

### 选择一个平台

|  需求  |  选择  |
|------|------|
|  一体化且包含提示管理  |  Langfuse  |
|  深度RAG评估 + 漂移检测  |  Phoenix  |
|  自动优化 + 护栏  |  Opik  |
|  开放许可，无ELv2  |  Langfuse（MIT）或Opik（Apache 2.0）  |
|  Datadog / New Relic集成  |  任意——它们都导出OTel  |

### 这种模式出错的地方

- **缺乏评估策略。** 没有评估的追踪只是昂贵的日志。
- **自建LLM裁判但未基于事实。** CRITIC模式（第05课）适用——裁判需要外部工具进行事实核查。
- **提示版本未与追踪关联。** 当生产环境出现回归时，你无法二分查找导致问题的提示版本。

## 动手构建

`code/main.py`实现了基于标准库的追踪收集器和LLM作为裁判的评估器：

- 接收GenAI形状的跨度。
- 按会话分组，标记失败的运行（触发护栏、低置信度评估）。
- 一个脚本化的LLM裁判，根据评分标准对代理响应打分。
- 类似仪表盘的摘要：失败率、主要失败原因、评估分数分布。

运行它：

```
python3 code/main.py
```

输出：每个会话的评估分数和失败分类，与Langfuse/Phoenix/Opik显示的内容一致。

## 使用它

- **Langfuse** 自托管或云端；通过OTel或其SDK接入。
- **Arize Phoenix** 自托管；自动检测OpenInference。
- **Comet Opik** 自托管或云端；自动优化循环。
- **Datadog LLM Observability** 适用于已运行Datadog的混合运维+机器学习团队。

## 发布

`outputs/skill-obs-platform-wiring.md`选择一个平台，将追踪、评估和提示版本接入现有代理。

## 练习

1. 导出一周的OTel追踪到Langfuse云端（免费层）。哪些会话失败了？为什么？
2. 为你所在的领域编写一个LLM裁判评分标准（事实正确性、语气、范围遵循）。在50条追踪上测试。
3. 比较Langfuse的提示版本管理和Phoenix的追踪聚类。哪个能更快告诉你哪里出了问题？
4. 阅读Opik的护栏文档。为你的一次代理运行添加一个PII编辑护栏。
5. 在你的语料库上对三个平台进行基准测试。忽略厂商发布的数字，测量你自己的。

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|----------------|------------------------|
|  追踪  |  "跨度收集器"  |  接收OTel / SDK跨度；按会话索引  |
|  提示管理  |  "提示CMS"  |  版本化提示并与追踪关联  |
|  LLM作为裁判  |  "自动评估"  |  独立LLM根据评分标准对代理输出打分  |
| 会话回放 | "轨迹回放" | 逐步调试过去的运行以进行调试 |
| RAG相关性 | "检索质量" | 检索到的上下文是否与查询匹配 |
| 追踪聚类 | "行为分组" | 对相似运行进行聚类以检测漂移 |
| 护栏执行 | "日志时策略" | 对记录内容进行PII/毒性/范围检查 |
