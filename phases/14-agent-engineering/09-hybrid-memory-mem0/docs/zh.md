# 混合记忆：向量+图+键值对(Mem0)

> Mem0 (Chhikara等人，2025)将记忆视为三个并行的存储——向量用于语义相似性，键值对用于快速事实查找，图用于实体-关系推理。在检索时，一个评分层融合三者。这是2026年外部记忆的生产标准。

**类型:** 构建
**语言:** Python (标准库)
**先决条件:** 阶段14·07 (MemGPT), 阶段14·08 (Letta 块)
**时间:** 约75分钟

## 学习目标

- 解释为什么单一存储（仅向量、仅图、仅键值对）不足以用于智能体记忆。
- 命名Mem0的三个并行存储以及各自优化的内容。
- 描述Mem0的融合评分——相关性、重要性、近因——以及为什么它是加权和而不是层级结构。
- 在标准库中实现一个玩具三存储记忆，包含一个写入所有三个存储的`add()`和一个融合结果的`search()`。

## 问题

单一存储对三类查询中的一类是错的：

- **语义相似性** — "上周我们讨论了关于智能体漂移的什么？"向量胜出；键值对和图缺失。
- **事实查找** — "用户的电话号码是什么？"键值对胜出；向量浪费资源，图大材小用。
- **关系推理** — "哪些客户共享同一个计费实体？"图胜出；向量和键值对无法回答。

生产环境中的智能体在一个会话中发出所有三类查询。单一存储记忆对其中两类总是错的。Mem0的贡献是将三者连接到一个单一的`add`/`search`接口后面，并附带一个融合它们的评分函数。

## 核心概念

### 三个并行存储

Mem0 (arXiv:2504.19413, 2025年4月) 在 `add(text, user_id, metadata)` 时：

1. 从文本中提取候选事实（由LLM驱动的步骤）。
2. 将每个事实写入向量存储（嵌入）以用于语义搜索。
3. 将每个事实写入键值对存储，键为(user_id, fact_type, entity)，用于O(1)查找。
4. 将每个事实写入图存储(Mem0g)作为类型化边，用于关系查询。

在 `search(query, user_id)` 时：

1. 向量存储返回基于嵌入余弦相似度的top-k。
2. 键值对存储返回基于查询派生的(user_id, type, entity)键的直接命中。
3. 图存储返回从查询实体可达的子图。
4. 一个评分层融合三者。

### 融合评分

```
score = w_relevance * relevance(q, record)
      + w_importance * importance(record)
      + w_recency * recency(record)
```

- **相关性** — 向量余弦、键值对精确匹配、图路径权重。
- **重要性** — 在写入时标记或学习（某些事实更重要：姓名、ID、策略）。
- **近因** — 自上次写入或读取以来的指数衰减。

权重按产品调整。聊天智能体使用更高的`w_recency`；合规智能体使用更高的`w_importance`；检索智能体使用更高的`w_relevance`。

### Mem0g和时间推理

Mem0g添加了一个冲突检测器。当新事实与现有边冲突时，现有边被标记为无效但不删除。时间查询（“用户三月份的城市是什么？”）遍历有效时间子图。

这是Letta的失效模式所泛化的合规级行为。

### 基准测试数字

Mem0论文报告（2025年）：

- **LoCoMo**（长篇幅对话记忆）：91.6
- **LongMemEval**（长周期情景记忆）：93.4
- **BEAM 1M**（1M令牌记忆基准）：64.1

比较基准（全上下文128k LLM、平面向量存储、平面KV）都落后10分以上。仅凭基准不能证明选择正确——运营形态才是关键——但数字表明融合设计不是四舍五入的误差。

### 范围分类

Mem0按范围划分记忆：

- **用户记忆** — 跨会话持久化，键为`user_id`。
- **会话记忆** — 在线程内持久化。
- **智能体记忆** — 每个智能体实例的状态。

每次写入选择一个范围。检索可以跨范围查询，并带不同范围权重。不加思考地混合范围会导致“助手告诉Alice关于Bob的项目”之类的事件。

### 这种模式出错的地方

- **嵌入漂移。** 前一百次查询中看似正确的向量结果随着语料库增长而退化。定期重新嵌入使用频率最高的前N条记录。
- **键值对结构蔓延。** `(user_id, type, entity)` 看起来简单，直到每个团队都添加自己的`type`。每季度审计类型集合。
- **图爆炸。** 一个嘈杂的提取器每消息添加50条边。限制每次`(user_id, type, entity)`调用的图写入；丢弃低置信度的边。

## 动手构建

`code/main.py` 在标准库中实现了三存储模式：

- `VectorStore` — 使用简单的令牌重叠相似度作为嵌入替代。
- `VectorStore` — 以`KVStore`为键的字典。
- `VectorStore` — 类型化边（主语，关系，宾语，有效）。
- `VectorStore` — 顶层外观，包含`KVStore`、`(user_id, fact_type, entity)`、融合评分和范围感知检索。
- 一个多用户、多会话对话的工作迹。

运行它：

```
python3 code/main.py
```

输出显示三个独立的召回路径以及融合后的top-k。翻转`main()`顶部的评分权重，观察排序变化。

## 使用它

- **Mem0（Apache 2.0）** — 生产就绪。使用Postgres + Qdrant + Neo4j自托管，或使用托管云。
- **Letta** — 三层核心/回忆/存档；自带向量和图后端。
- **Zep** — 商业替代方案，具有时序知识图谱和事实提取功能。
- **自定义构建** — 当需要对提取器（合规）或融合权重（时效性占主导的语音代理）进行精确控制时。

## 发布

`outputs/skill-hybrid-memory.md` 生成一个三存储记忆支架，集成了融合评分器、范围分类法和时间失效机制。

## 练习

1. 将演示向量相似度替换为真实嵌入模型（sentence-transformers、Ollama、OpenAI嵌入）。在合成长对话上测量recall@10。经过1000次写入后排名会漂移吗？
2. 添加一个时间查询：`search(query, as_of=timestamp)`。仅返回在该时间点或之前有效的记录。哪个存储需要最多工作？
3. 实现一个冲突检测器：如果传入事实与图边冲突，则使旧边失效并记录两者。测试“用户住在柏林” -> “用户住在里斯本”。
4. 将融合评分器移植以包含`search(query, as_of=timestamp)`维度（对检索到的记录点赞）。如何防止作弊（代理仅返回它已经喜欢的记录）？
5. 阅读Mem0文档（`search(query, as_of=timestamp)`）。将演示移植到`user_feedback`客户端调用。在相同的20个测试查询上比较检索质量。

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|----------------|------------------------|
|  混合记忆  |  "向量加图加KV"  |  三个存储并行写入，在检索时融合  |
|  事实提取  |  "记忆摄入"  |  将文本分解为（实体、关系、事实）三元组的LLM步骤  |
|  融合评分  |  "相关性排名"  |  相关性、重要性、时效性的加权和  |
|  范围  |  "记忆命名空间"  |  用户/会话/代理——决定谁看到什么  |
|  Mem0g  |  "记忆图"  |  具有时间有效性的类型化边，用于关系查询  |
|  时间失效  |  "软删除"  |  将冲突边标记为无效；永不删除  |
|  嵌入漂移  |  "检索衰减"  |  向量质量随语料增长而下降；定期重新嵌入  |
