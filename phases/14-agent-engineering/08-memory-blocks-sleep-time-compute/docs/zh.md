# 内存块与休眠时计算（Letta）

> MemGPT 在2024年更名为Letta。2026年的进化增加了两个概念：模型可以直接编辑的离散功能性记忆块，以及一个在主要代理空闲时异步整合记忆的睡眠时间代理。这就是你将记忆扩展到单次对话之外的方式。

**类型：** 构建
**语言：** Python (标准库)
**前置条件：** 阶段14 · 07 (MemGPT)
**时间：** ~75分钟

## 学习目标

- 说出Letta使用的三个记忆层级（核心、回忆、档案）以及每个层级的作用。
- 解释记忆块模式：人类块、角色块和用户定义的块作为第一类类型化对象。
- 描述睡眠时间计算是什么，为什么它不在关键路径上，以及为什么它可以运行比主要代理更强的模型。
- 实现一个脚本化的双代理循环，其中主要代理提供服务响应，而睡眠时间代理在轮次之间整合块。

## 问题

MemGPT（第07课）解决了虚拟内存的控制流问题。出现了三个生产问题：

1. **延迟。** 每次记忆操作都位于关键路径上。如果代理必须在用户等待时进行修剪、总结或协调，尾部延迟会激增。
2. **记忆腐烂。** 写入不断累积。矛盾的事实依然存在。检索被陈旧内容淹没。
3. **结构丢失。** 扁平化的档案存储无法表达“人类块始终在提示中；角色块始终在提示中；任务块按会话切换”。

Letta (letta.com) 是2026年的重写版。记忆块使结构显式化；睡眠时间计算将整合移出关键路径。

## 核心概念

### 三个层级

|  层级  |  范围  |  所在位置  |  由谁写入  |
|------|-------|----------------|------------|
|  核心  |  始终可见  |  主提示内部  |  代理工具调用 + 睡眠时间重写  |
|  回忆  |  对话历史  |  可检索  |  自动轮次日志  |
|  档案  |  任意事实  |  向量 + 键值 + 图  |  代理工具调用 + 睡眠时间摄入  |

核心是MemGPT的核心。回忆是带有被驱逐尾部的对话缓冲区。档案是外部存储。这种拆分清理了MemGPT的双层过载。

### 记忆块

块是核心层级中类型化、持久化、可编辑的部分。最初的MemGPT论文定义了两个：

- **人类块** — 关于用户的事实（姓名、角色、偏好、目标）。
- **角色块** — 代理的自我概念（身份、语气、约束）。

Letta将其推广到任意用户定义的块：一个用于当前目标的`Task`块，一个用于代码库事实的`Project`块，一个用于硬约束的`Safety`块。每个块都有`id`、`label`、`value`、`limit`（字符上限）、`description`（以便模型知道何时编辑它）。

块可通过工具表面进行编辑：

- `block_append(label, text)`
- `block_append(label, text)`
- `block_append(label, text)`
- `block_append(label, text)` — 压缩一个接近其上限的块。

### 睡眠时间计算

2025年Letta新增功能：在后台运行第二个代理，脱离关键路径。睡眠时间代理处理对话记录和代码库上下文，将`learned_context`写入共享块，并整合或失效档案记录。

由此产生的特性：

- **无延迟成本。** 主要响应无需等待记忆操作。
- **允许更强的模型。** 睡眠时间代理可以是更昂贵、更慢的模型，因为它不受延迟约束。
- **自然的整合窗口。** 在用户不等待时去重、总结、失效矛盾的事实。

这种模式与人类的工作方式相符：你完成任务，睡一觉，长期记忆在一夜之间沉淀。

### Letta V1 与原生推理

Letta V1 (`letta_v1_agent`, 2026) 弃用了`send_message`/心跳和内联`Thought:`标记，转而采用原生推理。Responses API (OpenAI) 和带有扩展思维的 Messages API (Anthropic) 在单独通道上发出推理，并通过轮次传递（在生产环境中跨提供者加密）。控制循环仍然是ReAct。思维轨迹是结构化的，而不是提示形状的。

### 这种模式出错的地方

- **块膨胀。** 无限的`block_append`很快达到上限。在写入超过上限之前，连接一个块总结器。
- **静默漂移。** 睡眠时间代理重写了一个块，而主要代理从未察觉。对块进行版本控制并在轨迹中显示差异。
- **中毒整合。** 睡眠时间代理将攻击者可访问的内容处理到核心中。第27课也适用于睡眠时间表面。

## 动手构建

`code/main.py` 实现：

- `Block` — id, label, value, limit, description。
- `Block` — CRUD + `BlockStore`辅助函数。
- 两个脚本化代理 — `Block`执行一轮，`BlockStore`在轮次之间整合。
- 一个轨迹显示三轮对话及其块写入，以及一次睡眠时间传递，该传递总结一个块并使一个陈旧事实失效。

运行它：

```
python3 code/main.py
```

记录显示了这种分离：主要轮次快速并产生原始写入；睡眠阶段压缩和清理。

## 使用它

- **Letta** (letta.com) 参考实现。自托管或托管云。
- **Claude Agent SDK 技能** 作为块状知识 — 技能是一个命名的、版本化的、可检索的指令块，代理按需加载。
- **自定义构建** 适用于希望控制存储后端的团队。使用Letta API合同，以便以后可以迁移。

## 发布

`outputs/skill-memory-blocks.md` 生成一个带有睡眠时间钩子的Letta形状块系统，适用于任何运行时，包括安全规则和引用连接。

## 练习

1. 添加一个`block_summarize`工具，当`near_limit`返回true时，用模型生成的摘要替换块值。哪个触发阈值能最小化摘要调用和块溢出？
2. 实现归档数据上的休眠去重：当两条记录文本的token重叠超过90%时，合并为一条。仅在休眠过程中执行，绝不在关键路径上执行。
3. 对块进行版本控制。每次写入时记录旧值和差异。暴露`block_summarize`，以便操作员调试"为什么代理忘记了X。"
4. 将休眠代理视为不可信的写入者。当它们触及Persona或安全块时，需要第二个代理审核后才能提交。
5. 将示例移植到使用Letta API（`block_summarize`）。块模式有何变化？原生推理如何改变追踪形状？

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|----------------|------------------------|
| 内存块 | "可编辑提示部分" | 类型化、持久化、LLM可编辑的核心内存段 |
| 人类块 | "用户内存" | 关于用户的事实，固定在核心中 |
| 人设块 | "代理身份" | 自我概念、语气、约束，固定在核心中 |
| 休眠计算 | "异步内存工作" | 第二个代理在关键路径之外进行整合 |
| 核心 / 回忆 / 归档 | "层级" | 三层内存划分：始终可见 / 对话 / 外部 |
| 块限制 | "上限" | 每块字符限制；强制摘要 |
| 原生推理 | "思考通道" | 提供者级别的推理输出，而非提示级别的`Thought:` |
| 学习到的上下文 | "休眠输出" | 休眠代理写入共享块的事实 |
