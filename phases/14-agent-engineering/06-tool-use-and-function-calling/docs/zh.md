# 工具使用与函数调用

> Toolformer (Schick et al., 2023) 开创了自监督工具标注。伯克利函数调用排行榜 V4 (Patil et al., 2025) 设定了2026年的标准：40% 智能体(Agentic)、30% 多轮(Multi-Turn)、10% 实时(Live)、10% 非实时(Non-Live)、10% 幻觉(Hallucination)。单轮(Single-turn)问题已解决。记忆(Memory)、动态决策(Dynamic Decision-Making)和长链工具(Long-Horizon Tool Chains)尚未解决。

**类型：** 构建
**语言：** Python (标准库)
**前置要求：** 阶段14 · 01 (智能体循环)、阶段13 · 01 (函数调用深入探讨)
**时间：** 约60分钟

## 学习目标

- 解释 Toolformer 的自监督训练信号：仅当工具执行降低了下一个令牌的损失时，才保留工具标注。
- 列出 BFCL V4 的五个评估类别及其各自衡量内容。
- 实现一个带有模式验证、参数强制类型转换和执行沙箱的标准库工具注册表。
- 诊断2026年的三个开放问题：长链工具、动态决策和记忆。

## 问题

早期的工具使用提问：模型能否预测正确的函数调用？现代的工具使用提问：模型能否在40个步骤中链式使用工具，具备记忆、部分可观察性、从工具故障中恢复，且不幻觉出不存在工具？

Toolformer 确立了基线：模型可以通过自监督学习何时调用工具。BFCL V4 定义了2026年的评估目标。两者之间的差距正是生产中智能体所处的空间。

## 核心概念

### Toolformer (Schick et al., NeurIPS 2023)

思路：让模型用自己的预训练语料库标注候选 API 调用。对于每个候选，执行它。仅当包含工具结果能降低下一个令牌的损失时才保留标注。在筛选后的语料库上微调。

涵盖的工具：计算器、问答系统、搜索引擎、翻译器、日历。自监督信号纯粹关于工具是否有助于预测文本——无需人工标签。

规模结果：工具使用随规模出现。较小的模型因工具标注而受损；较大的模型受益。这就是为什么2026年的前沿模型内置了强大的工具使用能力，而大多数7B模型需要明确的工具使用微调才能可靠。

### 伯克利函数调用排行榜 V4 (Patil et al., ICML 2025)

BFCL 是2026年的事实标准评估。V4 组成：

- **智能体 (40%)** — 完整的智能体轨迹：记忆、多轮、动态决策。
- **多轮 (30%)** — 带有工具链的交互式对话。
- **实时 (10%)** — 用户提交的真实提示（分布更困难）。
- **非实时 (10%)** — 合成测试用例。
- **幻觉 (10%)** — 检测何时不应调用任何工具。

V3 引入了基于状态的评估：在工具序列之后，检查 API 的实际状态（例如“文件是否创建？”），而不是匹配工具调用的 AST。V4 增加了网络搜索、记忆和格式敏感性类别。

2026年关键发现：单轮函数调用几乎已解决。失败集中在记忆（跨轮携带上下文）、动态决策（基于先前结果选择工具）、长链（20+步后漂移）和幻觉检测（拒绝调用不存在的工具）。

### 工具模式

每个提供商都有各自的模式。细节不同，但形状相同：

```
name: string
description: string (what it does, when to use it)
input_schema: JSON Schema (properties, required, types, enums)
```

Anthropic 直接使用 `input_schema`。OpenAI 使用 `function.parameters`。两者都接受 JSON Schema。描述至关重要——模型读取描述以选择正确的工具。糟糕的工具描述是选错工具失败的头号原因。

### 参数验证

不要信任任何工具调用。验证：

1. **类型强制转换。** 模型可能返回字符串 "5" 而模式要求整数。若明确则强制转换；否则拒绝。
2. **枚举验证。** 如果模式要求 `status in {"open", "closed"}` 而模型输出 `"in_progress"`，则拒绝并返回描述性错误。
3. **必填字段。** 缺少必填字段 -> 立即将错误观察返回给模型，而非崩溃。
4. **格式验证。** 日期、电子邮件、URL——使用具体解析器验证，而非正则表达式。

每次验证失败应返回结构化的观察结果，以便模型能够以正确的形状重试。

### 并行工具调用

现代提供商支持在一次助手轮次中进行并行工具调用。循环如下：

1. 模型发出3个带有不同 `tool_use_id` 的工具调用。
2. 运行时执行它们（如果独立则并行）。
3. 每个结果作为 `tool_use_id` 块返回，通过 `tool_result` 关联。

工程规则：将关联 ID 视为关键。交换它们会导致工具与结果路由错误。

### 沙箱化

工具执行是沙箱边界。详见第09课。简言之：每个工具应指定读/写表面、网络访问、超时、内存上限。通用的 `run_shell(cmd)` 是危险信号；具体的 `git_status()` 更安全。

## 动手构建

`code/main.py` 实现了生产级工具注册表：

- JSON Schema 子集验证器（仅标准库）。
- 工具注册，包含描述、输入模式、超时和执行器。
- 参数强制类型转换和枚举验证。
- 带关联 ID 的并行工具调度。
- 作为结构化字符串的错误观察。

运行它：

```
python3 code/main.py
```

跟踪显示一个小型智能体在一次轮次中调用三个工具，其中一次是故意格式错误的调用，被带有描述性错误拒绝，模型可根据此错误采取行动。

## 使用它

每个提供商都有自己的工具模式——Anthropic、OpenAI、Gemini、Bedrock。如果需要多提供商支持，请使用转换层（OpenAI Agents SDK、Vercel AI SDK、LangChain工具适配器）。BFCL是参考基准——如果工具使用是产品的核心，在发布前请针对你的智能体运行该基准。

## 发布

`outputs/skill-tool-registry.md` 为给定任务领域生成工具目录、模式和注册表。包括描述质量检查（每个工具的描述是否告诉了模型何时使用它？）。

## 练习

1. 添加一个“无操作”工具，让模型明确拒绝使用任何其他工具。在类似BFCL的幻觉测试上进行测量。
2. 实现整数作为字符串和浮点数作为字符串的参数强制转换。强制转换在什么时候开始隐藏真正的错误？
3. 为每个工具添加超时和断路器（连续3次失败后拒绝该工具60秒）。这对模型的恢复方式有何改变？
4. 阅读BFCL V4描述。选择一个类别（例如“多轮”），通过你的智能体运行10个示例提示。报告通过率。
5. 将stdlib验证器移植到Pydantic或Zod。Pydantic/Zod捕获了哪些玩具验证器遗漏的问题？

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|----------------|------------------------|
|  函数调用  |  "工具使用"  |  带验证模式的结构化输出工具调用  |
|  Toolformer  |  "自监督工具标注"  |  Schick 2023 — 保留其结果能减少下一个词损失的工具调用  |
|  BFCL  |  "伯克利函数调用排行榜"  |  2026基准测试：40%智能体、30%多轮、10%实时、10%非实时、10%幻觉  |
|  工具模式  |  "模型的函数签名"  |  名称、描述、参数的JSON Schema  |
|  tool_use_id  |  "关联ID"  |  将工具调用与其结果绑定；对于并行调度至关重要  |
|  幻觉检测  |  "知道何时不调用"  |  V4类别：当没有工具适合时拒绝调用  |
|  参数强制转换  |  "字符串到整数的修复"  |  对可预测的模式不匹配进行窄修复；如果歧义则拒绝  |
|  沙盒化  |  "工具执行边界"  |  每个工具的读/写表面、网络、超时、内存上限  |
