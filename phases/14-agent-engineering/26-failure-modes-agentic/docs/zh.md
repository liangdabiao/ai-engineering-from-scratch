# 故障模式：智能体为何崩溃

> MASFT（加州大学伯克利分校，2025）将14种多智能体故障模式(Multi-Agent Failure Mode)归为3类。微软的分类法(Microsoft's Taxonomy)记录了现有AI故障如何在智能体环境中被放大。行业现场数据汇聚出五种反复出现的模式：幻觉行为(Hallucinated Actions)、范围蔓延(Scope Creep)、级联错误(Cascading Errors)、上下文丢失(Context Loss)、工具误用(Tool Misuse)。

**类型：** 学习+构建
**语言：** Python（标准库）
**先决条件：** 阶段14 · 05（自我精炼与CRITIC），阶段14 · 24（可观测性）
**时间：** 约60分钟

## 学习目标

- 列举MASFT的三个故障类别，每个类别至少包含四种具体模式。
- 解释为什么智能体故障会放大现有的AI故障模式（偏差、幻觉）。
- 描述五种行业反复出现的模式及其缓解措施。
- 实现一个标准库检测器，用故障模式标签标记智能体轨迹。

## 问题

团队部署的智能体在90%的轨迹上工作正常。但那10%的失败并非随机噪声——它们属于少数几个反复出现的类别。一旦你能够命名它们，就可以进行监控并修复。

## 核心概念

### MASFT（加州大学伯克利分校，arXiv:2503.13657）

多智能体系统故障分类法(Multi-Agent System Failure Taxonomy)。14种故障模式聚类为3个类别。标注者间Cohen's Kappa系数为0.88——这些类别可靠可区分。

核心主张：故障是多智能体系统的根本设计缺陷，而非可通过更优基础模型修复的LLM局限性。

### 微软的《智能体AI系统故障模式分类法》(Microsoft Taxonomy of Failure Mode in Agentic AI Systems)

- 现有AI故障（偏差、幻觉、数据泄露）在智能体环境中被放大。
- 自主性催生了新的故障：大规模意外行为、工具误用、任务漂移。
- 该白皮书是智能体产品的风险登记册。

### 《智能体AI故障特征化》(Characterizing Faults in Agentic AI)（arXiv:2603.06847）

- 故障源于编排、内部状态演进和环境交互。
- 不仅仅是“糟糕的代码”或“糟糕的模型输出”。

### 《LLM智能体幻觉调查》(LLM Agent Hallucinations Survey)（arXiv:2509.18970）

两种主要表现：

1. **指令遵循偏差(Instruction-following Deviation)** ——智能体不遵循系统提示。
2. **长程上下文误用(Long-range Contextual Misuse)** ——智能体忘记或误用早期轮次的上下文。

子意图错误：遗漏（遗漏步骤）、冗余（重复步骤）、混乱（步骤顺序错误）。

### 五种行业反复出现的模式

Arize、Galileo、NimbleBrain在2024-2026年的现场分析汇聚于：

1. **幻觉行为(Hallucinated actions)。** 智能体调用不存在的工具或编造参数。
2. **范围蔓延(Scope creep)。** 智能体将任务扩展到用户要求之外（创建额外的PR、发送额外的邮件）。
3. **级联错误(Cascading errors)。** 一个错误的调用触发下游效应。一个幻影SKU幻觉触发四次API调用——成为多系统事故。
4. **上下文丢失(Context loss)。** 长时任务忘记早期轮次的约束。
5. **工具误用(Tool misuse)。** 用错误参数调用正确工具，或者完全调用错误工具。

级联是致命的。智能体无法区分“我失败了”与“任务不可能完成”，并常常在400错误上幻觉出一个成功消息以结束循环。

### 缓解措施：每一步设置关卡

在推理链的每一步设置自动验证关卡，对照环境状态检查事实依据。具体来说：

- 逐步骤安全分类器（第21课）。
- 工具调用参数验证（第06课）。
- 交叉检查检索内容与已知事实（第05课，CRITIC）。
- 通过重新探测状态检测成功幻觉（文件是否实际创建？）。

### 故障监控的误区

- **仅标记崩溃。** 大多数智能体故障会产生看似有效的输出。需要进行内容级检查。
- **没有基线。** 漂移检测需要已知的最后正常状态；没有它你就无法说“情况正在恶化”。
- **过度告警。** 每个故障都产生一页告警。应当聚类并限速。

## 动手构建

`code/main.py` 实现了一个标准库故障模式标记器：

- 一个覆盖五种模式的合成轨迹数据集。
- 每种模式的检测器函数（工具调用、输出、重复动作上的签名模式）。
- 一个标记每个轨迹并报告模式分布的标记器。

运行它：

```
python3 code/main.py
```

输出：每条轨迹的标签加上聚合分布，这是Phoenix轨迹聚类所呈现结果的廉价复现。

## 使用它

- **Phoenix** 用于生产漂移聚类（第24课）。
- **Langfuse** 用于会话回放+标注。
- **自定义** 用于你的可观测性平台无法检测的特定领域签名。

## 发布

`outputs/skill-failure-detector.md` 生成针对你的领域定制的故障模式检测器，并与轨迹存储连接。

## 练习

1. 为“成功幻觉”添加一个检测器：智能体返回成功但目标状态未改变。
2. 从你构建的产品中标记100条真实轨迹。哪种模式占主导？修复它的成本是多少？
3. 实现一个“级联半径”指标：给定步骤N的一个故障，它影响了下游多少步骤？
4. 阅读MASFT的14种故障模式。挑选三个适用于你产品的模式。编写检测器。
5. 将一个检测器接入CI任务：如果>=5%的轨迹标记了某种模式，则构建失败。

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|----------------|------------------------|
| MASFT  |  “多智能体失败分类”  |  Berkeley 14模式分类 |
| 级联错误  |  “涟漪失败”  |  一个早期错误通过N步传播 |
| 上下文丢失  |  “忘记约束”  |  长视界回合丢失早期回合的事实 |
| 工具误用  |  “工具错误/参数错误”  |  有效调用，错误调用方式 |
| 成功幻觉  |  “虚假完成”  |  智能体在400错误上声称成功；状态未改变 |
| 范围蔓延  |  “过度扩展”  |  智能体做了超出要求的事情 |
| 指令遵循偏差  |  “不服从”  |  忽略系统提示或用户约束 |
| 子意图错误  |  “计划缺陷”  |  计划执行中的遗漏、冗余、混乱 |
