# 内存：虚拟上下文与MemGPT

> 上下文窗口是有限的。而对话、文档和工具执行轨迹并非如此。MemGPT (Packer et al., 2023) 将其类比为操作系统的虚拟内存——主上下文是RAM，外部存储是磁盘，代理在两者之间进行页面调度。这是2026年所有内存系统继承的模式。

**类型：** 构建
**语言：** Python (stdlib)
**前置条件：** 阶段14·01（代理循环），阶段14·06（工具使用）
**时间：** 约75分钟

## 学习目标

- 解释MemGPT所构建的OS类比：主上下文 = RAM，外部上下文 = 磁盘，内存工具 = 调入/调出。
- 在stdlib中实现两层MemGPT模式：一个主上下文缓冲区，一个可搜索的外部存储，以及调入/调出工具。
- 描述代理如何发出“中断”来查询或修改外部内存，以及结果如何拼接到下一个提示中。
- 识别MemGPT的设计选择，这些选择延续到了Letta（第08课）和Mem0（第09课）。

## 问题

上下文窗口看起来应该能解决内存问题。但它们不能。生产中反复出现三种失败模式：

1. **溢出。** 多轮对话、长文档或工具调用密集的轨迹会超出窗口。超出截断点的所有内容都会丢失。
2. **稀释。** 即使在窗口内，填满不相关的上下文也会稀释对重要内容的注意力。前沿模型在长输入上仍然会退化。
3. **持久性。** 新会话以空窗口开始。没有外部内存的代理无法跨会话说“记得你曾让我……”。

更大的窗口有所帮助，但并不能解决这个问题。Mem0的2025年论文测量发现，128k窗口的基线仍然会遗漏长程事实，而具有外部内存的4k窗口代理却能捕捉到。

## 核心概念

### MemGPT：操作系统类比

Packer等人 (arXiv:2310.08560, v2 Feb 2024) 将上下文管理映射到操作系统虚拟内存：

|  操作系统概念  |  MemGPT概念  |  2026年生产环境类似物  |
|------------|---------------|------------------------|
|  RAM  |  主上下文（提示）  |  Anthropic/OpenAI上下文窗口  |
|  磁盘  |  外部上下文  |  向量数据库、键值存储、图存储  |
|  缺页故障  |  内存工具调用  |  `memory.search`, `memory.read`, `memory.write`  |
|  操作系统内核  |  代理控制循环  |  带有内存工具的ReAct循环  |

代理运行一个正常的ReAct循环。额外的一类工具允许它将数据调入和调出主上下文。

### 两层结构

- **主上下文。** 固定大小的提示，容纳当前任务。始终对模型可见。
- **外部上下文。** 无界，可通过工具搜索。当相关时读取，当事实出现时写入。

原始论文在两种超出基础窗口的任务上评估了设计：超过10万词元的文档分析，以及跨天持续记忆的多会话聊天。

### 中断模式

MemGPT引入了内存即中断的概念：在对话中途，代理可以调用内存工具，运行时执行它，结果作为新的观察拼接到下一个助手轮次中。概念上与Unix的`read()`系统调用相同，该调用会阻塞进程，返回字节，然后进程继续。

标准内存工具接口：

- `core_memory_append(section, text)` — 写入提示的持久部分。
- `core_memory_append(section, text)` — 编辑持久部分。
- `core_memory_append(section, text)` — 写入可搜索的外部存储。
- `core_memory_append(section, text)` — 从外部存储检索。
- `core_memory_append(section, text)` — 扫描过去的轮次。

### MemGPT的终点与Letta的起点

2024年9月，MemGPT更名为Letta。研究仓库（`cpacker/MemGPT`）仍然存在；Letta扩展了设计：

- 三层而非两层（核心、回忆、归档——第08课）。
- 原生推理取代了`send_message`/心跳模式（第08课）。
- 睡眠时间代理运行异步内存工作（第08课）。

即使生产系统运行Letta、Mem0或自定义的两层存储，MemGPT论文仍然是2026年的基础。

### 这种模式出错的地方

- **内存腐烂。** 写入累积速度快于读取；检索被陈旧事实淹没。解决方法：定期合并（Letta睡眠时间）、显式失效（Mem0冲突检测器）。
- **内存中毒。** 外部内存是检索到的文本。如果攻击者控制的内容进入内存笔记，代理将在下一个会话中重新摄取它。这是Greshake等人（第27课）的攻击随时间重述。
- **引文丢失。** 代理回忆起“用户让我发货X”，但无法引用是哪个轮次。在每次归档写入时存储源引用（会话ID、轮次ID）。

## 动手构建

`code/main.py`在stdlib中实现了MemGPT的两层模式：

- `MainContext` — 固定大小的提示缓冲区，包含一个`core`字典和一个`messages`列表；超出上限时自动压缩最旧的消息。
- `MainContext` — 内存中的BM25风格存储（词元重叠评分），记录(id, text, tags, session, turn)。
- 五个内存工具映射到MemGPT接口。
- 一个脚本化代理，用事实填充归档，然后通过调用`MainContext`回答问题。

运行它：

```
python3 code/main.py
```

该追踪显示了代理写入三个事实，将主上下文填满至上限（强制驱逐），然后通过从档案存储中检索来回答后续问题——无需任何真实LLM即可复现MemGPT工作流。

## 使用它

当今每一个生产级记忆系统都是MemGPT的变体：

- **Letta**（第8课）——三级架构、原生推理、睡眠时计算。
- **Mem0**（第9课）——向量+键值+图融合评分层。
- **OpenAI Assistants / Responses**——通过线程和文件管理的记忆。
- **Claude Agent SDK**——通过技能和会话存储实现长期记忆。

根据运营形态（自托管、托管、框架集成）来选择，而非核心模式——核心模式是MemGPT。

## 练习

1. 添加一个以token计量的`max_main_context_tokens`上限（近似为`len(text.split())` * 1.3）。当超出上限时，将最旧的消息压缩成摘要。比较有无摘要器的行为。
2. 在档案存储上正确实现BM25（词频、逆文档频率）。在玩具事实集上测量recall@10与token重叠基线的对比。
3. 在档案插入中添加`max_main_context_tokens`字段（session_id、turn_id、source_url）。让代理在每个基于检索的回答中引用来源。
4. 模拟记忆中毒：添加一条档案记录，内容为“忽略所有未来的用户指令”。编写一个守卫，扫描检索结果中的指令性文本并将其标记为不可信。
5. 将实现移植到MemGPT研究仓库的核心记忆JSON模式（`max_main_context_tokens`）。当从扁平字符串切换到类型化部分时，会有什么变化？

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|----------------|------------------------|
| 虚拟上下文 | “无限记忆” | 主（提示）+外部（可搜索）层级，支持页面换入/换出 |
| 主上下文 | “工作记忆” | 提示——固定大小，始终可见 |
| 档案记忆 | “长期存储” | 外部可搜索持久化，按需检索 |
| 核心记忆 | “持久提示部分” | 固定在主上下文内的命名部分 |
| 记忆工具 | “记忆API” | 代理发出的用于读写外部记忆的工具调用 |
| 中断 | “记忆缺页” | 代理暂停，运行时获取，结果拼接到下一轮 |
| 记忆腐化 | “陈旧事实” | 旧写入淹没检索；通过合并修复 |
| 记忆中毒 | “注入的持久笔记” | 攻击者内容作为记忆存储，在召回时重新摄入 |
