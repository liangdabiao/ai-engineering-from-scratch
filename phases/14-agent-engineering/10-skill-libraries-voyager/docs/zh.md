# 技能库与终身学习（Voyager）

> Voyager（Wang等，TMLR 2024）将可执行代码视为一种技能。技能具有名称、可检索、可组合，并能通过环境反馈进行优化。这是Claude Agent SDK技能、skillkit以及2026年技能库模式的参考架构。

**类型:** 构建
**语言:** Python (标准库)
**先决条件:** 阶段14·07 (MemGPT), 阶段14·08 (Letta 块)
**时间:** 约75分钟

## 学习目标

- 列举Voyager的三个组件——自动课程、技能库、迭代提示——以及各自的作用。
- 解释为什么Voyager将动作空间设为代码而非原始命令。
- 实现一个具有注册、检索、组合和失败驱动优化的stdlib技能库。
- 将Voyager的模式映射到2026年Claude Agent SDK技能和skillkit生态系统中。

## 问题

在每个会话中从头重建所有能力的智能体犯了三个错误：

1. **浪费令牌。** 每个任务都重新触发相同的推理。
2. **丢失进展。** 在会话A中学会的修正不会迁移到会话B。
3. **难以完成长期组合。** 复杂任务需要能力层级结构；一次性提示无法表达它们。

Voyager的答案：将每个可重用能力视为存储在库中的命名代码块，可通过相似性检索，与其他技能组合，并通过执行反馈进行优化。

## 核心概念

### 三个组件

Voyager（arXiv:2305.16291）围绕以下内容构建智能体：

1. **自动课程。** 一个好奇心驱动的提议器根据智能体当前的技能集和环境状态选择下一个任务。探索是自下而上的。
2. **技能库。** 每个技能都是可执行代码。当任务成功时添加新技能。通过查询与描述的相似性检索技能。
3. **迭代提示机制。** 失败时，智能体接收执行错误、环境反馈和自我验证输出，然后优化技能。

Minecraft评估（Wang等，2024）：与基线相比，独特物品增加3.3倍，石制工具速度提高8.5倍，铁制工具速度提高6.4倍，地图遍历距离延长2.3倍。这些数字是Minecraft特有的，但模式可以迁移。

### 动作空间 = 代码

大多数智能体发出原始命令。Voyager发出JavaScript函数。一个技能是：

```
async function craftIronPickaxe(bot) {
  await mineIron(bot, 3);
  await mineStick(bot, 2);
  await placeCraftingTable(bot);
  await craft(bot, 'iron_pickaxe');
}
```

由子技能组合而成。以描述和嵌入为键存储。作为程序检索，而非提示。

这就是2026年Claude Agent SDK技能：一个命名的、可检索的代码块以及智能体按需加载的指令。

### 技能检索

新任务“制作钻石镐”。智能体：

1. 嵌入任务描述。
2. 查询技能库以获取最相似的k个技能。
3. 检索`craftIronPickaxe`、`mineDiamond`、`placeCraftingTable`等。
4. 从检索到的原语和新逻辑组合出新技能。

这是MCP资源（阶段13）和Agent SDK技能实现的模式：在知识/代码表面上进行检索，范围限定于当前任务。

### 迭代优化

Voyager的反馈循环：

1. 智能体编写一个技能。
2. 技能在环境中运行。
3. 返回三种信号之一：`success`、`error`（带堆栈跟踪）、`self-verification failure`。
4. 智能体以信号为上下文重写技能。
5. 循环直至成功或达到最大轮次。

这是Self-Refine（第05课）应用于代码生成，并加入基于环境的验证。CRITIC（第05课）是相同的模式，但使用外部工具作为验证器。

### 课程与探索

Voyager的课程模块根据智能体已拥有和尚未完成的任务，提出诸如“在湖边建一个避难所”之类的任务。提议器利用环境状态和技能清单，选择一个略高于当前能力的任务——探索的甜蜜点。

对于生产级智能体，这转化为一个“缺失什么”的运算符：给定当前技能库和领域，我们尚未覆盖哪些技能？团队通常手动实现为课程审查。

### 这种模式出错的地方

- **技能库腐化。** 同一技能以略有差异的描述添加了10次。在写入时添加去重；检索只返回一个。
- **组合技能漂移。** 父技能依赖于一个已被优化的子技能。对技能进行版本控制；固定为v1的父技能不会神奇地获取v3。
- **检索质量。** 当库增长到几百个以上时，对技能描述的向量检索会退化。用标签过滤和硬约束（“仅包含`category=tooling`的技能”）作为补充。

## 动手构建

`code/main.py`实现了一个stdlib技能库：

- `Skill` — 名称、描述、代码（字符串形式）、版本、标签、依赖项。
- `Skill` — 注册、搜索（令牌重叠）、组合（依赖拓扑排序）和优化（更新时版本升级）。
- 一个脚本智能体，注册三个原始技能，组合出第四个，遇到失败然后优化。

运行它：

```
python3 code/main.py
```

跟踪显示库写入、检索、组合、一次失败的执行以及v2优化——Voyager的完整循环。

## 使用它

- **Claude Agent SDK技能**（Anthropic）——2026参考：每个技能包含描述、代码和指令；在智能体会话期间按需加载。
- **skillkit**（npm: skillkit）——面向32+个AI编码智能体的跨智能体技能管理。
- **自定义技能库**——领域特定（数据智能体的SQL技能，基础设施智能体的Terraform技能）。Voyager模式可以向下扩展。
- **OpenAI Agents SDK `tools`**——低端方案；每个工具是一个轻量级技能。

## 发布

`outputs/skill-skill-library.md` 生成一个Voyager形状的技能库，内置注册、检索、版本控制和优化功能，适用于任何目标运行时。

## 练习

1. 向`compose()`添加依赖循环检测器。当技能A依赖技能B，而技能B又依赖技能A时会发生什么？错误还是警告？
2. 实现每个技能的版本锁定。当父技能组合子技能`compose()`时，对`crafting@1`的优化不得静默升级父技能。
3. 用sentence-transformers嵌入（或BM25标准库实现）替换token重叠检索。在包含50个技能的玩具库上测量retrieval@5。
4. 添加一个“课程(cirriculum)”代理：给定当前库和领域描述，提出5个缺失的技能。每周调用一次。
5. 阅读Anthropic的Claude Agent SDK技能文档。将玩具库移植到SDK的技能模式。可发现性会如何改变？

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|----------------|------------------------|
|  技能  |  "可重用能力"  |  命名的代码块+描述，可通过相似性检索  |
|  技能库  |  "代理的如何做记忆"  |  技能的持久存储，可搜索且可组合  |
|  课程  |  "任务提出者"  |  由当前能力差距驱动的自下而上的目标生成器  |
|  组合  |  "技能DAG"  |  技能调用技能；执行时拓扑排序  |
|  迭代优化  |  "自纠正循环"  |  环境反馈+错误+自我验证折叠回下一个版本  |
|  动作空间即代码  |  "程序化动作"  |  发出函数而非原始命令，以实现时间延伸的行为  |
|  写入时去重  |  "技能坍缩"  |  近似重复的描述坍缩为一个规范技能  |
