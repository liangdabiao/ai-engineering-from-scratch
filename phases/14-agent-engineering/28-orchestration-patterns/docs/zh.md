# 编排模式：监督者、集群、分层

> 在2026年的框架中，四种编排模式反复出现：主管-工人、群体/点对点、层级、辩论。Anthropic的指导原则："构建适合你需求的系统。"从简单开始；只有当单个智能体加上五种工作流模式不够时才添加拓扑结构。

**类型：** 学习+构建
**语言：** Python（标准库）
**先决条件：** 第14阶段·12（工作流模式），第14阶段·25（多智能体辩论）
**时间：** ~60分钟

## 学习目标

- 说出四种反复出现的编排模式以及各自适用的场景。
- 描述2026年LangChain的建议：基于工具调用的监督与监督库。
- 解释Anthropic的"构建正确系统"规则以及它如何决定拓扑选择。
- 使用标准库针对一个通用脚本化LLM实现所有四种模式。

## 问题

团队在需要之前就追求"多智能体"。四种模式在框架中反复出现；一旦你能命名它们，就可以选择正确的模式——或者完全跳过拓扑结构。

## 核心概念

### 主管-工人

- 一个中央路由LLM将任务分派给专家智能体。
- 决定：返回自身、移交给专家、终止。
- 专家之间不直接通信；所有路由都通过主管。

框架：LangGraph `create_supervisor`，Anthropic orchestrator-workers，CrewAI层级流程。

**2026年LangChain建议：** 通过直接工具调用进行监督，而不是使用`create_supervisor`。这样可以更精细地控制上下文工程——你可以精确决定每个专家看到什么。

### 群体/点对点

- 智能体通过共享工具接口直接移交。
- 没有中央路由器。
- 延迟低于主管模式（跳数更少）。
- 更难推理（没有单一控制点）。

框架：LangGraph群体拓扑，OpenAI Agents SDK移交（当所有智能体可以相互移交时）。

### 层级

- 主管管理子主管，子主管管理工人。
- 在LangGraph中实现为嵌套子图；在CrewAI中实现为嵌套团队。
- 可扩展到大量智能体群体，但代价是操作复杂性增加。

何时需要：当单个主管的上下文预算无法容纳所有专家的描述时。

### 辩论

- 并行提议者 + 迭代交叉批评（第25课）。
- 不完全是编排——更像是验证——但在框架中作为一种拓扑选择出现。

### CrewAI Crew vs Flow

CrewAI形式化了两种部署模式：

- **Flow**用于确定性事件驱动自动化（生产环境的推荐起点）。
- **Crew**用于自主角色型协作。

这与上述四种模式正交，但映射到拓扑：Flow通常是主管或层级；Crew通常是带LLM路由器的主管。

### Anthropic的指导原则

"在LLM领域，成功不在于构建最复杂的系统，而在于构建适合你需求的系统。"

决策顺序：

1. 单个智能体 + 工作流模式（第12课）——从这里开始。
2. 主管-工人——当你有2-4个专家时。
3. 群体——当延迟比推理清晰度更重要时。
4. 层级——仅在主管上下文预算失败时。
5. 辩论——当准确性比成本更重要时。

### 这种模式出错的地方

- **拓扑优先思维。** 在确定多智能体能解决什么问题之前就说"我们需要多智能体"。
- **群体中的弹跳移交。** A -> B -> A -> B。使用跳数计数器。
- **虚假层级。** 因为"企业"而设置三层，但实际只有两个团队。简化。

## 动手构建

`code/main.py`使用标准库针对脚本化LLM实现了所有四种模式：

- `Supervisor`——中央路由器。
- `Supervisor`——点对点直接移交。
- `Supervisor`——主管的主管。
- `Supervisor`——并行提议者+批评。

每种模式处理相同的三意图任务（退款/缺陷/销售）。追踪形状不同。

运行它：

```
python3 code/main.py
```

输出：每种模式的追踪+操作计数。主管模式最清晰；群体模式最短；层级模式最深；辩论模式最昂贵。

## 使用它

- **LangGraph**用于主管和层级（嵌套子图）。
- **OpenAI Agents SDK**用于作为工具的移交（主管形状）。
- **CrewAI Flow**用于生产确定性。
- **自定义**用于辩论或当你需要精确控制时。

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|----------------|------------------------|
|  监督者-工作者  |  "路由器+专家"  |  中央LLM分发任务给专家；他们不互相通信  |
|  群  |  "点对点"  |  通过共享工具直接交接；没有中央路由器  |
|  层次化  |  "监督者的监督者"  |  针对大量节点的嵌套子图  |
|  辩论  |  "提议者+批评者"  |  并行提议者，交叉批评（第25课）  |
|  基于工具调用的监督(Tool-call-based supervision)  |  "无库的监督者"  |  将监督者实现为直接工具调用以进行上下文控制  |
|  团队(Crew)  |  "自治团队"  |  CrewAI的基于角色的协作模式  |
|  流程(Flow)  |  "确定性工作流"  |  CrewAI的事件驱动生产模式  |
