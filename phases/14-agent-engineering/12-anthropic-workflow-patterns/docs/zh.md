# Anthropic 的工作流模式：简单优于复杂

> Schluntz 和 Zhang（Anthropic，2024 年 12 月）区分了工作流（预定义路径）和智能体（动态工具使用）。五种工作流模式覆盖了大多数情况。从直接 API 调用开始。仅当步骤无法预测时才添加智能体。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前提：** 阶段 14 · 01（智能体循环）
**时间：** 约 60 分钟

## 学习目标

- 说出 Anthropic 的五种工作流模式：提示链(prompt chaining)、路由(routing)、并行化(parallelization)、协调器-工作者(orchestrator-workers)、评估器-优化器(evaluator-optimizer)。
- 解释智能体与工作流的区别以及各自的工程成本。
- 确定何时选用工作流而非智能体（反之亦然）。
- 使用标准库针对脚本化 LLM 实现所有五种模式。

## 问题

团队会为那些本只需一次函数调用的问题去使用多智能体框架。代价是实实在在的：框架增加了层级，掩盖了提示词，隐藏了控制流，并招致过早的复杂性。Schluntz 和 Zhang 在 2024 年 12 月的文章是业界最常引用的反对意见：从简单开始，只在必要时才增加复杂性。

## 核心概念

### 工作流与智能体

- **工作流。** 通过预定义代码路径编排 LLM 和工具。工程师拥有图。
- **智能体。** LLM 动态地引导自己的工具并执行自己的步骤。模型拥有图。

两者各有其用。工作流更便宜、更快且更容易调试。智能体可以解决开放性问题，但使失败模式更难推理。

### 增强型 LLM

所有五种模式的基础：一个具备三种能力的 LLM——搜索（检索）、工具（动作）、记忆（持久化）。任何 API 调用都可以使用这些能力。

### 五种模式

1. **提示链。** 调用 1 的输出是调用 2 的输入。用于具有清晰线性分解的任务。步骤之间可选的程序化门控。

2. **路由。** 一个分类器 LLM 选择调用哪个下游 LLM 或工具。用于不同类别的输入需要不同处理时（一级支持 vs 退款 vs 缺陷 vs 销售）。

3. **并行化。** 并发运行 N 个 LLM 调用，汇总结果。两种形式：分区（不同块）和投票（相同提示，N 次运行，多数/综合）。

4. **协调器-工作者。** 一个协调器 LLM 动态决定运行哪些工作者（也是 LLM）并综合它们的输出。类似于智能体循环，但协调器不会无限循环。

5. **评估器-优化器。** 一个 LLM 提出答案，另一个 LLM 评估它。迭代直到评估器通过。这是广义的自优化(Self-Refine)（第 05 课）。

### 何时工作流优于智能体

- **可预测的任务。** 如果你能枚举步骤，就应该这么做。
- **受成本限制的任务。** 工作流有固定的步骤数；智能体可能失控。
- **受合规限制的任务。** 审计人员希望阅读图，而不是从轨迹中推断。

### 何时智能体优于工作流

- **开放式研究。** 下一步取决于上一步返回结果时。
- **变长任务。** 分钟到小时的工作，步骤数未知。
- **新颖领域。** 当你还不知道正确的工作流时——先探索，后规范化。

### 上下文工程伴侣

"《为 AI 智能体进行有效的上下文工程》" (Anthropic 2025) 将这一相邻学科正式化：20 万词窗口是一种预算，而不是一个容器。包含什么，何时压缩，何时让上下文增长。在阶段 14 关于上下文压缩的课程中详细讨论（阶段 14 之前的课程 06，在重新编号之前）。

## 动手构建

`code/main.py` 针对一个 `ScriptedLLM` 实现了所有五种工作流模式：

- `prompt_chain(input, steps)` — 顺序。
- `prompt_chain(input, steps)` — 分类 + 分发。
- `prompt_chain(input, steps)` — N 次运行，汇总。
- `prompt_chain(input, steps)` — 协调器选择工作者。
- `prompt_chain(input, steps)` — 循环直到通过。

运行它：

```
python3 code/main.py
```

每个模式打印其轨迹。每个模式的代码行数约为 10-15 行；框架的代价则以千行计。

## 使用它

- 大多数任务使用直接 API 调用。
- 仅当模式真正需要持久状态（LangGraph）、参与者模型并发（AutoGen v0.4）或角色模板（CrewAI）时才使用框架。
- 当你想要 Claude Code 工具的形状而不重新构建时，使用 Claude Agent SDK。

## 练习

1. 实现带置信度阈值的路由。低于阈值 -> 升级给人工。对于一级支持用例，阈值应该设在多少？
2. 给 `parallel_vote` 添加超时。当一个调用挂起时会发生什么？如何在缺少投票的情况下汇总？
3. 将 `parallel_vote` 变成一种赌博机：在迭代中保留前两个输出，这样后面好的结果不会被后面坏的结果覆盖。
4. 将提示链与路由结合：一个路由器选择三个链中的一个。测量令牌成本与单个大提示替代方案的对比。
5. 从你的生产功能中选一个。画出工作流图。统计步骤。这里智能体真的更好吗？

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|----------------|------------------------|
|  工作流  |  "预定义流程"  |  工程师拥有的 LLM 和工具调用图  |
|  智能体  |  "自主 AI"  |  模型拥有的图；动态工具方向  |
| 增强型LLM | "带工具的LLM" | LLM + 搜索 + 工具 + 内存；基本单元 |
| 提示链式调用 | "顺序调用" | 第N次调用的输出是第N+1次调用的输入 |
| 路由 | "分类器调度" | 选择哪个链或模型处理输入 |
| 并行化 | "扇出" | N个并发调用；通过分块或投票聚合 |
| 编排器-工作器 | "调度代理" | 编排器LLM动态选择专业LLM |
| 评估器-优化器 | "提议者+评判者" | 迭代直到评估通过；自优化泛化 |
