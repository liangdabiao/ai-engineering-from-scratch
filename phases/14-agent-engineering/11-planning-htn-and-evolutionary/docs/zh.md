# 使用 HTN 和进化搜索进行规划

> 符号规划处理计划可证明正确的情形。进化代码搜索处理适应度函数可机器检查的情形。ChatHTN (2025) 和 AlphaEvolve (2025) 展示了每种方法与LLM结合后解锁的功能。

**类型：** 构建
**语言：** Python (标准库)
**先决条件：** 阶段14 · 02 (ReWOO 和 Plan-and-Execute)
**时间：** 约75分钟

## 学习目标

- 解释层次化任务网络（Hierarchical Task Networks）：任务、方法、操作符、前提条件、效果。
- 描述ChatHTN的混合循环——符号搜索与LLM后备分解。
- 解释AlphaEvolve的进化循环以及为什么它只适用于编程评估器。
- 在标准库中实现一个玩具HTN规划器和一个玩具进化搜索。

## 问题

ReWOO（第02课）、Plan-and-Execute和ReAct涵盖了大多数智能体规划。但它们未能很好覆盖两种情况：

1. **具有可证明正确性的计划。** 调度、飞行路线规划、合规工作流——计划必须在结构上合理。一个流畅但有时会幻觉一个步骤的LLM计划是不可接受的。
2. **具有可机器检查的适应度函数的优化。** 矩阵乘法、调度启发式、编译器优化步骤——目标不是“一个正确的计划”而是“最佳计划”。

HTN规划和AlphaEvolve解决了这两个不同的问题。两者都将LLM用作放大器，而非替代品。

## 核心概念

### 层次化任务网络（Hierarchical Task Networks）

HTN包含：

- **任务（Tasks）** — 复合任务（需分解）和基本任务（可直接执行）。
- **方法（Methods）** — 将复合任务分解为子任务的方法，带有前提条件。
- **操作符（Operators）** — 具有前提条件和效果的基本动作。
- **状态（State）** — 一组事实。

规划：给定一个目标任务和初始状态，找到分解为基本操作符的方式，这些操作符的前提条件按顺序得到满足。

HTN比LLM更古老，至今仍是可证明正确计划的参考标准。

### ChatHTN (Gopalakrishnan 等人, 2025)

ChatHTN (arXiv:2505.11814) 将符号HTN与LLM查询交替进行：

1. 尝试用现有方法分解当前复合任务。
2. 如果没有适用方法，询问LLM：“在状态`s`下，你会如何分解`task`？”
3. 将LLM响应转换为候选子任务。
4. 根据操作符模式进行验证；拒绝无效分解。
5. 递归进行。

论文的核心主张：生成的每个计划都是可证明合理的，因为LLM建议仅作为候选分解进入，从不直接编辑计划。符号层拥有正确性；LLM扩展方法库。

在线方法学习（OpenReview `gwYEDY9j2x`，2025年后续工作）增加了一个学习者，通过回归泛化LLM生成的分解——将LLM查询频率降低高达75%。

### AlphaEvolve (Novikov 等人, 2025)

AlphaEvolve (arXiv:2506.13131, DeepMind, 2025年6月) 是另一种不同的方法：由Gemini 2.0 Flash/Pro组合协调的进化代码搜索。

循环：

1. 从种子程序加上一个编程评估器（返回适应度分数）开始。
2. LLM组合提出变异。
3. 通过评估器运行变异。
4. 保留最佳；再次变异。

已公布的成果：

- 56年来首次在4x4复数矩阵乘法上超越Strassen（48次标量乘法）。
- 通过Borg调度启发式恢复了谷歌0.7%的计算资源。
- 在前沿工作负载上FlashAttention加速32%。

硬约束：适应度函数必须可机器检查。对散文答案进行进化搜索不会收敛。

### 何时使用哪种

|  问题类别  |  使用  |  原因  |
|---------------|-----|-----|
|  带有硬约束的调度  |  HTN + ChatHTN  |  可证明合理性  |
|  编译器优化  |  AlphaEvolve  |  可机器检查的适应度  |
|  多步任务执行  |  ReAct / ReWOO  |  LLM在循环中，无形式化保证  |
|  通过测试改进代码  |  AlphaEvolve  |  测试即评估器  |
|  受策略约束的自动化  |  HTN  |  前提条件编码策略  |

### 这种模式出错的地方

- **没有运算符的HTN。** 没有前提/效果模式，正确性声明就崩溃了。ChatHTN的“LLM建议分解”需要模式来拒绝无效的动作。
- **没有真正评估器的AlphaEvolve。** “让LLM判断代码是否更好”不是一个适应度函数。评估器必须是确定且快速的。
- **过度工程化。** 大多数智能体任务并不需要这两者。先考虑使用ReAct或ReWOO。

## 动手构建

`code/main.py`实现了两个玩具：

- 一个带有运算符、方法、前提、效果以及一个在无方法匹配复合任务时启动的`LLMFallback`的标准库HTN规划器。“LLM”是一个脚本化的分解器，因此规划器离线运行。
- 一个对算术程序进行标准库进化搜索：生成表达式，其输出在测试集上最小化`LLMFallback`。评估器是确定性的。

运行它：

```
python3 code/main.py
```

跟踪显示了HTN规划器分解一个复合任务（带有规划中期的LLM后备）以及进化循环收敛到目标表达式的过程。

## 使用它

- **HTN规划器** — `pyhop`、`SHOP3`，或自己构建用于领域特定策略执行。
- **ChatHTN** — 研究代码；该模式（符号+LLM后备）可干净地移植到任何HTN规划器。
- **AlphaEvolve** — DeepMind论文；该模式（集成+评估器）是可复现的。OpenEvolve和类似的开源分支正在涌现。
- **智能体框架** — 目前还没有哪一个内置一流的HTN或AlphaEvolve。将其构建为子智能体或后台工作器。

## 练习

1. 扩展HTN规划器以支持回溯：当运算符的后置条件在运行时失败时，回滚并尝试下一个方法。
2. 为ChatHTN添加LLM方法缓存：当LLM在状态模式`P`下分解任务`T`时，存储结果。下一次调用时首先重新检查方法库。
3. 将进化搜索评估器替换为真正的测试套件。进化一个通过20个测试用例的排序函数；报告收敛时的代数。
4. 阅读AlphaEvolve的评估器设计笔记。为你关心的领域（SQL查询优化、测试套件最小化、部署YAML）设计一个评估器。
5. 组合：使用HTN将复合任务分解为子任务，然后在每个子任务的原语运算符上使用进化搜索。它在哪些方面表现出色，哪些方面过度工程化？

## 关键术语

|  术语  |  人们的说法  |  实际含义  |
|------|----------------|------------------------|
|  HTN  |  “层次化规划器”  |  带有运算符、前提、效果的任务分解  |
|  方法  |  “分解规则”  |  将复合任务分解为子任务的方式  |
|  运算符  |  “原语动作”  |  具有前提和效果的具体步骤  |
|  ChatHTN  |  “LLM + HTN”  |  当无方法匹配时，符号规划器询问LLM  |
|  AlphaEvolve  |  “进化代码搜索”  |  集成LLMs变异代码；确定性评估器选择  |
|  适应度函数  |  “评估器”  |  对输出进行确定性的、机器可检查的评分  |
|  在线方法学习  |  “缓存的LLM分解”  |  存储并泛化LLM计划以降低查询成本  |
